🛠️ Lektion 06 — TDD: Neues Feature test-first

Ziel: Ein komplettes neues Chatbot-Feature von Anfang bis Ende mit TDD entwickeln — ohne eine Zeile Produktionscode vorab zu schreiben.

Das Feature: "hilfe: <befehl>"

Neue Anforderung: Der Bot soll spezifische Hilfe zu einem Command geben können. "hilfe: moin" soll eine Erklärung nur zum moin-Command zurückgeben.

Wir entwickeln nur die Parse-Logik — kein echtes Claude-Aufruf, nur Prefix-Erkennung und Extraktion.

TDD-Ablauf: Schritt für Schritt

  1. Anforderung klären: Was soll die Funktion tun?
  2. Ersten (einfachsten) Test schreiben
  3. Fehlschlag bestätigen (Red)
  4. Minimaler Code (Green)
  5. Nächsten Test schreiben (Grenzfall)
  6. Erneut Red → Green
  7. Refactoren wenn Duplikate entstehen

Zyklus 1 — Prefix-Erkennung

🔴 RED
# tests/test_bot.py — am Ende anhängen:

def test_hilfe_befehl_prefix():
    text = "hilfe: moin"
    assert text.lower().startswith("hilfe:")
    assert text[6:].strip() == "moin"
# Ausführen:
pytest tests/test_bot.py::test_hilfe_befehl_prefix -v
# PASSED (sofort!) — warum?
Achtung: Dieser Test testet nur Python-String-Operationen, nicht unseren Code. Er ist sofort grün, weil startswith und String-Slicing immer funktionieren. Das ist ein Design-Hinweis: Prefix-Erkennung braucht keine eigene Funktion — sie passiert direkt im Handler.

Zyklus 2 — Leeren Befehl behandeln

🔴 RED
def test_hilfe_ohne_befehl():
    text = "hilfe:"
    assert text[6:].strip() == ""
# PASSED sofort — wieder kein Code nötig

Zyklus 3 — hilfe: in known_prefixes

🔴 RED
def test_hilfe_ist_bekannter_befehl():
    known_prefixes = (
        "task:", "status:", "fokus:", "verschieben:", "lern:",
        "idee:", "habit:", "termin:", "projekt:", "teach:", "erinnere",
        "implement-plan:", "abort-plan:", "backlog:", "suche:", "hilfe:",
    )
    assert "hilfe: moin".lower().startswith(known_prefixes)

# FAILS — "hilfe:" fehlt noch in bot.py!
🟢 GREEN

In bot.py die KNOWN_PREFIXES Konstante um "hilfe:" erweitern (oder wo immer Prefixes definiert sind). Test wird grün.

Zyklus 4 — Hilfe-Text Struktur

🔴 RED
def test_hilfe_text_enthaelt_befehl_erklaerungen():
    from bot import HILFE_TEXT
    # Hilfe soll alle Commands erklären
    for cmd in ["moin", "abend", "task:", "status:"]:
        assert cmd in HILFE_TEXT
# Ausführen → prüfen ob alle Commands dokumentiert sind
🟢 GREEN

Falls ein Command fehlt: HILFE_TEXT in bot.py ergänzen. Tests grün machen, einen nach dem anderen.

🔵 REFACTOR

Gibt es Duplikate in HILFE_TEXT? Ist die Liste der Commands an einer oder mehreren Stellen definiert? Wenn mehreren Stellen: zu einer zentralen Konstante zusammenfassen.

Was du aus diesem Zyklus lernst

  • Manchmal braucht Code keinen eigenen Test — wenn das Verhalten triviale Python-Operationen sind
  • Tests enthüllen Design-Probleme — "schwer zu testen" bedeutet oft "schlecht designed"
  • Kurze Zyklen — nicht fünf Features auf einmal, ein Test nach dem anderen
  • Red ist Information — der Fehler zeigt dir genau was fehlt

🧠 Quiz

Frage 1: Du schreibst einen Test und er ist sofort grün ohne Code zu schreiben. Was bedeutet das?

A) Der Test ist überflüssig — löschen
❌ Nicht unbedingt. Manchmal ist das Verhalten bereits korrekt (z.B. wenn du ein Refactoring machst). Aber: untersuche warum es grün ist.
B) Entweder das Feature ist schon vorhanden, oder der Test testet triviales Verhalten
✅ Richtig! Beide Fälle sind möglich — beide sind informativ.

Frage 2: Du merkst während TDD dass deine Funktion schwer zu testen ist. Was tust du?

A) Mocking hinzufügen damit der Test funktioniert
❌ Vielleicht. Aber zuerst: warum ist es schwer? Oft ist "schwer zu testen" ein Hinweis auf schlechtes Design.
B) Das Design der Funktion überdenken — zu viele Abhängigkeiten deuten auf schlechtes Design hin
✅ Richtig! TDD erzwingt testbares Design. Wenn der Test schmerzt, ist das wertvolles Feedback.

Aufgabe: Schreibe einen TDD-Zyklus für eine neue Funktion extract_command(text) die den Befehlsnamen aus "task: Einkauf" extrahiert → "task".

Zeig mir die Lösung
# Red:
def test_extract_command_task():
    assert extract_command("task: Einkauf") == "task"

# Green:
def extract_command(text):
    return text.split(":")[0].strip().lower()

# Refactor: braucht es lower()? Hängt von Anforderungen ab.

📋 Zusammenfassung

  • TDD für neue Features: kleine Zyklen, ein Test nach dem anderen
  • Sofort grüne Tests → untersuchen warum (triviales Verhalten oder Fehler im Test?)
  • "Schwer zu testen" ist ein Design-Problem, kein Test-Problem
  • Grenzfälle (leere Eingabe, Sonderzeichen) als eigene TDD-Zyklen
  • Refactoring nur wenn alle Tests grün — Schritt für Schritt
