🟢 Lektion 02 — pytest Basics

Ziel: pytest aufsetzen, die Dateistruktur verstehen, und erste Tests im Chatbot-Projekt schreiben — im TDD-Stil.

Was ist pytest?

pytest ist das Standard-Testframework für Python. Es ist einfacher als das eingebaute unittest, erkennt Tests automatisch, und gibt dir lesbare Fehlermeldungen. Dein Chatbot-Projekt nutzt es bereits — schau in tests/.

Installation

# Im Projektordner:
pip install pytest

# Testen ob es funktioniert:
pytest --version

Wie erkennt pytest Tests?

  • Dateien die mit test_ beginnen oder auf _test.py enden
  • Funktionen die mit test_ beginnen
  • Klassen die mit Test beginnen (optional, seltener genutzt)

Ein einfacher Test — Anatomie

# tests/test_beispiel.py

def test_addition():           # Name fängt mit test_ an
    ergebnis = 2 + 3           # Arrange + Act
    assert ergebnis == 5       # Assert — hier prüfen wir

Das assert-Statement ist alles was du brauchst. Wenn es fehlschlägt, zeigt pytest dir genau was erwartet vs. was tatsächlich kam.

Tests ausführen

# Alle Tests:
pytest

# Einen bestimmten Test:
pytest tests/test_bot.py::test_doppelpunkt_lower

# Mit ausführlicher Ausgabe:
pytest -v

# Mit kurzem Summary (was ist rot/grün):
pytest -v --tb=short

Assertions in pytest

Was du prüfstAssertion
Gleichheitassert x == 5
Ungleichheitassert x != 5
Wahrheitswertassert flag is True
None-Prüfungassert result is None
Enthältassert "moin" in text
Exception erwartetwith pytest.raises(ValueError): ...
Typ-Prüfungassert isinstance(x, dict)

Dein erster TDD-Zyklus mit pytest

Stell dir vor, die Funktion normalize_voice existiert noch nicht. So würde TDD aussehen:

# SCHRITT 1 — Red: Test schreiben
# tests/test_bot.py
def test_doppelpunkt():
    assert normalize_voice("task doppelpunkt Einkauf") == "task: Einkauf"

# pytest ausführen → FEHLER: ImportError oder NameError ← gewollt!
# SCHRITT 2 — Green: minimale Implementierung
# bot.py
def normalize_voice(text):
    return text.replace(" doppelpunkt ", ": ")

# pytest ausführen → PASSED ✓
# SCHRITT 3 — Refactor: weitere Ersetzungen hinzufügen
# (neuer Test zuerst, dann Erweiterung)
conftest.py ist eine spezielle pytest-Datei für gemeinsame Fixtures. In deinem Projekt setzt sie Environment-Variablen (TELEGRAM_TOKEN, GROQ_API_KEY) für alle Tests. Du musst sie nicht anfassen — sie ist schon richtig.

Nützliche Optionen

# Nur fehlgeschlagene Tests erneut laufen:
pytest --lf

# Test stoppen beim ersten Fehler:
pytest -x

# Tests nach Namen filtern:
pytest -k "doppelpunkt"

🧠 Quiz

Frage 1: Welche Datei erkennt pytest automatisch als Testdatei?

A) bot_tests.py
❌ Falsch. pytest erkennt Dateien die mit test_ beginnen oder auf _test.py enden.
B) test_bot.py
✅ Richtig! Beginnt mit test_.
C) tests.py
❌ Falsch. Kein test_-Prefix.

Frage 2: Was macht pytest -x?

A) Führt nur Tests mit "x" im Namen aus
❌ Falsch. Für Namen-Filter benutzt du -k.
B) Stoppt beim ersten fehlgeschlagenen Test
✅ Richtig! Nützlich wenn viele Tests rot sind — du fixst einen nach dem anderen.

Aufgabe: Schreibe einen Test der prüft dass "moin" in HILFE_TEXT vorkommt. Führe ihn aus — ist er grün?

Zeig mir die Lösung
from bot import HILFE_TEXT

def test_hilfe_enthaelt_moin():
    assert "moin" in HILFE_TEXT
Dieser Test sollte sofort grün sein — er testet existierendes Verhalten.

📋 Zusammenfassung

  • pytest erkennt Dateien/Funktionen mit test_-Prefix automatisch
  • assert ist alles was du zum Prüfen brauchst
  • pytest -v für ausführliche, pytest -x für früh-stop Ausgabe
  • conftest.py für gemeinsame Setup-Logik (Fixtures, ENV-Variablen)
  • TDD-Zyklus: Test schreiben → rot sehen → minimal grün machen → refactoren
