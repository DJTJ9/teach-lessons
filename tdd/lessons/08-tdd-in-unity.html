🎮 Lektion 08 — TDD in Unity: Spielemechanik test-first

Ziel: Eine Spielemechanik (Inventory-System) vollständig mit TDD entwickeln — kein Code ohne roten Test davor.

Das Szenario: Inventory-System

Wir bauen ein einfaches Inventory für ein Spiel. Anforderungen:

  • Items können hinzugefügt werden
  • Maximale Kapazität (z.B. 10 Items)
  • Items können entfernt werden
  • Prüfen ob ein Item vorhanden ist

Wichtig: Das Inventory hat keine MonoBehaviour-Abhängigkeit → EditMode Tests.

Schritt 1 — Red: Erstes Item hinzufügen

🔴 RED
// Assets/Tests/EditMode/InventoryTests.cs
using NUnit.Framework;

public class InventoryTests
{
    [Test]
    public void AddItem_ErhoehCountUm1()
    {
        var inventory = new Inventory(capacity: 10);
        inventory.Add("Schwert");
        Assert.AreEqual(1, inventory.Count);
    }
}

// Ausführen: KOMPILIERUNGSFEHLER — Inventory existiert nicht ← gewollt!

Schritt 2 — Green: Minimale Inventory-Klasse

🟢 GREEN
// Assets/Scripts/Inventory.cs
using System.Collections.Generic;

public class Inventory
{
    private List<string> items = new List<string>();
    private int capacity;

    public int Count => items.Count;

    public Inventory(int capacity)
    {
        this.capacity = capacity;
    }

    public void Add(string item)
    {
        items.Add(item);
    }
}

// Test Runner: PASSED ✓

Zyklus 2 — Kapazitäts-Limit

🔴 RED
[Test]
public void Add_BeiVollemInventory_WirftException()
{
    var inventory = new Inventory(capacity: 2);
    inventory.Add("Schwert");
    inventory.Add("Schild");
    
    Assert.Throws<System.InvalidOperationException>(() => {
        inventory.Add("Heiltrank");  // soll Exception werfen
    });
}

// FAILS — Add wirft noch keine Exception
🟢 GREEN
public void Add(string item)
{
    if (items.Count >= capacity)
        throw new System.InvalidOperationException("Inventory voll");
    items.Add(item);
}

// PASSED ✓

Zyklus 3 — Item vorhanden prüfen

🔴 RED
[Test]
public void Contains_VorhandenesItem_ReturnsTrue()
{
    var inventory = new Inventory(10);
    inventory.Add("Schwert");
    Assert.IsTrue(inventory.Contains("Schwert"));
}

[Test]
public void Contains_NichtVorhandenes_ReturnsFalse()
{
    var inventory = new Inventory(10);
    Assert.IsFalse(inventory.Contains("Schwert"));
}
🟢 GREEN
public bool Contains(string item) => items.Contains(item);

Zyklus 4 — Item entfernen

🔴 RED
[Test]
public void Remove_VorhandenesItem_ReduziertCount()
{
    var inventory = new Inventory(10);
    inventory.Add("Schwert");
    inventory.Remove("Schwert");
    Assert.AreEqual(0, inventory.Count);
}

[Test]
public void Remove_NichtVorhandenes_WirftException()
{
    var inventory = new Inventory(10);
    Assert.Throws<System.ArgumentException>(() => {
        inventory.Remove("Schwert");
    });
}
🟢 GREEN
public void Remove(string item)
{
    if (!items.Contains(item))
        throw new System.ArgumentException($"{item} nicht im Inventory");
    items.Remove(item);
}

Refactor — SetUp nutzen

🔵 REFACTOR

Jeder Test erstellt new Inventory(10) — das ist Duplikation. Mit [SetUp] einmal initialisieren:

public class InventoryTests
{
    private Inventory inventory;

    [SetUp]
    public void Setup()
    {
        inventory = new Inventory(capacity: 10);
    }

    [Test]
    public void AddItem_ErhoehCountUm1()
    {
        inventory.Add("Schwert");
        Assert.AreEqual(1, inventory.Count);
    }

    // ... alle anderen Tests nutzen jetzt inventory direkt
}

// Alle Tests nochmal ausführen: alle PASSED ✓
Kernlektion: Das Inventory hat jetzt 6 Tests, die alle Kantenfälle abdecken. Wenn du später das Inventory intern umbauen willst (z.B. von List<string> zu einer eigenen Klasse), sagen dir die Tests sofort ob du etwas kaputt gemacht hast. Das ist TDD-Sicherheitsnetz in Aktion.

🧠 Quiz

Frage 1: Du willst ein Inventory-Item als ScriptableObject statt als string nutzen. Was musst du beim Refactoring nicht ändern?

A) Die Tests — das Verhalten bleibt gleich
❌ Die Tests müssen angepasst werden, da sich der Typ ändert. Aber das Verhalten (Add/Remove/Contains/Count) bleibt gleich.
B) Das Verhalten der Inventory-Klasse — nur der interne Typ ändert sich
✅ Richtig! Das externe Verhalten bleibt gleich — Kapazität, Exceptions, Count. Nur die Tests für den Typ müssen minimal angepasst werden.

Frage 2: Wozu dient [SetUp] in NUnit?

A) Läuft einmal vor allen Tests
❌ Falsch. Das wäre [OneTimeSetUp]. [SetUp] läuft vor jedem einzelnen Test.
B) Läuft vor jedem einzelnen Test — für saubere Initialisierung
✅ Richtig! Jeder Test startet so mit einem frischen Zustand — isolated.

Aufgabe: Schreibe den Test: "Add mit leerem String soll ArgumentException werfen."

Zeig mir die Lösung
[Test]
public void Add_LeererString_WirftException()
{
    Assert.Throws<System.ArgumentException>(() => {
        inventory.Add("");
    });
}
Dann in Add(): if (string.IsNullOrWhiteSpace(item)) throw new ArgumentException(...);

📋 Zusammenfassung — TDD Kurs abgeschlossen!

  • TDD in Unity funktioniert genauso wie in Python: Red → Green → Refactor
  • EditMode für reine Logik (Inventory, Health, Scoring) — schnell
  • [SetUp] verhindert Duplikation und sichert Isolation
  • Tests als Sicherheitsnetz für Refactoring — ohne Tests ist Umbauen riskant
  • Grenzfälle (volles Inventory, leerer String, nicht vorhandenes Item) immer testen
