🔬 Lektion 04 — Was testen wir eigentlich?

Ziel: Unit-, Integrations- und E2E-Tests unterscheiden, die FIRST-Prinzipien kennen, und wissen welchen Testtyp man wann schreibt.

Die Testpyramide

E2E Tests (wenige)
Integration Tests (mittel)
Unit Tests (viele)

Je weiter oben, desto langsamer, teurer, fragiler. Je weiter unten, desto schneller und zuverlässiger.

Unit Tests

Ein Unit Test prüft eine einzelne Funktion oder Klasse in Isolation. Keine Datenbank, kein Netzwerk, keine Filesystem-Zugriffe.

# UNIT TEST — Chatbot-Beispiel
# Testet normalize_voice() ohne externe Abhängigkeiten
def test_doppelpunkt_lower():
    assert normalize_voice("task doppelpunkt bug fixen") == "task: bug fixen"

# Läuft in Millisekunden, immer deterministisch

Im Chatbot: Die meisten Tests in test_bot.py sind Unit Tests — sie prüfen Funktionen wie normalize_voice, load_settings, save_settings direkt.

Integration Tests

Ein Integration Test prüft wie mehrere Komponenten zusammenarbeiten. Oft mit echten Dateisystemen, aber noch ohne echtes Netzwerk.

# INTEGRATION TEST — Chatbot-Beispiel
def test_save_load_roundtrip(tmp_path):
    # Prüft: save_settings() UND load_settings() zusammen
    save_settings({"notifications_enabled": False}, tmp_path)
    assert load_settings(tmp_path) == {"notifications_enabled": False}

# tmp_path = pytest Fixture für temporäres Verzeichnis

Im Chatbot: test_save_load_roundtrip und test_save_load_reminders_roundtrip sind Integration Tests — sie schreiben und lesen echte Dateien.

E2E Tests (End-to-End)

E2E Tests prüfen den gesamten Ablauf — Telegram schickt Nachricht, Bot antwortet, Notion wird aktualisiert. Selten, langsam, fragil. Im Chatbot-Projekt haben wir (noch) keine E2E-Tests.

Welcher Testtyp wann?

FrageTesttyp
Macht diese Funktion das Richtige?Unit
Spielen diese zwei Komponenten korrekt zusammen?Integration
Funktioniert der komplette User-Flow?E2E
Läuft dieser Code schnell und oft?Unit

FIRST-Prinzipien — Merksatz für gute Tests

F — Fast
Tests müssen schnell sein. Langsame Tests werden nicht laufend ausgeführt.
I — Isolated
Kein Test hängt von einem anderen ab. Reihenfolge egal.
R — Repeatable
Immer gleiches Ergebnis. Kein Datum, kein Zufall, kein externes System.
S — Self-validating
Test sagt selbst Pass/Fail. Kein manuelles Prüfen der Ausgabe.
T — Timely
Test wird vor dem Produktionscode geschrieben (TDD-Prinzip).

Test-Smells — Warnzeichen für schlechte Tests

  • Langer Setup-Code vor dem eigentlichen Test → Funktion hat zu viele Abhängigkeiten
  • Test prüft interne Implementierung statt Verhalten → zerbricht bei jedem Refactor
  • Test testet mehrere Dinge → wenn er rot wird, weiß man nicht warum
  • Test hängt von Zeit/Datum ab → morgen könnte er rot sein
  • Tests stören sich gegenseitig → Reihenfolge ändert Ergebnis
Chatbot-Beispiel für Test-Smell:
test_save_load_reminders_roundtrip benutzt monkeypatch um den echten Dateipfad zu überschreiben. Das ist gut — er ist dadurch isolated. Ohne monkeypatch würde er echte Productionsdaten lesen/schreiben und nicht repeatable sein.

🧠 Quiz

Frage 1: Welches FIRST-Prinzip verletzt ein Test der das aktuelle Datum nutzt?

A) Fast
❌ Falsch. Das Datum macht ihn nicht langsam.
B) Repeatable
✅ Richtig! Heute grün, morgen rot — nicht wiederholbar.

Frage 2: Du hast 100 Tests, 90 davon sind Unit Tests. Ist das gut oder schlecht?

A) Schlecht — mehr Integration Tests wären besser
❌ Falsch. Die Testpyramide empfiehlt viele Unit Tests an der Basis.
B) Gut — das entspricht der Testpyramide
✅ Richtig! Viele schnelle Unit Tests + wenige langsame Integration Tests.

Frage 3: Welcher Test-Smell liegt vor wenn Test A fehlschlägt weil Test B vorher gelaufen ist?

A) Verletzung von Isolated
✅ Richtig! Tests dürfen sich nicht gegenseitig beeinflussen.
B) Verletzung von Timely
❌ Falsch. Timely betrifft wann der Test geschrieben wird, nicht die Ausführungsreihenfolge.

📋 Zusammenfassung

  • Unit Test: eine Funktion in Isolation — schnell, viele davon
  • Integration Test: mehrere Komponenten zusammen — mittel, wenige
  • E2E Test: kompletter Flow — langsam, sehr wenige
  • FIRST: Fast, Isolated, Repeatable, Self-validating, Timely
  • Test-Smells erkennen: langer Setup, externe Abhängigkeiten, mehrere Prüfpunkte pro Test
  • Die meisten TDD-Tests sind Unit Tests — klein und schnell
