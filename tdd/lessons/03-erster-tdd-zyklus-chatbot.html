🔴🟢🔵 Lektion 03 — Erster TDD-Zyklus am Chatbot

Ziel: Ein neues Feature im Chatbot komplett test-first entwickeln — kein Produktionscode ohne roten Test davor.

Das Szenario

Dein Chatbot soll Spracheingaben normalisieren. Die Funktion normalize_voice existiert bereits. Wir bauen eine neue Erweiterung: Zahlen sollen als Sprachersatz erkannt werden.

Anforderung: "eins komma fünf" → "eins, fünf"

Stell dir vor, diese Anforderung kam gerade rein. Wir fangen sofort mit dem Test an.

Schritt 1 — Red: Test schreiben

🔴 RED
# Öffne: tests/test_bot.py
# Füge am Ende ein:

def test_komma_ersatz():
    assert normalize_voice("eins komma fünf") == "eins, fünf"

Jetzt ausführen:

pytest tests/test_bot.py::test_komma_ersatz -v

Erwartet: FAILED — weil normalize_voice "komma" noch nicht kennt. Falls er grün ist, stimmt etwas nicht (die Funktion macht schon zufällig das Richtige, oder der Test ist falsch).

Prüfe den Fehler! pytest zeigt dir: AssertionError: assert 'eins komma fünf' == 'eins, fünf'. Das bedeutet: der Test schlägt fehl weil die Transformation fehlt — genau richtig.

Schritt 2 — Green: minimaler Code

🟢 GREEN

Öffne bot.py und such die normalize_voice Funktion. Füge die minimale Erweiterung hinzu:

# In bot.py — nur das Nötigste:
def normalize_voice(text):
    replacements = {
        " doppelpunkt ": ": ",
        " komma ": ", ",        # ← NEU
        # ...weitere Ersetzungen
    }
    for old, new in replacements.items():
        text = text.replace(old, new)
    return text

Ausführen:

pytest tests/test_bot.py -v

Erwartet: PASSED — alle Tests, inklusive dem neuen. Und kein alter Test darf jetzt rot sein!

Schritt 3 — Refactor

🔵 REFACTOR

Ist der Code lesbar? Gibt es Duplikate? In diesem Fall ist das Dict-basierte replacements-System bereits sauber. Refactoring nicht nötig.

Aber was wenn die ursprüngliche Implementierung so aussah?

# Vor Refactor — unübersichtlich:
def normalize_voice(text):
    text = text.replace(" doppelpunkt ", ": ")
    text = text.replace(" komma ", ", ")
    text = text.replace(" punkt ", ".")
    text = text.replace(" ausrufezeichen ", "!")
    return text
# Nach Refactor — gleiche Logik, klarer:
VOICE_REPLACEMENTS = {
    " doppelpunkt ": ": ",
    " komma ": ", ",
    " punkt ": ".",
    " ausrufezeichen ": "!",
}

def normalize_voice(text):
    for old, new in VOICE_REPLACEMENTS.items():
        text = text.replace(old, new)
    return text

Ausführen nach dem Refactor:

pytest tests/test_bot.py -v  # Alle müssen noch grün sein!

Mehrere Testfälle für eine Funktion

Gute Tests decken auch Grenzfälle ab. Was passiert wenn der Text "komma" am Anfang steht? Was wenn kein Ersatz nötig ist?

def test_normalize_voice_kein_ersatz():
    assert normalize_voice("moin") == "moin"

def test_normalize_voice_kombiniert():
    assert normalize_voice("status doppelpunkt sport komma laufen") == "status: sport, laufen"
Reihenfolge beim TDD: Erst der einfachste Fall, dann Grenzfälle. Nicht alle Fälle auf einmal — einen Test, dann green machen, dann nächster Test.

Was haben wir gerade gemacht?

  1. Anforderung bekommen → sofort Test geschrieben
  2. Test gesehen (rot) → bestätigt dass er wirklich prüft
  3. Minimaler Code → grün gemacht
  4. Refactored → Code verbessert, Tests grün geblieben
  5. Grenzfälle → weitere Tests geschrieben (Zyklus wiederholt)

🧠 Quiz

Frage 1: Du schreibst einen Test und er ist sofort grün (ohne dass du Produktionscode geändert hast). Was bedeutet das?

A) Super — der Code ist schon richtig!
❌ Falsch. Wenn der Test nie rot war, beweist er nichts. Entweder ist die Anforderung schon erfüllt, oder der Test ist falsch geschrieben.
B) Entweder ist das Verhalten bereits implementiert, oder der Test testet das Falsche
✅ Richtig! Untersuche warum er grün ist — beabsichtigt oder Testfehler?

Frage 2: Du refactorst und ein Test wird rot. Was tust du?

A) Den Test anpassen damit er wieder grün ist
❌ Vorsicht! Tests anpassen verschleiert Bugs. Zuerst verstehen warum er rot wurde.
B) Das Refactoring rückgängig machen und verstehen warum der Test rot wurde
✅ Richtig! Rot beim Refactor = du hast Verhalten geändert. Das ist ein Bug, kein Testproblem.

Aufgabe: Schreibe einen Test für normalize_voice("Ende punkt") → "Ende.". Lauf ihn rot, dann mach ihn grün.

Zeig mir die Lösung
def test_punkt_ersatz():
    assert normalize_voice("Ende punkt") == "Ende."
Dann in VOICE_REPLACEMENTS: " punkt ": "." hinzufügen. Aber Achtung: "punkt" am Satzende hat kein Leerzeichen danach — edge case!

📋 Zusammenfassung

  • TDD am echten Code: Test → rot sehen → minimal grün machen → refactoren
  • Ein Test der nie rot war, beweist nichts — immer den roten Zustand bestätigen
  • Refactoring ist sicher wenn Tests grün sind — sie passen auf
  • Grenzfälle nach dem Hauptfall testen — auch als eigene TDD-Zyklen
  • Guter TDD-Rhythmus: klein denken, einen Zyklus auf einmal
