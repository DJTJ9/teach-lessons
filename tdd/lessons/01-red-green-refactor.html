🔴🟢🔵 Lektion 01 — Red-Green-Refactor

Ziel: Den TDD-Kernzyklus verstehen und erklären können — bevor du eine Zeile Produktionscode schreibst.

Was ist TDD?

Test-Driven Development (testgetriebene Entwicklung) ist eine Methode, bei der du erst den Test schreibst, dann den Code der ihn grün macht, und schließlich aufräumst. Immer in diesem Zyklus:

🔴 Red

Test für Verhalten schreiben das noch nicht existiert. Der Test muss fehlschlagen.

→

🟢 Green

Minimalen Code schreiben damit der Test besteht. Noch keine Perfektion.

→

🔵 Refactor

Codestruktur verbessern ohne Verhalten zu ändern. Tests bleiben grün.

Warum fehlschlagen lassen? Ein Test der nie rot war, beweist nichts. Er könnte immer grün sein — egal was dein Code macht. Ein roter Test beweist, dass er wirklich prüft.

Das Problem ohne TDD

Ohne TDD schreibst du Code, bist fertig, und schreibst danach Tests die zum existierenden Code passen. Das nennt man post-hoc Testing. Genau das passiert bisher im Chatbot-Projekt — die Tests in tests/ wurden nach dem Code geschrieben.

Das Problem: Du testest was du gebaut hast, nicht was du brauchst. Lücken in den Anforderungen bleiben unbemerkt.

Konkretes Beispiel — Schritt für Schritt

# SCHRITT 1 — Red: Test schreiben (add() existiert noch nicht)
def test_add_zwei_zahlen():
    assert add(2, 3) == 5

# Ausführen: NameError: name 'add' is not defined  ← GEWOLLT!
# SCHRITT 2 — Green: minimaler Code
def add(a, b):
    return a + b

# Ausführen: PASSED ✓
# SCHRITT 3 — Refactor: gibt es etwas zu verbessern?
# Nein — der Code ist bereits minimal und klar.
# Bei komplexerem Code würden wir hier aufräumen, solange Tests grün bleiben.

Regeln des Zyklus

  • Kein Code ohne roten Test davor — immer erst der Test
  • Ein Test auf einmal — nicht fünf auf einmal schreiben
  • Refactor nur wenn grün — nie mit rotem Test umbauen
  • Tests nie löschen — sie dokumentieren Verhalten dauerhaft

Was TDD dir gibt

  • Du weißt sofort wenn du etwas kaputt machst
  • Code ist von Anfang an testbar (kein Spaghetti-Code)
  • Tests sind die Dokumentation — sie zeigen was der Code tun soll
  • Du kannst selbstbewusst refactoren, weil die Tests aufpassen

🧠 Quiz — klick auf eine Antwort

Frage 1: In welcher Reihenfolge läuft der TDD-Zyklus?

A) Green → Red → Refactor
❌ Falsch. Du musst erst wissen was du willst (Test schreiben = Red), dann bauen.
B) Red → Green → Refactor
✅ Richtig! Erst scheitern, dann bestehen, dann verbessern.
C) Code → Test → Refactor
❌ Das ist post-hoc Testing — genau was TDD verhindert.

Frage 2: Was beweist ein Test der noch nie rot war?

A) Dass der Code korrekt ist
❌ Falsch. Er beweist gar nichts — er könnte immer grün sein, egal was passiert.
B) Nichts — er muss erst scheitern um zu beweisen dass er wirklich prüft
✅ Richtig! Ein Test der nie rot war, ist unbewiesen.

Frage 3: Wann darf ich refactoren?

A) Jederzeit, auch wenn Tests rot sind
❌ Falsch. Refactoring bei rotem Test ändert zwei Dinge gleichzeitig — du weißt nicht was etwas kaputt gemacht hat.
B) Nur wenn alle Tests grün sind
✅ Richtig! Grüne Tests = sicheres Netz für Umbau.

📋 Zusammenfassung

  • TDD = Test-first: Red → Green → Refactor
  • Red: Test für nicht-existierendes Verhalten schreiben — muss scheitern
  • Green: minimaler Code damit Test besteht
  • Refactor: Code verbessern, Tests grün halten
  • Ein Test der nie rot war, ist nicht vertrauenswürdig
  • TDD zwingt dich, erst über was nachzudenken, dann über wie
