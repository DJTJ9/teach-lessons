🎭 Lektion 05 — Mocking mit pytest

Ziel: Externe Abhängigkeiten (Telegram API, Dateisystem, subprocess) in Tests durch Fakes ersetzen — damit Unit Tests ohne Netzwerk laufen.

Das Problem: externe Abhängigkeiten

Dein Chatbot ruft echte externe Dienste auf: Telegram API, Notion, Claude CLI (subprocess). Du kannst in Tests nicht:

  • Echte Telegram-Nachrichten senden
  • Echte Notion-Datenbanken schreiben
  • Echte Claude CLI-Prozesse starten

Lösung: Diese Abhängigkeiten durch kontrollierte Fakes (Mocks) ersetzen.

Zwei Werkzeuge in pytest

WerkzeugWann nutzen
monkeypatchFunktionen/Attribute/Pfade temporär überschreiben (pytest built-in)
unittest.mock.patchKomplexere Mocks, Aufruf-Zähler, Return-Values

monkeypatch — einfache Substitution

Du hast das bereits in deinen Tests gesehen:

# aus test_bot.py:
def test_load_reminders_empty(tmp_path, monkeypatch):
    import bot
    monkeypatch.setattr(bot, "REMINDERS_PATH", tmp_path / "reminders.json")
    assert bot.load_reminders() == []

Was passiert:

  • monkeypatch.setattr ersetzt bot.REMINDERS_PATH für die Dauer dieses Tests
  • Danach wird es automatisch zurückgesetzt
  • Der Test berührt keine echten Produktionsdaten
# Weiteres Beispiel — Umgebungsvariable überschreiben:
def test_telegram_token_fehlt(monkeypatch):
    monkeypatch.delenv("TELEGRAM_TOKEN", raising=False)
    # jetzt kann ich testen wie der Code ohne Token reagiert

unittest.mock — mächtigere Mocks

Wenn du prüfen willst ob etwas aufgerufen wurde und mit welchen Argumenten:

from unittest.mock import patch, MagicMock

def test_send_message_called():
    with patch("bot.requests.post") as mock_post:
        mock_post.return_value.status_code = 200
        
        # Rufe Funktion auf die intern requests.post benutzt
        send_telegram_message("Hallo!")
        
        # Prüfe dass requests.post aufgerufen wurde
        mock_post.assert_called_once()
        
        # Prüfe mit welchen Argumenten:
        args, kwargs = mock_post.call_args
        assert "Hallo!" in str(kwargs)

subprocess mocken — Claude CLI

Dein Bot startet Claude als subprocess. Das kannst du so mocken:

from unittest.mock import patch, MagicMock

def test_run_teach_startet_subprocess():
    with patch("bot.subprocess.Popen") as mock_popen:
        # Simuliere dass subprocess Erfolg zurückgibt
        mock_process = MagicMock()
        mock_process.stdout.readline.return_value = b""
        mock_process.poll.return_value = 0
        mock_popen.return_value = mock_process
        
        # Jetzt kannst du testen ohne echte CLI
        from bot import _run_teach
        _run_teach(12345, "Python Grundlagen")
Wann mocken? Wenn das echte System: langsam ist, externe Abhängigkeiten hat, schwer kontrollierbar ist, oder Seiteneffekte hat (Dateien schreiben, Nachrichten senden).
Vorsicht beim Mocken: Zu viel Mocking testet nur noch den Mock selbst, nicht den echten Code. Mock externe Grenzen (API, Filesystem), nicht interne Logik.

tmp_path — sicherer Filesystem-Zugriff

pytest stellt tmp_path als Fixture bereit — ein temporäres Verzeichnis das nach dem Test automatisch gelöscht wird:

def test_settings_gespeichert(tmp_path):
    settings_file = tmp_path / "settings.json"
    
    save_settings({"notifications_enabled": True}, tmp_path)
    
    # Echte Datei lesen — aber im tmp-Verzeichnis, nicht in Production
    import json
    data = json.loads(settings_file.read_text())
    assert data["notifications_enabled"] is True

🧠 Quiz

Frage 1: Was passiert mit monkeypatch-Änderungen nach dem Test?

A) Sie bleiben dauerhaft
❌ Falsch. monkeypatch macht alle Änderungen automatisch rückgängig nach jedem Test.
B) Sie werden automatisch zurückgesetzt
✅ Richtig! Das macht monkeypatch zu einem sicheren Werkzeug — kein State leakt zwischen Tests.

Frage 2: Du willst prüfen ob eine Funktion genau einmal aufgerufen wurde. Welches Werkzeug?

A) monkeypatch
❌ monkeypatch kann Funktionen ersetzen, aber nicht Aufruf-Zähler verfolgen.
B) unittest.mock mit assert_called_once()
✅ Richtig! MagicMock verfolgt alle Aufrufe automatisch.

Aufgabe: Was würdest du mocken wenn du load_settings() testen willst ohne echte Dateien zu berühren?

Zeig mir die Lösung
tmp_path nutzen — pytest gibt dir ein temporäres Verzeichnis und load_settings() schreibt/liest dort statt in Production:

def test_load_settings(tmp_path):
  result = load_settings(tmp_path)
  assert result == {"notifications_enabled": True}

📋 Zusammenfassung

  • Mock: Ersatz für externe Abhängigkeiten in Tests
  • monkeypatch: einfach, pytest built-in, automatisch rückgesetzt
  • unittest.mock: Aufrufzähler, Return-Values, Argument-Prüfung
  • tmp_path: sicherer Filesystem-Zugriff in Tests
  • Mock externe Grenzen (APIs, Files), nicht interne Logik
  • Zu viel Mocking = Test testet nur den Mock, nicht den Code
