Stille Env-Var-Fehler: os.environ.get() vs []

Ziel: Verstehen, warum fehlende Umgebungsvariablen in Python unsichtbar versagen — und wie man es verhindert.

Der Bug (Root Cause dieser Feature-Session)

# telegram_notify.py — der Bug
TOKEN = os.environ.get("TOKEN")       # Var "TOKEN" existiert nie
if not TOKEN:
    sys.exit(0)  # stilles Beenden — keine Nachricht, kein Error
Ergebnis: Script lief täglich, schickte nie eine Benachrichtigung. Kein Error-Log — nur Stille. Bug war Monate unbemerkt.

Der Unterschied

AusdruckVar vorhandenVar fehlt
os.environ["KEY"]WertKeyError — sichtbarer Crash
os.environ.get("KEY")WertNone — stilles Versagen
os.environ.get("KEY", "x")Wert"x" — expliziter Fallback

Faustregel: Für kritische Konfiguration ist [] besser — ein Crash ist sichtbar, Stille nicht.

Wann ist None-Rückgabe OK?

OK — bewusste Nicht-Aktion: 
token = get_notify_token(settings)  # None = kein Bot konfiguriert
if not token:
    sys.exit(0)  # absichtlich: kein Send wenn nicht konfiguriert
Nicht OK — unbeabsichtigtes Versagen: 
TOKEN = os.environ.get("TOKEN")  # immer None (falsche Var)
# Script macht nichts, meldet nichts, kehrt zurück mit Exit 0

Der Fix

# vorher
TOKEN = os.environ.get("TOKEN")          # falsche Var, immer None

# nachher
from core.routing import get_notify_token
token = get_notify_token(load_settings()) # routing-aware Token-Auflösung

# on_stop.py: vorher
TOKEN = os.environ.get("TELEGRAM_TOKEN")  # nie definiert

# on_stop.py: nachher
TOKEN = os.environ.get("TOKEN_PERMISSIONS")  # korrekte Var

Erkennungsmuster

grep -r 'os.environ.get' scripts/   # alle get()-Aufrufe finden
env -i python script.py             # leere Umgebung: alle fehlenden Vars crashen

Quiz

Frage: os.environ.get("MISSING") → None → if not None: sys.exit(0). Was passiert?

Antwort

Script beendet sich mit Exit 0. Kein Fehler, keine Ausgabe. Perfektes stilles Versagen.

Aufgabe: Welcher Ausdruck wirft sichtbaren Fehler wenn TOKEN_BRAIN nicht gesetzt?

Antwort
TOKEN = os.environ["TOKEN_BRAIN"]  # KeyError: 'TOKEN_BRAIN'

Zusammenfassung

  • get() gibt None — kein Fehler, stilles Versagen möglich
  • Für kritische Config: os.environ["KEY"] oder expliziter None-Check mit Logging
  • Root Cause hier: falsche Var-Namen + get() + stilles sys.exit(0)
  • Fix: routing-aware Token-Auflösung statt hartkodierter Var-Namen
