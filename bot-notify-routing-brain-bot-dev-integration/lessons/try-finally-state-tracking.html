try/finally State-Tracking

Ziel: active_session sicher setzen und garantiert löschen — auch wenn ein Subprocess fehlschlägt.

Das Problem ohne try/finally

def _run_plan(plan_text):
    result = subprocess.run(["claude", "-p", plan_text], ...)
    # Wenn subprocess crasht → active_session bleibt gesetzt
    # Nächste Session routet an falschen Bot
Problem: Subprocess-Crash → active_session nie gelöscht → nächste Session routet Fragen zum falschen Bot.

Die Lösung: try/finally

Kernprinzip: finally wird immer ausgeführt — egal ob try normal endet, eine Exception wirft, oder return aufgerufen wird. 
def _run_plan(plan_text):
    s = load_settings()
    s["active_session"] = "organizer"
    save_settings(s)

    try:
        result = subprocess.run(["claude", "-p", plan_text], ...)
        if result.returncode != 0:
            result = subprocess.run(["claude", "-p", plan_text], ...)
    finally:
        s = load_settings()       # frischer Read — nicht die alte Variable
        s["active_session"] = None
        save_settings(s)

Warum load_settings() im finally nochmal?

settings.json kann während des Subprocesses von außen geändert werden. Frischer Read verhindert Race Conditions:

# Falsch:
finally:
    s["active_session"] = None  # s ist veraltet — überschreibt externe Änderungen!
    save_settings(s)

# Richtig:
finally:
    s = load_settings()         # aktuellen Stand lesen
    s["active_session"] = None
    save_settings(s)

Quiz

Frage: Unbehandelte Exception im try-Block — wird finally noch ausgeführt?

Antwort

Ja. finally läuft immer, dann wird die Exception weitergeleitet.

Aufgabe: Warum s = load_settings() im finally statt äußere Variable weiterverwenden?

Antwort

settings.json kann während Subprocess extern geändert worden sein. Frischer Read verhindert, dass externe Änderungen überschrieben werden.

Zusammenfassung

  • finally = garantiertes State-Cleanup, auch bei Exception
  • State vor try setzen, im finally löschen
  • Im finally nochmal load_settings() — kein veralteter State
  • Für Markdown-Phasen: set_session.py am Anfang und Ende
