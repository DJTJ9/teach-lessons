Funktionen & Module

def, Default-Argumente, Rückgabewerte — und wie send_message() und run_claude() in bot.py aufgebaut sind

Funktionen deklarieren — def statt Rückgabetyp

C#

string Begruesse(string name)
{
    return $"Hallo, {name}!";
}
// void für kein Rückgabewert
void Log(string msg) { ... }

Python

def begruesse(name: str) -> str:
    return f"Hallo, {name}!"

# None = void
def log(msg: str) -> None:
    print(msg)

Ohne return-Statement gibt eine Python-Funktion automatisch None zurück — das Äquivalent zu C#s void. Type Hints (-> str, -> None) sind optional aber empfohlen.

Default-Argumente und Keyword-Argumente

def sende(chat_id: int, text: str, markup=None, timeout: int = 30) -> None:
    ...

# Alle diese Aufrufe sind gültig:
sende(123, "Hallo")                     # markup=None, timeout=30
sende(123, "Hallo", keyboard)            # positional
sende(text="Hallo", chat_id=123)        # keyword, Reihenfolge egal
sende(123, "Hallo", timeout=60)         # default überspringen
Genau wie C#s Optional Parameters und Named Arguments — gleiche Konzepte, schlankere Syntax.

Mehrere Rückgabewerte

Python gibt mehrere Werte als Tuple zurück — automatisch ausgepackt beim Empfang. In C# brauchst du out-Parameter oder (string, string) ValueTuple.

def parse_befehl(text: str) -> tuple[str, str]:
    teile = text.split(":", 1)
    cmd = teile[0].strip().lower()
    body = teile[1].strip() if len(teile) > 1 else ""
    return cmd, body

# Auspacken:
cmd, body = parse_befehl("status: sport erledigt")
# cmd = "status", body = "sport erledigt"

Module — eigene Dateien importieren

# helpers.py
def parse_befehl(text: str) -> tuple[str, str]: ...
def format_task(name: str, prio: str = "Mittel") -> str: ...

# main.py — importiert helpers.py
from helpers import parse_befehl, format_task
import helpers  # alternativ: helpers.parse_befehl(...)

Externe Pakete: pip install requests groq — wie NuGet. Abhängigkeiten in requirements.txt auflisten (wie .csproj).

🤖 bot.py erklärt — send_message() und run_claude()

send_message():

def send_message(chat_id, text, reply_markup=None):
    payload = {"chat_id": chat_id, "text": text}
    if reply_markup:
        payload["reply_markup"] = reply_markup
    requests.post(f"{BASE}/sendMessage", json=payload)

Kein Rückgabewert → implizit None (void). reply_markup=None ist Default — wird weggelassen wenn kein Keyboard nötig. Der if reply_markup:-Check funktioniert weil None in Python als False ausgewertet wird.

run_claude() — das Herzstück:

def run_claude(prompt, system_prompt=None, cwd=None):
    cmd = ["claude", "--dangerously-skip-permissions"]
    if system_prompt:
        cmd += ["--system-prompt", system_prompt]
    cmd += ["-p", prompt]
    result = subprocess.run(cmd, capture_output=True, text=True,
                            encoding="utf-8", timeout=1200, cwd=cwd or WORK_DIR)
    return (result.stdout or "").strip() or "(keine Antwort)"

Warum subprocess statt direktem LLM SDK? Die claude-CLI hat vollen MCP-Tool-Zugriff (Notion, Filesystem). Ein direkter API-Call hätte nur LLM-Zugriff ohne Notion-Integration. bot.py delegiert die gesamte Tool-Orchestrierung an Claude Code durch diesen CLI-Aufruf.

cwd or WORK_DIR: wenn cwd=None (der Default), wird WORK_DIR genutzt. Der or-Operator gibt den ersten "truthy" Wert zurück — None or WORK_DIR ergibt WORK_DIR. Kompakter als ein if cwd is None-Block.
