Strings & Textformatierung

f-Strings, mehrzeilige Strings und warum bot.py 200-zeilige System-Prompts als Konstanten schreibt

f-Strings — wie C#s $-Strings, aber mächtiger

f-Strings (Präfix f) werten beliebige Python-Ausdrücke in {} aus — nicht nur Variablen.

C# Interpolation

string name = "Claude";
int n = 3;
string s = $"Hallo {name}! {n*2} Tasks";

Python f-String

name = "Claude"
n = 3
s = f"Hallo {name}! {n*2} Tasks"

Format-Spezifizierer nach : im Platzhalter: f"{3.14159:.2f}" → "3.14". f"{1000000:,}" → "1,000,000".

Mehrzeilige Strings — Triple Quotes

Triple-Anführungszeichen (""" oder ''') erzeugen Strings die Zeilenumbrüche enthalten — ohne \n schreiben zu müssen.

C# Verbatim String

string prompt = @"Du bist ein Assistent.
Antworte kurz.
Kein Markdown.";

Python Triple-Quote

prompt = """Du bist ein Assistent.
Antworte kurz.
Kein Markdown."""
Triple-Quote kombiniert mit f-Präfix: f"""Hallo {name}, heute ist {date}.""" — das ist das Pattern das bot.py für parametrisierte Prompts nutzt.

Wichtige String-Methoden

text = "  Status: Sport erledigt  "
text.strip()              # "Status: Sport erledigt" — wie Trim()
text.lower()              # "  status: sport erledigt  " — ToLower()
text.startswith("Status") # True — StartsWith()
text.replace("Sport", "Yoga")  # Replace()
text.split(":", 1)          # ["  Status", " Sport erledigt  "]
", ".join(["a", "b", "c"])  # "a, b, c" — String.Join()
Python-Strings sind immutable — wie in C#. Methoden geben immer einen neuen String zurück, sie verändern den Original nicht.

🤖 bot.py erklärt — System-Prompts als Konstanten

Alle KI-Anweisungen in bot.py sind mehrzeilige Konstanten:

TASK_SYSTEM_PROMPT = """Du bist ein Notion-Task-Assistent. Der Nutzer schickt eine Aufgabe als Freitext.
Lege die Aufgabe im Tagesorganizer an (data_source_id: c9d2abbe-...).
Leite aus dem Text ab: Name, Datum (ISO 8601, heute falls nicht angegeben)...
Antworte NUR mit einer Zeile: ✅ Task angelegt: [Name] · [Datum] · [Priorität] · [Bereich]"""

Warum Konstanten statt Funktionen? Prompts ändern sich selten und brauchen keine Logik — eine Konstante ist direkter als eine Funktion die nur einen String zurückgibt. Wenn ein Wert einzubetten ist (z.B. eine Datenbank-ID), wird ein f-String genutzt:

MOIN_SYSTEM_PROMPT = f"""...
Lies die Habits-Datenbank (data_source_id: {HABITS_DATA_SOURCE_ID}).
..."""

normalize_voice() — String-Methoden in Aktion:

def normalize_voice(text: str) -> str:
    text = re.sub(r' Doppelpunkt\b', ':', text, flags=re.IGNORECASE)
    text = re.sub(r' Komma\b', ',', text, flags=re.IGNORECASE)
    return text

Sprachnachrichten transkribieren "Doppelpunkt" statt ":". Diese Funktion ersetzt per Regex. re.sub(pattern, repl, string) entspricht C#s Regex.Replace(). Das \b im Pattern ist eine Wortgrenze — so wird "Doppelpunkt" am Ende eines Wortes erkannt, nicht mitten in einem Wort.

Dispatcher-Logik nutzt String-Methoden intensiv:

if text.lower().startswith("status:"):
    status_text = text[7:].strip()  # Slice ab Index 7, dann trim

text[7:] ist Python-Slicing — wie text.Substring(7) in C#, aber knapper und mächtiger.
