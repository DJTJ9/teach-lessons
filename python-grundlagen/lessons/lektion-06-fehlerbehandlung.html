Fehlerbehandlung

try/except, Exception-Hierarchie, finally — und warum bot.py jeden externen Call absichert

try / except — wie C#s try / catch

C#

try {
    var x = int.Parse(input);
}
catch (FormatException e) {
    Console.WriteLine(e.Message);
}
finally {
    // immer
}

Python

try:
    x = int(eingabe)
except ValueError as e:
    print(e)
finally:
    # immer

Mehrere Exception-Typen: except (ValueError, TypeError) as e: — wie C# catch (FormatException) catch (NullReferenceException). Nacktes except: fängt alles inkl. KeyboardInterrupt — fast nie gewollt.

Wichtige Exception-Typen

ValueError        # int("abc") — falscher Wert, richtiger Typ
TypeError         # "hallo" + 42 — falscher Typ
KeyError          # d["fehlt"] — Key nicht in Dict
IndexError        # liste[99] — Index out of range
AttributeError    # None.strip() — Attribut nicht vorhanden
FileNotFoundError # open("nicht-da.txt")
subprocess.TimeoutExpired              # subprocess timeout
requests.exceptions.RequestException  # Netzwerkfehler
Alle Exceptions erben von Exception. except Exception as e: fängt alle normalen Fehler. except BaseException: fängt auch SystemExit/KeyboardInterrupt — fast nie gewollt.

Eigene Exceptions und raise

class ConfigError(Exception):
    pass

if token is None:
    raise ConfigError("TELEGRAM_TOKEN not set")

# Exception weiterwerfen (re-raise):
try:
    result = api_call()
except requests.exceptions.Timeout:
    raise  # gleiche Exception nach oben

🤖 bot.py erklärt — Fehlerbehandlung im Bot

run_claude() — TimeoutExpired absichern:

def run_claude(prompt, system_prompt=None, cwd=None):
    try:
        result = subprocess.run(
            cmd, capture_output=True, text=True,
            encoding="utf-8", timeout=1200, cwd=cwd or WORK_DIR
        )
        return (result.stdout or "").strip() or "(keine Antwort)"
    except subprocess.TimeoutExpired:
        return "⏱ Timeout — Claude hat zu lange gebraucht"
    except Exception as e:
        return f"❌ Fehler: {e}"

Warum? Bot läuft als dauerhafter Prozess. Unbehandelte Exception beendet den Bot-Loop. Bei Claude-Calls kann subprocess hängen (LLM-Timeout, Notion-API down) — ohne timeout und except würde der Bot für immer blockieren.

Der Haupt-Loop — resilient pattern:

while True:
    try:
        updates = get_updates(offset)
        for update in updates:
            handle_update(update)
    except Exception as e:
        print(f"Loop error: {e}")
        time.sleep(5)  # kurz warten, dann retry

Catch all, log, kurz warten, weitermachen. Kein Crash des gesamten Bots wegen eines einzelnen fehlerhaften Updates. In C#: gleiches Pattern mit try/catch(Exception) in while(true)-Schleife.
