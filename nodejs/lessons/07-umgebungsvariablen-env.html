🔐 Lektion 07 — Umgebungsvariablen & .env

Ziel: Secrets sicher auslagern, .env-Dateien nutzen, und process.env in Node.js verstehen — genau wie du es schon im Chatbot-Projekt siehst.

Warum Umgebungsvariablen?

Dein Chatbot nutzt bereits Umgebungsvariablen — TELEGRAM_TOKEN, GROQ_API_KEY. Das ist bewährte Praxis:

  • Secrets (API-Keys, Passwörter) gehören nicht in den Code
  • Verschiedene Umgebungen (Entwicklung, Produktion) brauchen verschiedene Werte
  • .env-Dateien werden nie ins Git committed
Sicherheitsregel: Niemals einen API-Key, Passwort, oder Token direkt in eine .js-Datei schreiben. Niemals .env in Git committen. Immer in .gitignore aufnehmen.

Vergleich: Python vs. Node.js

🐍 Python (dein Chatbot)

import os

TOKEN = os.environ.get("TELEGRAM_TOKEN")
API_KEY = os.getenv("GROQ_API_KEY")

# .env laden mit python-dotenv:
from dotenv import load_dotenv
load_dotenv()
TOKEN = os.getenv("TELEGRAM_TOKEN")

🟡 Node.js

// Node.js built-in (process.env):
const TOKEN = process.env.TELEGRAM_TOKEN;
const API_KEY = process.env.GROQ_API_KEY;

// .env laden mit dotenv:
require('dotenv').config();
const TOKEN = process.env.TELEGRAM_TOKEN;

dotenv installieren und nutzen

npm install dotenv
// Am Anfang deiner app.js (vor allem anderen):
require('dotenv').config();

// Jetzt sind alle .env Variablen in process.env verfügbar:
const express = require('express');
const app = express();

const PORT = process.env.PORT || 3000;  // Fallback auf 3000
const TOKEN = process.env.TELEGRAM_TOKEN;

if (!TOKEN) {
    console.error('TELEGRAM_TOKEN nicht gesetzt!');
    process.exit(1);  // Abbrechen wenn kritische Variable fehlt
}

app.listen(PORT, () => {
    console.log(`Server läuft auf Port ${PORT}`);
});

Konfiguration als Modul

// config.js — zentrales Konfigurations-Modul
require('dotenv').config();

const config = {
    port: Number(process.env.PORT) || 3000,
    telegramToken: process.env.TELEGRAM_TOKEN,
    debug: process.env.DEBUG === 'true',
    apiBaseUrl: process.env.API_BASE_URL || 'http://localhost:3001',
};

// Validierung:
const erforderlich = ['telegramToken'];
for (const key of erforderlich) {
    if (!config[key]) {
        throw new Error(`Konfiguration fehlt: ${key}`);
    }
}

module.exports = config;

// Nutzen in app.js:
const config = require('./config');
console.log(`Port: ${config.port}`);
Verbindung zum Chatbot: Dein Python-Chatbot macht dasselbe — conftest.py setzt TELEGRAM_TOKEN und GROQ_API_KEY für Tests. In Node.js macht dotenv das Gleiche für die Laufzeit, und in Tests würdest du process.env direkt setzen.

Umgebungsvariablen beim Ausführen

# In der Kommandozeile setzen (temporär für diesen Prozess):
TELEGRAM_TOKEN=mytoken node app.js   # Linux/Mac
$env:TELEGRAM_TOKEN="mytoken"; node app.js  # PowerShell (Windows)

🧠 Quiz

Frage 1: Warum darf .env nicht in Git committed werden?

A) Weil .env Dateien zu groß sind
❌ Falsch. .env Dateien sind winzig. Der Grund ist Sicherheit.
B) Weil sie Secrets enthalten die öffentlich werden würden
✅ Richtig! Git-History ist dauerhaft. Einmal ein Token committed, ist er kompromittiert.

Frage 2: Was gibt process.env.NICHT_GESETZT zurück?

A) null
❌ Falsch. In Node.js gibt process.env für nicht gesetzte Variablen undefined zurück.
B) undefined
✅ Richtig! Deshalb nutzt man process.env.PORT || 3000 für Fallbacks.

📋 Zusammenfassung

  • process.env.VAR = Python os.environ.get("VAR")
  • dotenv: .env-Datei in process.env laden (= Python python-dotenv)
  • .env: Secrets für lokale Entwicklung — NIE in Git
  • .env.example: Template ohne echte Werte — darf in Git
  • Fallback: process.env.PORT || 3000
  • Validation: prüfen ob kritische Variablen gesetzt sind, sonst process.exit(1)
