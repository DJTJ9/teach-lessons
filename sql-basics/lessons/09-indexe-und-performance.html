Indexe und Performance

Wann und warum Indexe einsetzen — EXPLAIN ANALYZE lesen

Was ist ein Index?

Ein Index ist eine separate Datenstruktur die der Datenbank erlaubt, Zeilen schnell zu finden ohne die gesamte Tabelle zu durchsuchen. Vergleich: Index eines Buchs — du schaust nicht jede Seite durch, du gehst direkt zur richtigen Seite.

Ohne Index: PostgreSQL liest jede Zeile sequentiell (Sequential Scan). Mit Index: PostgreSQL springt direkt zu den Treffern (Index Scan).

-- Tabelle: 1 Million Score-Eintraege -- Abfrage: alle Scores von Spieler 42 SELECT * FROM scores WHERE spieler_id = 42; -- OHNE Index: PostgreSQL liest alle 1.000.000 Zeilen ~ 500ms -- MIT Index auf spieler_id: direkt zu Zeilen springen ~ 1ms

Index erstellen

-- Einfacher Index CREATE INDEX idx_scores_spieler_id ON scores (spieler_id); -- Zusammengesetzter Index (Composite) CREATE INDEX idx_scores_spieler_level ON scores (spieler_id, level); -- Index fuer absteigende Sortierung (Highscore-Abfragen) CREATE INDEX idx_scores_punkte_desc ON scores (punkte DESC); -- UNIQUE Index (wie UNIQUE Constraint, aber explizit) CREATE UNIQUE INDEX idx_spieler_username ON spieler (username); -- Index loeschen DROP INDEX idx_scores_spieler_id;
PRIMARY KEY und UNIQUE Constraints erstellen automatisch Indexe. Du brauchst keinen manuellen Index auf id oder username wenn dort ein PK/UNIQUE liegt.

EXPLAIN ANALYZE: Performance verstehen

EXPLAIN ANALYZE SELECT * FROM scores WHERE spieler_id = 42; -- Ohne Index: Seq Scan on scores (cost=0.00..25000.00 rows=150 width=28) (actual time=0.012..487.234 rows=150 loops=1) Filter: (spieler_id = 42) Rows Removed by Filter: 999850 Planning Time: 0.1 ms Execution Time: 487.3 ms -- Mit Index auf spieler_id: Index Scan using idx_scores_spieler_id on scores (cost=0.43..482.50 rows=150 width=28) (actual time=0.020..0.891 rows=150 loops=1) Index Cond: (spieler_id = 42) Planning Time: 0.3 ms Execution Time: 0.9 ms

Seq Scan = schlecht fuer grosse Tabellen. Index Scan = gut. Die Execution Time zeigt den realen Unterschied.

Wann Indexe erstellen — und wann nicht

+
Erstelle Index auf: Foreign Keys (spieler_id in scores), Spalten in WHERE-Bedingungen die haeufig vorkommen, Spalten in ORDER BY bei grossen Tabellen, Spalten die in JOINs verwendet werden.
-
Kein Index bei: Tabellen mit weniger als ~1000 Zeilen (Seq Scan ist schneller), Spalten die selten in WHERE vorkommen, Boolean-Spalten (ist_aktiv: nur 2 Werte, Index bringt wenig), Tabellen mit sehr vielen Writes (jeder INSERT/UPDATE muss auch den Index aktualisieren).
Zu viele Indexe verlangsamen Writes (INSERT, UPDATE, DELETE). Jede Aenderung an einer Tabelle muss auch alle Indexe aktualisieren. Faustregel: 2-4 Indexe pro Tabelle sind normal, 10+ Indexe sind ein Warnsignal.

Composite Index: Reihenfolge wichtig

-- Index auf (spieler_id, level) CREATE INDEX idx ON scores (spieler_id, level); -- NUTZT den Index: SELECT * FROM scores WHERE spieler_id = 42; -- ja SELECT * FROM scores WHERE spieler_id = 42 AND level = 3; -- ja -- NUTZT den Index NICHT (erste Spalte fehlt): SELECT * FROM scores WHERE level = 3; -- nein

Composite Indexes werden von links nach rechts gelesen. Wenn die erste Spalte nicht in der WHERE-Klausel ist, wird der Index nicht genutzt.

Alle Indexe einer Tabelle anzeigen

-- In psql: \d scores -- Zeigt auch alle Indexe am Ende der Ausgabe -- Oder via SQL: SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'scores';

Quiz

1. "Seq Scan" in EXPLAIN ANALYZE bedeutet:

B — Sequential Scan = jede Zeile wird geprueft. Fuer kleine Tabellen OK. Fuer Tabellen mit vielen Zeilen und WHERE auf einer Spalte ohne Index: sehr langsam.

2. Du hast Index auf (spieler_id, punkte). Wird er bei WHERE punkte > 5000 genutzt?

B und C sagen das Gleiche. Composite Index (spieler_id, punkte): erste Spalte MUSS in WHERE sein. WHERE punkte > 5000 ohne spieler_id -> Seq Scan. Loesung: separater Index nur auf punkte erstellen.

3. Warum verlangsamen zu viele Indexe INSERT-Operationen?

A — Jeder Index ist eine separate Datenstruktur. Neue Zeile eingefuegt -> Haupttabelle aktualisieren + JEDER Index der Tabelle muss aktualisiert werden. 10 Indexe = 11 Schreiboperationen pro INSERT.
