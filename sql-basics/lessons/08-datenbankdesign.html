SQL BASICS · LEKTION 08

Datenbankdesign fuer Spiele und Apps

Beziehungen, Normalisierung, Foreign Keys — wie echte Datenbankstrukturen entstehen

Die drei Beziehungstypen

1:1 — Eins zu Eins
Spieler hat genau ein Profil. Profil gehoert zu genau einem Spieler.
spieler (id, username) spieler_profil (id, spieler_id UNIQUE, avatar_url, bio)

Selten benoetigt. Meist koennte man die Felder direkt in spieler aufnehmen. Sinnvoll wenn Profil optional oder sehr gross ist.

1:N — Eins zu Viele (am haeufigsten)
Ein Spieler hat viele Scores. Jeder Score gehoert zu genau einem Spieler.
spieler (id, username) -- 1 scores (id, spieler_id, punkte) -- N (spieler_id ist Foreign Key)

Der Foreign Key steht immer auf der "viele"-Seite. spieler_id in scores zeigt auf spieler.id.

N:M — Viele zu Viele (braucht Junction-Tabelle)
Ein Spieler kann viele Items haben. Ein Item kann vielen Spielern gehoeren.
spieler (id, username) items (id, name, typ) spieler_items (spieler_id, item_id, menge) -- Junction-Tabelle

N:M immer ueber eine Zwischentabelle (Junction Table). Die Junction hat zwei Foreign Keys, oft als zusammengesetzter Primary Key.

Foreign Keys: Datenintegritaet erzwingen

-- scores.spieler_id muss immer auf spieler.id zeigen CREATE TABLE scores ( id SERIAL PRIMARY KEY, spieler_id INTEGER NOT NULL REFERENCES spieler(id) ON DELETE CASCADE, punkte INTEGER NOT NULL ); -- ON DELETE CASCADE: Score wird automatisch geloescht wenn Spieler geloescht wird -- ON DELETE RESTRICT (Standard): Spieler kann nicht geloescht werden solange Scores existieren -- ON DELETE SET NULL: spieler_id wird NULL gesetzt
Ohne Foreign Keys kann spieler_id in scores auf eine nicht-existente spieler.id zeigen — "verwaiste Datensaetze". Foreign Keys verhindern das auf Datenbankebene, bevor es passiert.

Vollstaendiges Spiel-Schema

-- Komplettes Schema fuer ein einfaches Spiel CREATE TABLE spieler ( id SERIAL PRIMARY KEY, username TEXT NOT NULL UNIQUE, erstellt_am TIMESTAMPTZ NOT NULL DEFAULT NOW() ); CREATE TABLE items ( id SERIAL PRIMARY KEY, name TEXT NOT NULL UNIQUE, typ TEXT NOT NULL -- 'waffe', 'ruestung', 'verbrauchbar' ); CREATE TABLE scores ( id SERIAL PRIMARY KEY, spieler_id INTEGER NOT NULL REFERENCES spieler(id) ON DELETE CASCADE, punkte INTEGER NOT NULL CHECK (punkte >= 0), level INTEGER NOT NULL DEFAULT 1, erreicht_am TIMESTAMPTZ NOT NULL DEFAULT NOW() ); CREATE TABLE inventar ( spieler_id INTEGER NOT NULL REFERENCES spieler(id) ON DELETE CASCADE, item_id INTEGER NOT NULL REFERENCES items(id), menge INTEGER NOT NULL DEFAULT 1 CHECK (menge > 0), PRIMARY KEY (spieler_id, item_id) -- zusammengesetzter PK );

Normalisierung: kurze Version

Normalisierung bedeutet: Daten-Redundanz eliminieren. Fuer Junior-Dev reichen die ersten drei Normalformen:

1NF: Atomare Werte
Kein "items = 'Schwert,Schild,Trank'" in einer Spalte. Jedes Item bekommt eine eigene Zeile (in inventar).
2NF: Kein partieller Abhaengigkeit
Jede Nicht-Schluessel-Spalte haengt vom gesamten Primary Key ab. In inventar: menge haengt von (spieler_id, item_id) ab — korrekt. item.name gehoert nicht in inventar — gehoert in items.
3NF: Kein transitive Abhaengigkeit
Kein Nicht-Schluessel-Feld haengt von einem anderen Nicht-Schluessel-Feld ab. scores.spieler_name in scores waere falsch — name haengt von spieler_id ab, nicht von scores.id direkt.
Praxistipp: Wenn du dieselbe Information an zwei Stellen eintragen musst, ist das Schema falsch normalisiert. Daten sollten nur an einer Stelle leben und per JOIN abgefragt werden.

Quiz

1. Spieler-Inventar: ein Spieler kann viele Items haben, ein Item kann vielen Spielern gehoeren. Welche Beziehung?

N:M — Mehrere Spieler koennen dasselbe Item besitzen, und jeder Spieler kann mehrere Items haben. Loesung: inventar-Tabelle mit spieler_id und item_id als Foreign Keys.

2. Was macht ON DELETE CASCADE bei einem Foreign Key?

B — Spieler geloescht -> alle scores dieses Spielers werden automatisch geloescht. Ohne CASCADE wuerde PostgreSQL den DELETE verweigern solange noch Scores existieren (RESTRICT ist der Standard).

3. Warum ist es falsch, spieler_name TEXT in die scores-Tabelle zu schreiben?

B — Redundanz. Der Name steht schon in spieler.username. Wenn Alice ihren Namen aendert, muesste jede einzelne Score-Zeile aktualisiert werden. Mit Foreign Key stattdessen: immer JOIN und der aktuelle Name kommt aus spieler.
