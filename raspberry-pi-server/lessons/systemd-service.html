Bot als systemd-Service

Service-Datei schreiben, Bot als Service starten, Autostart beim Boot einrichten — echter 24/7-Betrieb.

Warum diese Lektion?

Dies ist der Kern der Mission: Der Bot soll 24/7 laufen, automatisch starten wenn der Pi hochfährt und sich nach einem Absturz selbst neu starten. Genau das macht systemd.

Was ist systemd?

systemd — der Prozess-Manager des Pi

systemd ist der Init-Dienst von Raspberry Pi OS. Er startet alle Systemdienste beim Booten (SSH, Netzwerk, etc.). Du kannst eigene Programme als systemd-Service registrieren — dann läuft der Bot genauso zuverlässig wie SSH selbst.

Schritt 1: Service-Datei schreiben

1

Service-Datei erstellen

Auf dem Pi (via SSH):

sudo nano /etc/systemd/system/telegram-notion-bot.service

Inhalt:

[Unit]
Description=Telegram Notion Bot
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=pi
WorkingDirectory=/home/pi/projekte/telegram-notion-bot
ExecStart=/home/pi/projekte/telegram-notion-bot/venv/bin/python3 bot.py
EnvironmentFile=/home/pi/projekte/telegram-notion-bot/.env
Restart=on-failure
RestartSec=10
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target

Speichern: Strg+X → Y → Enter

Was bedeuten diese Zeilen?

ZeileBedeutung
After=network-online.targetErst starten wenn Netzwerk bereit ist
User=piService läuft als normaler Nutzer, nicht als root
ExecStart=.../venv/bin/python3Python aus dem venv nutzen — kein source activate nötig
EnvironmentFile=.../.envAPI-Keys aus der .env-Datei laden
Restart=on-failureBot automatisch neu starten wenn er abstürzt
RestartSec=1010 Sekunden warten vor Neustart (verhindert Restart-Schleifen)
StandardOutput=journalLogs gehen in journalctl (Lektion 8)

Schritt 2: Service aktivieren und starten

2

systemd über neue Datei informieren

sudo systemctl daemon-reload

Muss immer ausgeführt werden wenn eine Service-Datei erstellt oder geändert wird.

3

Service aktivieren (Autostart beim Boot)

sudo systemctl enable telegram-notion-bot
4

Service sofort starten

sudo systemctl start telegram-notion-bot
5

Status prüfen

sudo systemctl status telegram-notion-bot

active (running) = Bot läuft. ✅

✅ Bot läuft als Service!

Trenne die SSH-Verbindung — Bot läuft weiter. Starte den Pi neu — Bot startet automatisch mit. Schicke eine Telegram-Nachricht zur Bestätigung.

Schritt 3: Reboot-Test

6

Pi neu starten und Bot-Status prüfen

sudo reboot

Warte 2–3 Minuten. Dann via SSH verbinden und prüfen:

sudo systemctl status telegram-notion-bot

active (running) → Autostart funktioniert. ✅

Service verwalten

BefehlBedeutung
sudo systemctl start telegram-notion-botService starten
sudo systemctl stop telegram-notion-botService stoppen
sudo systemctl restart telegram-notion-botService neu starten
sudo systemctl status telegram-notion-botStatus anzeigen
sudo systemctl disable telegram-notion-botAutostart deaktivieren

Nach git pull: sudo systemctl restart telegram-notion-bot

🧠 Wissenscheck

1. Wozu dient After=network-online.target?
Stellt sicher, dass das Netzwerk bereit ist bevor der Bot startet. Ohne diese Zeile könnte der Bot beim Booten scheitern, weil Telegram/Notion-API noch nicht erreichbar sind.
2. Was ist der Unterschied zwischen systemctl enable und systemctl start?
enable richtet den Autostart ein (nächster Boot). start startet den Service sofort. Man braucht beide: enable für die Zukunft, start für jetzt.
3. Warum nutzen wir den direkten Pfad zum venv-Python statt python3?
systemd kennt keine aktivierten venvs. Durch den direkten Pfad (venv/bin/python3) wird automatisch das richtige Python mit allen installierten Paketen genutzt.

✅ Das hast du in dieser Lektion gelernt

  • systemd ist der Prozess-Manager der Services beim Booten startet und überwacht
  • Service-Datei in /etc/systemd/system/ beschreibt wie der Bot gestartet wird
  • Restart=on-failure + RestartSec=10: Bot startet automatisch nach Absturz
  • systemctl enable + start: Bot läuft 24/7 und überlebt Neustarts
