Lektion 03

Tasks & Tagesorganizer: Die Verbindung herstellen

Lektion 3 von ~6 · Dauer: ~25 min · 2026-06-10

Was du heute lernst

Du hast Projekte und Epics in Notion. Jetzt kommt die entscheidende Frage: Wo landen die konkreten Tasks? Und wie kommen sie in deinen täglichen Workflow?

Heute treffen wir eine Design-Entscheidung — und setzen sie um.

Die Designfrage: Eigene Tasks-Datenbank?

Es gibt zwei naheliegende Ansätze. Lass uns beide anschauen:

✘ NICHT EMPFOHLEN

Eigene Tasks-DB

  • Neue DB: Tasks → Epics → Projekte
  • Tasks manuell in Tagesorganizer kopieren
  • Zwei Systeme parallel pflegen
  • Doppelarbeit bei Status-Updates
✔ EMPFOHLEN

Tagesorganizer = Task-Ebene

  • Tagesorganizer bekommt "Projekt" + "Epic"
  • Tasks landen direkt im bestehenden System
  • Ein einziger Ort für alles
  • Bot-Befehle funktionieren ohne Änderung
Kernprinzip: Der Tagesorganizer ist bereits deine Tasks-Datenbank. Er hat Status, Datum, Priorität — alles was ein Task braucht. Wir müssen ihn nur um eine Verbindung zu Projekten erweitern.

Was sich ändert

Wir fügen dem Tagesorganizer zwei neue Properties hinzu:

PropertyTypZweck
Nametitlevorhanden
Statusstatusvorhanden
Prioritätselectvorhanden
Datumdatevorhanden
Bereichselectvorhanden
Notizrich_textvorhanden
Projekt neu select Welches Projekt? z.B. "Dart-App", "Notion-Bot"
Epic neu select Welcher Themenbereich? z.B. "User Login", "Statistik"
Warum select statt relation? Eine Relation wäre sauberer, macht aber jede Task-Erstellung aufwändiger (du musst den richtigen Epic in einer anderen DB suchen). Ein Select-Feld ist schneller im Alltag — und der Bot kann genauso danach filtern.

Der Tagesorganizer bleibt dein Tages-Hub. Die neuen Properties geben jedem Task seinen Projektkontext — ohne zweites System.

Schritt 1: Properties hinzufügen

Nutze diesen Prompt in Claude Code:

Claude Code → eingebenFüge der Notion-Datenbank mit data_source_id "c9d2abbe-5607-44c2-bbf4-9aa673e0c4a0" (Tagesorganizer) zwei neue Properties hinzu: 1. "Projekt" (select): Dart-App, Notion-Bot 2. "Epic" (select): (leer lassen — ich fülle die Werte manuell)
Hinweis: Notion-MCP kann Properties zu bestehenden Datenbanken hinzufügen. Bestehende Tasks bleiben unberührt — sie bekommen die neuen Felder einfach leer.

Schritt 2: Erste Projekt-Tasks anlegen

Sobald die Properties existieren, lege Tasks für deine Dart-App-Epics an. So sehen fertige Tasks aus:

User Login Dart-App Hoch Login-Screen implementieren — Datum: heute — Status: Not started
Statistik Dart-App Mittel Würfe-Daten in Datenbank speichern — Datum: morgen — Status: Not started

Prompt für den ersten Task:

Claude Code → eingebenLege im Tagesorganizer (data_source_id: c9d2abbe-5607-44c2-bbf4-9aa673e0c4a0) einen neuen Task an: - Name: "Login-Screen implementieren" - Status: Not started - Priorität: Hoch - Datum: 2026-06-10 - Bereich: Arbeit - Projekt: Dart-App - Epic: User Login

Warum dieser Aufbau funktioniert

1
Morgen-Workflow bleibt unverändert
Bot liest Tagesorganizer wie gehabt. Projekt-Tasks tauchen automatisch mit auf.
2
Bot kann nach Projekt filtern
Befehl dart: zeig offene Tasks → Bot filtert Tagesorganizer auf Projekt = "Dart-App".
3
Epic-Kontext bleibt erhalten
Du siehst beim Task direkt, zu welchem Themenbereich er gehört — ohne in die Epics-DB zu wechseln.

Praxis-Aufgabe

Führe diese drei Schritte durch:

  • Schritt 1: Properties "Projekt" und "Epic" zum Tagesorganizer hinzufügen (Prompt oben nutzen)
  • Schritt 2: 2–3 Tasks für deine Dart-App-Epics anlegen — mit Datum heute
  • Schritt 3: Morgen-Workflow testen: moin an den Bot — tauchen die neuen Tasks auf?

Ziel: Mindestens ein Projekt-Task im Tagesorganizer, sichtbar im Morgen-Workflow.

Quiz – Kurz nachdenken
