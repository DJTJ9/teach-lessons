2. Indexierungs-Script: Projekt einlesen

Dieses Script liest alle .cs-Dateien aus deinem Unity-Projekt, teilt sie in Chunks auf und speichert sie in ChromaDB.

# indexer.py — einmalig ausführen wenn sich Dateien ändern
import os
import chromadb
from langchain_text_splitters import RecursiveCharacterTextSplitter
import ollama

# --- Konfiguration ---
PROJECT_DIR = r"C:\Projekte\MeinUnityProjekt\Assets\Scripts"
DB_DIR      = r"C:\Projekte\MeinUnityProjekt\.rag_db"
COLLECTION  = "unity_codebase"

# --- Alle .cs-Dateien finden ---
all_files = []
for root, _, files in os.walk(PROJECT_DIR):
    for f in files:
        if f.endswith(".cs"):
            all_files.append(os.path.join(root, f))

print(f"Gefunden: {len(all_files)} .cs-Dateien")

# --- Text laden ---
texts, metas = [], []
for path in all_files:
    try:
        content = open(path, encoding="utf-8").read()
        texts.append(content)
        metas.append({"source": os.path.relpath(path, PROJECT_DIR)})
    except Exception as e:
        print(f"Fehler bei {path}: {e}")

# --- Chunking ---
splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,
    chunk_overlap=80,
    separators=["\n\n", "\n", "{", "}", " "],
)
chunks, chunk_metas = [], []
for text, meta in zip(texts, metas):
    splits = splitter.split_text(text)
    chunks.extend(splits)
    chunk_metas.extend([meta] * len(splits))

print(f"Chunks erstellt: {len(chunks)}")

# --- Embeddings via Ollama ---
def embed_batch(texts_batch):
    resp = ollama.embed(model="nomic-embed-text", input=texts_batch)
    return resp.embeddings

# --- ChromaDB speichern ---
client = chromadb.PersistentClient(path=DB_DIR)
# Alte Collection löschen falls vorhanden (Re-Indexierung)
try: client.delete_collection(COLLECTION)
except: pass
collection = client.create_collection(COLLECTION)

BATCH = 50
for i in range(0, len(chunks), BATCH):
    batch_texts = chunks[i:i+BATCH]
    batch_metas = chunk_metas[i:i+BATCH]
    batch_embeds = embed_batch(batch_texts)
    batch_ids = [f"chunk_{i+j}" for j in range(len(batch_texts))]
    collection.add(
        documents=batch_texts,
        embeddings=batch_embeds,
        metadatas=batch_metas,
        ids=batch_ids,
    )
    print(f"Indexiert: {i+len(batch_texts)}/{len(chunks)}")

print("Indexierung abgeschlossen!")
Separators für C#: { und } als Trenner hinzugefügt — so versucht der Splitter, an Methoden/Klassen-Grenzen zu trennen statt mitten in Code. Ergibt semantisch kohärentere Chunks.

3. Query-Script: Fragen stellen

Dieses Script nimmt deine Frage, findet die relevantesten Code-Abschnitte und gibt sie zusammen mit dem LLM-Output aus.

# query.py — bei jeder Frage ausführen
import chromadb
import ollama

DB_DIR     = r"C:\Projekte\MeinUnityProjekt\.rag_db"
COLLECTION = "unity_codebase"
LLM_MODEL  = "qwen2.5-coder:7b-instruct-q4_K_M"
TOP_K      = 3  # Wie viele Chunks als Kontext

def query_rag(question: str):
    # 1. Frage embedden
    q_embed = ollama.embed(model="nomic-embed-text", input=[question]).embeddings[0]

    # 2. Ähnlichste Chunks finden
    client = chromadb.PersistentClient(path=DB_DIR)
    coll = client.get_collection(COLLECTION)
    results = coll.query(query_embeddings=[q_embed], n_results=TOP_K, include=["documents", "metadatas"])

    # 3. Kontext zusammenbauen
    context_parts = []
    for doc, meta in zip(results["documents"][0], results["metadatas"][0]):
        context_parts.append(f"--- {meta['source']} ---\n{doc}")
    context = "\n\n".join(context_parts)

    # 4. LLM-Prompt mit Kontext
    prompt = f"""Du bist ein C#/Unity-Experte. Beantworte die Frage basierend auf dem folgenden Code:

{context}

Frage: {question}

Antworte auf Deutsch. Beziehe dich konkret auf den gezeigten Code."""

    # 5. LLM antworten lassen
    print("\n=== Kontext gefunden ===")
    for meta in results["metadatas"][0]:
        print(f"  · {meta['source']}")

    print("\n=== Antwort ===")
    for chunk in ollama.chat(model=LLM_MODEL, messages=[{"role": "user", "content": prompt}], stream=True):
        print(chunk.message.content, end="", flush=True)
    print()

if __name__ == "__main__":
    import sys
    q = " ".join(sys.argv[1:]) if len(sys.argv) > 1 else input("Frage: ")
    query_rag(q)

Nutzung:

# Einmalig indexieren:
python indexer.py

# Dann fragen:
python query.py "Wie funktioniert der PlayerController?"
python query.py "Welche Klassen erben von MonoBehaviour?"
python query.py "Wo wird OnCollisionEnter verwendet?"

4. Re-Indexierung bei Codeänderungen

Wann immer du deinen Code signifikant änderst, führe indexer.py erneut aus. Die alte Collection wird automatisch gelöscht und neu aufgebaut.

Performance: Ein Unity-Projekt mit 100 .cs-Dateien indexiert in 1–3 Minuten (CPU-Embeddings). ChromaDB legt den Index unter .rag_db/ ab — diese ~50-200 MB bleiben dauerhaft. Re-Queries dauern <1 Sekunde.

Quiz

1. Warum wird die ChromaDB-Collection im indexer.py vor dem Neu-Indexieren gelöscht?

A) ChromaDB unterstützt keine Updates — alles muss immer neu angelegt werden
B) Beim Re-Indexieren sollen alte Chunks (von gelöschten/umbenannten Dateien) nicht als Zombies in der DB bleiben
C) Es spart VRAM, da alte Vektoren aus dem Speicher entfernt werden
Wenn du eine Datei umbenennst oder löschst, ist ihr alter Chunk noch in der DB. Bei einer Suche würde er trotzdem auftauchen und auf eine nicht mehr existierende Datei zeigen. Einfachste Lösung: Collection löschen und komplett neu aufbauen. Für große Projekte gibt es effizientere Strategien (Timestamps, Hashes), aber für den Start ist Delete+Rebuild am robustesten.

2. Du erhöhst TOP_K von 3 auf 10. Was ist der Tradeoff?

A) Schnellere Antworten, weil mehr Kontext = weniger Nachdenken für das LLM
B) Mehr relevante Chunks im Kontext, aber auch mehr Rauschen — und längerer Prompt braucht mehr Kontext-Fenster und erzeugt langsamere Antworten
C) Kein Unterschied — LLMs ignorieren irrelevante Teile des Kontexts automatisch
Mehr Chunks bedeuten: bessere Chance, relevante Info zu finden (gut), aber auch mehr irrelevanter Text im Prompt (schlecht). LLMs haben begrenzte Kontextfenster (z.B. 8192 Tokens) und werden langsamer mit mehr Kontext. TOP_K=3-5 ist ein guter Startpunkt; erhöhe auf 5-8 wenn Antworten zu wenig Kontext haben.

3. Du willst auch Markdown-Dokumentation (.md) und Unity-Asset-Beschreibungen indexieren. Was musst du ändern?

A) Im Datei-Scan f.endswith(".cs") zu f.endswith((".cs", ".md", ".txt")) erweitern — alles andere bleibt gleich
B) Ein separates Embedding-Modell für Markdown-Dateien verwenden
C) Separate ChromaDB-Collections für jeden Dateityp anlegen
Das Embedding-Modell und die Vector-Suche funktionieren für beliebigen Text — egal ob C#, Markdown oder JSON. Du musst nur die Dateierweiterungen im Scan-Teil erweitern. Alle Chunks landen in derselben Collection und werden gemeinsam durchsucht. Das Modell versteht sowohl Code als auch natürlichsprachliche Doku.
