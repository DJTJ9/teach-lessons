Fine-Tuning Grundlagen — QLoRA und Unsloth

Wie du ein Modell auf deinen eigenen Daten trainierst — auf der RTX 3070 Ti

1. Was ist Fine-Tuning?

Ein vortrainiertes Modell (z.B. Qwen2.5-Coder) hat allgemeines Wissen. Fine-Tuning spezialisiert es weiter — durch Training auf deinen eigenen Beispielen. Das Modell lernt Muster, die im Vortraining nicht explizit vertreten waren.

MethodeWas sie kannWas sie nicht kann
PromptingVerhalten über Anweisung steuernTiefe Anpassung von Stil/Wissen
RAGEigene Dokumente als KontextEchtes Lernen, Stilanpassung
Fine-TuningStil, proprietäre APIs, SpezialverhaltenWissen nach Trainingscut-off, billiges Update

Gute Fine-Tuning-Anwendungsfälle

  • Modell soll immer s&box-spezifische APIs statt Unity-APIs nutzen
  • Bestimmter Kommentar- und Naming-Stil deines Teams
  • Proprietäres Framework, das im Internet kaum dokumentiert ist
  • Antworten auf einer bestimmten Sprache / in einem bestimmten Format erzwingen

2. Full Fine-Tuning vs. LoRA vs. QLoRA

Full Fine-Tuning (alle Gewichte aktualisieren) braucht GPU-Cluster. Auf einem Consumer-Laptop nicht möglich. Die Lösung: LoRA (Low-Rank Adaptation).

Originale Gewichte W (eingefroren, nicht verändert) + LoRA-Adapter A × B (sehr kleine Matrizen, werden trainiert) = Effektive Gewichte W + A×B W hat Milliarden Parameter. A×B hat nur Millionen — z.B. 0,1-1% der Originalgewichte.

QLoRA = Quantized LoRA. Die Originalgewichte werden in 4-Bit quantisiert geladen (statt 16-Bit), was VRAM um ~75% reduziert. Die LoRA-Adapter selbst laufen in 16-Bit für gute Gradienten.

MethodeVRAM (7B-Modell)Qualität
Full Fine-Tuning~80 GB+Bestmöglich
LoRA (16-Bit)~28 GBSehr gut
QLoRA (4-Bit)~6-8 GBGut (leicht schlechter als LoRA)
RTX 3070 Ti (8GB VRAM): QLoRA auf einem 7B-Modell ist machbar — braucht ~6-7 GB VRAM mit Batch-Size 1. Ein 13B-Modell braucht bereits ~12 GB und passt nicht mehr.

3. Unsloth — schnelleres Training

Unsloth ist eine Python-Library, die QLoRA-Training deutlich beschleunigt (2-5x gegenüber vanilla HuggingFace). Es optimiert CUDA-Kernels und VRAM-Nutzung speziell für LoRA-Training.

# Installation (CUDA 11.8 oder 12.1+)
pip install unsloth
# Bei CUDA-Problemen: spezifische Version
pip install "unsloth[cu121-torch240] @ git+https://github.com/unslothai/unsloth.git"
Windows-Einschränkung: Unsloth läuft auf Windows nur über WSL2 (Windows Subsystem for Linux 2) oder direkt in einer Linux-Umgebung. Empfehlung: WSL2 mit Ubuntu 22.04 einrichten — CUDA-Zugriff auf die RTX 3070 Ti funktioniert dort problemlos.

4. Das Datensatz-Format

Fine-Tuning-Daten bestehen aus Instruktions-Paaren: eine Eingabe und die erwartete Ausgabe. Das gebräuchlichste Format für Instruktions-Tuning:

// Alpaca-Format (JSONL — eine Zeile = ein Beispiel)
{
  "instruction": "Erkläre was diese Methode macht",
  "input": "private void OnTriggerEnter(Collider other) { if (other.CompareTag(\"Player\")) { StartCoroutine(SpawnEnemy()); } }",
  "output": "Die Methode erkennt wenn ein Collider die Trigger-Zone betritt. Falls der Collider den Tag 'Player' hat, startet eine Coroutine zum Spawnen eines Gegners."
}

// Chat-Format (für Instruction-Following-Modelle bevorzugt)
{
  "messages": [
    {"role": "user", "content": "Schreibe eine s&box-Component die Health trackt"},
    {"role": "assistant", "content": "public sealed class HealthComponent : Component { [Property] public float MaxHealth { get; set; } = 100f; ... }"}
  ]
}

6. Der Training-Loop (Übersicht)

1
Basis-Modell laden (4-Bit quantisiert mit Unsloth)
2
LoRA-Adapter anlegen (Rank, Target Modules konfigurieren)
3
Datensatz laden (JSONL → tokenisieren)
4
Training starten (Forward Pass → Loss → Backprop auf Adapter)
5
Adapter speichern (wenige 100 MB, nicht das ganze Modell)
6
Merger + GGUF-Export (Adapter + Basis → quantisierte GGUF für Ollama)

Quiz

1. Warum ist QLoRA für die RTX 3070 Ti (8GB) möglich, full Fine-Tuning aber nicht?

A) QLoRA braucht keine GPU, es läuft auf der CPU
B) QLoRA friert die quantisierten Originalgewichte ein (4-Bit, ~4GB) und trainiert nur kleine Adapter-Matrizen — der VRAM-Bedarf sinkt von ~80GB auf ~6-8GB
C) QLoRA komprimiert das Modell dauerhaft kleiner, so dass es nie wieder mehr VRAM braucht
Full Fine-Tuning muss Gradienten für alle Milliarden Parameter speichern — das braucht vielfaches des Modell-VRAMs. QLoRA lädt das Basismodell in 4-Bit (statt 16-Bit, ~4x weniger VRAM) und trainiert nur die kleinen LoRA-Adapter in 16-Bit. Die Qualitätsverluste durch 4-Bit sind beim Training-Input gering, weil die Adapter die Anpassungen tragen.

2. Du hast 50 selbstgeschriebene Beispiel-Paare. Ist das genug für sinnvolles Fine-Tuning?

A) Ja — weniger ist mehr beim Fine-Tuning, 50 Beispiele sind ideal
B) Nur für sehr enge Stil-Anpassungen. Für echte Spezialisierung sind 100-500+ hochqualitative Beispiele realistisch
C) Nein — Fine-Tuning ist erst ab 10.000 Beispielen sinnvoll
50 Beispiele können reichen, wenn das Ziel sehr eng ist (z.B. "immer mit 'Du' antworten") und die Beispiele perfekt sind. Aber das Modell overfittet leicht auf so wenige Beispiele — es "lernt" die 50 Beispiele auswendig statt zu generalisieren. Für eine echte API-Spezialisierung sind 200-500+ Beispiele ein realistisches Minimum. Starte mit wenigen, prüfe die Ergebnisse, füge mehr hinzu.

3. Was ist der Unterschied zwischen dem gespeicherten LoRA-Adapter und dem fertigen Merged-Modell?

A) Der Adapter (paar 100 MB) enthält nur die Anpassungen; das Merged-Modell (mehrere GB) verbindet Adapter + Originalgewichte zu einer eigenständigen GGUF-Datei für Ollama
B) Es gibt keinen Unterschied — beide sind vollständige Modelle
C) Der Adapter ist das fertige Modell; Merging ist optional und verschlechtert die Qualität
Der LoRA-Adapter allein ist nur ein "Patch" — er braucht das Basismodell zum Laufen (Adapter + Basismodell werden zur Inference zusammengeführt). Für Ollama braucht man eine standalone GGUF-Datei. Dazu: Adapter in das Basismodell "einmergen" → ein vollständiges Modell → zu GGUF konvertieren → per Modelfile in Ollama laden.
