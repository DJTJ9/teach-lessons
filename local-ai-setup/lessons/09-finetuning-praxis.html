2. Datensatz erstellen — s&box Beispiele

Der Datensatz ist das Herzstück des Fine-Tunings. Hier: Beispiele für s&box-spezifische C#-Components.

# Datensatz als dataset.jsonl anlegen (Chat-Format)
# Jede Zeile = ein Trainingsbeispiel

// Beispiel 1
{"messages": [{"role": "user", "content": "Schreibe eine s&box Component die einen Timer implementiert"}, {"role": "assistant", "content": "public sealed class TimerComponent : Component\n{\n\t[Property] public float Duration { get; set; } = 5f;\n\t[Property] public Action OnTimerEnd { get; set; }\n\n\tprivate TimeUntil _timeUntil;\n\n\tprotected override void OnStart()\n\t{\n\t\t_timeUntil = Duration;\n\t}\n\n\tprotected override void OnUpdate()\n\t{\n\t\tif ( _timeUntil <= 0 )\n\t\t{\n\t\t\tOnTimerEnd?.Invoke();\n\t\t\tEnabled = false;\n\t\t}\n\t}\n}"}]}

// Beispiel 2
{"messages": [{"role": "user", "content": "s&box: Health-System mit Damage-Events"}, {"role": "assistant", "content": "public sealed class HealthComponent : Component\n{\n\t[Property, Sync] public float Health { get; private set; } = 100f;\n\t[Property] public float MaxHealth { get; set; } = 100f;\n\n\t[Property] public Action<float> OnDamaged { get; set; }\n\t[Property] public Action OnDeath { get; set; }\n\n\tpublic void TakeDamage( float amount )\n\t{\n\t\tHealth = Math.Max( 0, Health - amount );\n\t\tOnDamaged?.Invoke( amount );\n\t\tif ( Health <= 0 ) OnDeath?.Invoke();\n\t}\n}"}]}
Tipp: Minimum 100-200 solcher Beispiele für sinnvolle Spezialisierung. Schreibe sie aus deinem echten Projekt-Code — das gibt dem Modell deinen spezifischen Stil.

3. Training-Script (Unsloth + QLoRA)

# train.py — in WSL2 ausführen
from unsloth import FastLanguageModel
from datasets import load_dataset
from trl import SFTTrainer
from transformers import TrainingArguments

# --- Modell laden (4-Bit QLoRA) ---
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name = "unsloth/Qwen2.5-Coder-7B-Instruct-bnb-4bit",
    max_seq_length = 2048,
    load_in_4bit = True,
)

# --- LoRA-Adapter konfigurieren ---
model = FastLanguageModel.get_peft_model(
    model,
    r = 16,
    lora_alpha = 32,
    lora_dropout = 0.05,
    target_modules = ["q_proj", "k_proj", "v_proj", "o_proj",
                      "gate_proj", "up_proj", "down_proj"],
    use_gradient_checkpointing = "unsloth",
)

# --- Datensatz laden und formatieren ---
dataset = load_dataset("json", data_files="dataset.jsonl", split="train")

def format_chat(sample):
    messages = sample["messages"]
    return {"text": tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=False)}

dataset = dataset.map(format_chat)

# --- Training starten ---
trainer = SFTTrainer(
    model = model,
    tokenizer = tokenizer,
    train_dataset = dataset,
    dataset_text_field = "text",
    max_seq_length = 2048,
    args = TrainingArguments(
        per_device_train_batch_size = 1,
        gradient_accumulation_steps = 4,
        num_train_epochs = 3,
        learning_rate = 2e-4,
        fp16 = True,
        logging_steps = 10,
        output_dir = "./output",
        save_strategy = "epoch",
    ),
)
trainer.train()

# --- Adapter speichern ---
model.save_pretrained("./sbox-adapter")
tokenizer.save_pretrained("./sbox-adapter")
print("Training abgeschlossen! Adapter unter ./sbox-adapter")
VRAM-Monitoring: Führe nvidia-smi -l 1 in einem zweiten Terminal aus, um VRAM-Auslastung live zu sehen. Typisch: 6-7.5 GB beim Training mit diesen Einstellungen.

4. Export nach GGUF und Ollama

Schritt 1: Adapter mergen und zu GGUF exportieren

# export.py — nach dem Training
from unsloth import FastLanguageModel

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name = "./sbox-adapter",
    load_in_4bit = True,
)

# Direkt als GGUF exportieren (Unsloth macht Merge + Konvertierung intern)
model.save_pretrained_gguf(
    "sbox-qwen-7b",
    tokenizer,
    quantization_method = "q4_k_m"  # oder "q5_k_m" für bessere Qualität
)
# Erzeugt: sbox-qwen-7b-Q4_K_M.gguf

Schritt 2: Modell in Ollama registrieren

# Modelfile erstellen
cat > Modelfile << 'EOF'
FROM ./sbox-qwen-7b-Q4_K_M.gguf

SYSTEM """
Du bist ein erfahrener C#-Entwickler spezialisiert auf s&box (Facepunch).
Nutze immer s&box-spezifische APIs: Component-System, GameResource, [Property]-Attribute, Sync-Attribute.
Antworte auf Deutsch mit sauberem, idomatischem s&box-C#.
"""

PARAMETER temperature 0.3
PARAMETER num_ctx 4096
EOF

# In Ollama laden
ollama create sbox-assistant -f Modelfile

# Testen
ollama run sbox-assistant "Schreibe eine Component für Inventar-Verwaltung"

6. Kombination: Fine-Tuning + RAG

Die stärkste Konfiguration kombiniert beide Ansätze: ein fine-getuntes Modell (das deinen Coding-Stil kennt) plus RAG (das aktuellen Projekt-Code nachschlägt).

Empfohlener Stack:
  1. Qwen2.5-Coder auf s&box-Beispielen fine-getuned → versteht dein Framework
  2. ChromaDB-Index deines aktuellen Projekts → kennt deinen spezifischen Code
  3. query.py schickt relevante Code-Chunks als Kontext → präzise, projektspezifische Antworten

Quiz

1. Du trainierst 3 Epochs und siehst, dass das Modell auf neue Fragen wortwörtlich Trainings-Beispiele antwortet. Was ist die wahrscheinlichste Ursache?

A) Learning Rate zu niedrig — das Modell hat nichts gelernt
B) Overfitting — das Modell hat die Trainingsbeispiele auswendig gelernt statt zu generalisieren, wahrscheinlich zu wenig Datensatz-Vielfalt
C) Das ist normales Verhalten — Fine-Tuning soll Beispiele memorieren
Wenn ein Modell Trainingsbeispiele wortwörtlich reproduziert, ist es overfittet. Ursachen: zu wenige Beispiele (Modell sieht dieselben zu oft), zu viele Epochs, zu hohe Learning Rate. Fixes: mehr Daten, num_train_epochs auf 1-2 senken, lora_dropout erhöhen, learning_rate auf 1e-4 reduzieren.

2. Warum nutzt das Training-Script gradient_accumulation_steps=4 bei per_device_train_batch_size=1?

B) Simuliert eine effektive Batch-Size von 4 ohne 4x VRAM zu brauchen — Gradienten werden über 4 Schritte akkumuliert bevor ein Optimizer-Schritt gemacht wird
A) Macht das Training 4x schneller durch Parallelisierung
C) Ist eine Sicherheitseinstellung gegen Speicherüberläufe, hat keine Auswirkung auf Qualität
Mit Batch-Size 1 und 8GB VRAM wäre die effektive Batch-Size zu klein — Training wird instabil. Gradient Accumulation löst das: 4 Forward Passes mit je 1 Sample, Gradienten addieren, dann ein einziger Optimizer-Schritt mit akkumulierten Gradienten. Effekt identisch zu Batch-Size 4, aber nur VRAM für 1 Sample gleichzeitig.

3. Dein fine-getuntes Modell ist in Ollama geladen. Wie integrierst du es am besten mit dem RAG-System aus Lektion 7?

A) RAG und Fine-Tuning schließen sich aus — du musst dich für eines entscheiden
B) Das fine-getunte Modell braucht kein RAG mehr, es kennt deinen Code bereits
C) Im query.py einfach LLM_MODEL auf das neue Modell setzen — das fine-getunte Modell versteht s&box-Patterns besser, RAG liefert weiterhin aktuellen Projekt-Code als Kontext
Fine-Tuning und RAG ergänzen sich: Das fine-getunte Modell "denkt" besser in s&box-Patterns (durch Training gelernt). RAG liefert den aktuellen Projektstand (durch Indexierung). Im query.py nur LLM_MODEL = "sbox-assistant" setzen — alles andere bleibt. Das ist der beste Stack: Stil durch Fine-Tuning, Fakten durch RAG.
