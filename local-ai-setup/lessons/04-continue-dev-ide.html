Continue.dev — Lokaler Copilot in der IDE

Qwen2.5-Coder direkt in VS Code oder Rider — Tab-Completion, Chat, Code-Auswahl ohne Cloud

1. Was ist Continue.dev?

Continue.dev ist ein Open-Source IDE-Plugin, das GitHub Copilot ersetzt — aber mit lokalen Modellen via Ollama. Es läuft in VS Code und JetBrains-IDEs (Rider, IntelliJ) und bietet dieselben Kern-Features wie Copilot, ohne Cloud-Abo.

FeatureGitHub CopilotContinue.dev + Ollama
Tab-Completion✓ (Cloud)✓ (lokal)
Chat in IDE✓ (Cloud)✓ (lokal)
Code auswählen + erklären✓ (Cloud)✓ (lokal)
DatenschutzCode geht an Microsoft/OpenAIAlles bleibt lokal
Kosten10–19€/Monat0€ laufend
Offline-NutzungNeinJa

2. Installation in VS Code

1
Extensions öffnen (Ctrl+Shift+X) → "Continue" suchen → Continue - Codestral, Claude, and more installieren
2
Continue-Panel erscheint rechts → ⚙ Configure Continue klicken → öffnet ~/.continue/config.json
3
config.json anpassen (siehe nächster Abschnitt)

Performance-Erwartung: Tab-Completion auf einem lokalen 7B-Modell dauert ~0,5–2 Sekunden pro Vorschlag — langsamer als GitHub Copilot (Cloud). Der debounceDelay: 500 verhindert, dass jeder Tastendruck sofort eine GPU-Anfrage auslöst.

6. @-Kontext-Provider: Eigene Dateien laden

Im Chat mit @ verschiedene Kontext-Quellen referenzieren:

@-BefehlWas geladen wird
@currentFileGesamte aktuelle Datei
@file PlayerController.csSpezifische Projektdatei
@terminalLetzter Terminal-Output (z.B. Compiler-Fehler)
@code MyClass.MyMethodSpezifische Methode oder Klasse
Praxis-Tipp: Unity-Build-Fehler aufgetreten? Im Continue-Chat @terminal + "Was bedeutet dieser Fehler und wie behebe ich ihn?" — das Modell liest die Fehlermeldung direkt. Kein manuelles Copy-Paste.

Quiz

1. Warum ist debounceDelay: 500 in der Autocomplete-Konfiguration sinnvoll?

A) Erhöht Qualität durch 500ms extra Denkzeit
B) Wartet 500ms nach dem letzten Tastendruck bevor eine GPU-Anfrage startet — verhindert Anfrage bei jedem getippten Zeichen
C) Gibt dem Modell 500ms mehr VRAM-Zeit
Debounce: Warte N ms nach dem letzten Ereignis. Bei 500ms: Erst wenn 0,5 Sekunden nach dem letzten Tastendruck vergangen sind, wird eine Completion-Anfrage gestartet. Wichtig für flüssiges Tippen — ohne Debounce würde jedes Zeichen eine GPU-Anfrage auslösen.

2. Du nutzt VS Code und Rider gleichzeitig. Musst du Continue.dev in beiden separat konfigurieren?

A) Nein — beide lesen dieselbe ~/.continue/config.json, Konfiguration wird geteilt
B) Ja — jede IDE hat eine eigene Konfigurationsdatei
C) Nur VS Code nutzt config.json, Rider nutzt XML
Continue.dev speichert die Konfiguration im Benutzerverzeichnis (~/.continue/config.json). Sowohl VS Code als auch JetBrains-Plugins lesen diese Datei — einmal konfigurieren, überall verfügbar.

3. Unity-Compilerfehler in der Konsole. Schnellster Weg, das Modell um Hilfe zu bitten?

A) Fehlertext manuell in den Chat copy-pasten
B) Im Continue-Chat @terminal eingeben — fügt automatisch den letzten Terminal-Output als Kontext ein
C) Fehlerdatei als .txt speichern und hochladen
Der @terminal Kontext-Provider fügt den letzten Output des integrierten Terminals automatisch in den Chat-Kontext ein. Dann einfach "Erkläre diesen Fehler" schreiben — kein Copy-Paste.

4. Was ist der größte Vorteil gegenüber GitHub Copilot für dieses Setup?

A) Schneller als Copilot
B) Kennt Unity-APIs besser
C) Code verlässt nie den eigenen Rechner — kein Cloud-Zugriff auf Quellcode, offline-fähig, keine laufenden Kosten
Hauptvorteil: Privatsphäre + Offline-Fähigkeit + Kostenkontrolle. Copilot ist oft schneller und besser bei allgemeinen Coding-Tasks. Aber für ein unabhängiges, offline-fähiges, dauerhaft kostenloses Setup ist Continue.dev + lokales Modell die richtige Wahl.
