Anti-Patterns und haeufige Fehler

Schlechte Specs erkennen, benennen und reparieren

Specs die schlecht sind, fallen meistens in eines von fuenf Mustern. Wer diese Muster kennt, kann sie in fremden (und eigenen) Specs sofort erkennen.

Anti-Pattern 1: The God Spec

Symptom: Eine Spec beschreibt das gesamte Produkt
Feature-Boundaries werden nicht gesetzt. Eine Spec deckt Login, Datenbankdesign, UI und Deployment gleichzeitig ab. Niemand kann sie vollstaendig umsetzen.
# Unser Telegram-Bot — Requirement Spec ## Anforderungen - FR-01: Bot soll Nachrichten empfangen - FR-02: Bot soll Notion verbinden - FR-03: Bot soll Erinnerungen senden - FR-04: Bot soll Suche haben - FR-05: Bot soll Dart-Daten migrieren ...
Pro Feature eine Spec. "suche: Dispatcher" ist ein Feature. "Erinnerungen" ist ein Feature. Trennen.

Anti-Pattern 2: Implementation Requirement

Symptom: Requirement Spec enthaelt Klassen, Bibliotheken, Dateinamen
Die Loesung wird in der Anforderung festgeschrieben bevor das Design ueberhaupt begonnen hat. Verhindert alternative Loesungsansaetze.
## Funktionale Anforderungen - FR-01: Das System MUSS SQLite als Datenbank verwenden - FR-02: Die Klasse NotionClient MUSS als Singleton implementiert werden - FR-03: handlers/search.py MUSS den MessageHandler verwenden
Requirements beschreiben Nutzerverhalten und Systemverhalten — nie Implementierung. "Das System MUSS Daten persistent speichern" statt "Das System MUSS SQLite verwenden".

Anti-Pattern 3: Orphan Design

Symptom: Design Spec existiert ohne Requirement Spec
Der Entwickler sprang direkt in technische Details. Man weiss wie es gebaut wird, aber nicht mehr warum. Spaetere Frage "Brauchen wir das wirklich?" kann nicht mehr mit Requirements beantwortet werden.
# Search-Handler — Design Spec (kein Verweis auf Requirement Spec) ## Komponenten - handlers/search.py mit MessageHandler... [Zwei Monate spaeter: "Warum haben wir das gebaut?"]
Jede Design Spec hat im Header einen Verweis auf die Requirement Spec. Wenn kein Requirement existiert: erst eins schreiben.

Anti-Pattern 4: Macro Task

Symptom: Ein Task beruehrt 5+ Dateien
Ein Task der "das gesamte Feature implementiert" ist kein Task — es ist eine Mini-Design-Spec. KI-Agenten treffen dabei Dutzende implizite Architektur-Entscheidungen ohne Uebersicht.
TASK-01: Implementiere das komplette suche:-Feature Dateien: bot.py, handlers/search.py, services/notion_search.py, config.py, tests/test_search.py, requirements.txt
Wenn ein Task mehr als 2-3 Dateien beruehrt: aufteilen. Ein Task = eine Datei oder eine zusammenhaengende Funktion.

Anti-Pattern 5: Vanishing Out-of-Scope

Symptom: Kein "Nicht im Scope"-Abschnitt
Ohne expliziten Out-of-Scope wird jede verwandte Idee automatisch als "in Scope" behandelt. Feature-Creep passiert dann im Gespraech ("koennen wir noch schnell...") oder still durch den KI-Agenten.
# suche: Dispatcher — Requirement Spec ## Anforderungen [keine Out-of-Scope-Sektion] [KI-Agent implementiert auch Suche-History, Export-Funktion und Filteroption — weil "sinnvoll erschien"]
Immer einen "Nicht im Scope"-Abschnitt. Mindestens 2-3 Punkte die man sich vorstellen koennte aber bewusst auslaesst.
Wenn du eine Spec schreibst und merkst dass du in ein Anti-Pattern gleitest — das ist normal. Die Kunst ist es zu erkennen und zu stoppen, nicht es von Anfang an perfekt zu machen.

Spec-Review-Checkliste

Requirement Spec

Problem klar beschrieben (Wer hat das Problem? Warum?)
Ziel messbar (kein "schnell", "benutzerfreundlich", "gut")
Jedes FR hat MUSS/SOLL/KANN
Kein Implementierungsdetail in FRs
Out-of-Scope vorhanden mit mindestens 2 Punkten

Design Spec

Verweis auf Requirement Spec vorhanden
Alle Komponenten gelistet mit Verantwortlichkeit
Datenfluss Schritt-fuer-Schritt (nicht nur "X ruft Y auf")
Schnittstellen als Signaturen (kein vollstaendiger Code)
Entscheidungen mit Begruendung dokumentiert

Task Spec

Max. 1-2 Dateien pro Task
Abhaengigkeiten explizit angegeben
Akzeptanzkriterien als Checkliste (testbar, Ja/Nein)
KI-Kontext: verfuegbare Services, Nicht-aendern-Liste
Verweis auf Design Spec und Datenfluss-Schritt

Quiz

1. "Das System MUSS Python 3.11 verwenden und SQLite3 als Datenbank." — Welches Anti-Pattern?

Implementation Requirement — Requirements nennen Technologie-Entscheidungen die erst im Design getroffen werden sollten. "Python" und "SQLite" gehoeren in die Design Spec.

2. Ein Task beruehrt bot.py, handlers/search.py, services/notion_search.py, config.py und requirements.txt. Problem?

Macro Task. Jede Datei sollte ihr eigener Task sein. Wenn bot.py und requirements.txt kleine Aenderungen sind, koennen sie zusammen sein — aber handlers/search.py + services/notion_search.py sind jeweils eigene Tasks.

3. Welche Spec-Checkliste prueft "Kein Implementierungsdetail in FRs"?

Requirement Spec — FRs sind Functional Requirements, die nur in Requirement Specs vorkommen. Dort gilt: keine Implementierungsdetails.
