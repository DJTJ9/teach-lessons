AI SPECS · LEKTION 05

Die Spec-Kette in der Praxis

Ein Feature komplett von Requirement bis Tasks durchspecen — am Beispiel "Erinnerungen"

Der Prozess

Die Spec-Kette ist kein Wasserfall den man einmal komplett durchlaeuft. Es ist ein iterativer Prozess: grob spechen, dann verfeinern. Aber die Reihenfolge Req > Design > Tasks ist immer gleich.

1

Problem identifizieren

Was fehlt? Wer braucht es? Warum jetzt? Das ist die Eingabe fuer die Requirement Spec.

2

Requirement Spec schreiben (30-60 Min)

WAS soll das System koennen? Messbarer Zielzustand, FRs mit MUSS/SOLL/KANN, Out of Scope.

3

Design Spec ableiten (45-90 Min)

WIE wird es gebaut? Komponenten, Datenfluss, Signaturen, Entscheidungen.

4

Tasks ableiten (30 Min)

Jeden Schritt im Datenfluss in einen atomaren Task uebersetzen. Abhaengigkeiten eintragen.

5

KI implementiert Tasks sequentiell

Jeden Task einzeln an KI-Agent uebergeben. Akzeptanzkriterien pruefen. Naechsten Task starten.

REQUIREMENT Schritt 2: Was soll das Erinnerungen-Feature koennen?
# Erinnerungen — Requirement Spec ## Problem Nutzer vergessen Tasks und Termine, die sie im Telegram-Bot angelegt haben. Es gibt keine automatische Benachrichtigung zu faelligen Punkten. ## Ziel Nutzer erhalten um eine selbst festgelegte Uhrzeit taeglich eine Telegram-Nachricht mit ihren heutigen Tasks und Terminen. ## Funktionale Anforderungen - FR-01: Das System MUSS taeglich um eine konfigurierte Uhrzeit eine Zusammenfassung senden - FR-02: Die Zusammenfassung MUSS alle heutigen Tasks (Tagesorganizer) enthalten - FR-03: Nutzer MUSS die Uhrzeit per Bot-Befehl aendern koennen - FR-04: Nutzer KANN die Erinnerung per Befehl deaktivieren - FR-05: Die Uhrzeit SOLL persistent gespeichert werden (ueberleben Neustart) ## Nicht-funktionale Anforderungen - NFR-01: Nachricht kommt innerhalb von 1 Minute der konfigurierten Uhrzeit an - NFR-02: Kein Absturz wenn Notion API nicht erreichbar — Fehler als Nachricht ## Nicht im Scope - Push-Benachrichtigungen an mehrere Nutzer - Anpassbare Nachrichteninhalte (Format ist fix) - Integration mit externen Kalender-Apps
DESIGN Schritt 3: Wie wird es technisch umgesetzt?
# Erinnerungen — Design Spec Referenz: Requirement Spec Erinnerungen ## Uebersicht Python APScheduler sendet taeglich zur konfigurierten Uhrzeit eine Notion-Tagesansicht als Telegram-Nachricht. Konfiguration wird in reminders.json persistent gespeichert. ## Komponenten - `services/reminder.py` (neu) — Scheduler-Setup, Job-Logik - `handlers/reminder_cmd.py` (neu) — /erinnere HH:MM, /keine-erinnerung - `reminders.json` (neu) — persistente Konfiguration {chat_id: "HH:MM"} - `bot.py` (veraendert) — Scheduler-Start beim Bot-Start ## Datenfluss Setup: 1. Bot-Start: reminder.py laedt reminders.json 2. Fuer jede chat_id: APScheduler-Job anlegen mit cron-Trigger Taeglich: 3. APScheduler triggert send_reminder(chat_id) 4. Laedt heutige Tasks aus Notion (Tagesorganizer, Datum=heute) 5. Formatiert als Telegram-Nachricht 6. Sendet an chat_id Konfiguration: 7. /erinnere 08:00 -> handler schreibt in reminders.json, aktualisiert Job 8. /keine-erinnerung -> handler loescht Eintrag, entfernt Job ## Schnittstellen - `setup_scheduler(app: Application) -> AsyncIOScheduler` - `send_reminder(chat_id: int, bot: Bot) -> None` - `handle_set_reminder(update, context) -> None` - `handle_cancel_reminder(update, context) -> None` ## Entscheidungen - APScheduler: bereits bekanntes Scheduling-Muster, leichtgewichtig - reminders.json: einfach, kein Datenbank-Overhead fuer single-user Setup - Cron-Trigger statt Interval: praezise Uhrzeit-Kontrolle ## Offene Fragen - Zeitzone: Europe/Berlin hardcoded oder konfigurierbar?
TASKS Schritt 4: Atomare Tasks aus dem Datenfluss ableiten
TASK-01: Erstelle reminders.json Schema und Lese/Schreib-Helfer in services/reminder.py Abh: keine | Datei: services/reminder.py (neu) - Funktion load_reminders() -> dict - Funktion save_reminders(data: dict) -> None - Bei fehlendem File: leeres dict zurueckgeben TASK-02: Implementiere APScheduler-Setup in services/reminder.py Abh: TASK-01 | Datei: services/reminder.py - setup_scheduler(app) erstellt AsyncIOScheduler - Laedt alle Eintraege aus reminders.json - Erstellt CronTrigger-Job pro Eintrag - Gibt scheduler-Instanz zurueck TASK-03: Implementiere send_reminder() in services/reminder.py Abh: TASK-01 | Datei: services/reminder.py - Laedt heutige Tasks aus Notion (Datum=heute, Status!=Done) - Formatiert als Markdown-Nachricht - Sendet via bot.send_message(chat_id) TASK-04: Erstelle handlers/reminder_cmd.py mit beiden Command-Handlern Abh: TASK-01, TASK-02 | Datei: handlers/reminder_cmd.py (neu) - handle_set_reminder: parst HH:MM, speichert, aktualisiert Job - handle_cancel_reminder: loescht Eintrag aus JSON, entfernt Job TASK-05: Integriere Scheduler und Handler in bot.py Abh: TASK-02, TASK-04 | Datei: bot.py - Importiere setup_scheduler und Handler - Starte Scheduler nach application.initialize() - Registriere /erinnere und /keine-erinnerung CommandHandler
Erkennst du das Muster? Jeder Schritt im Datenfluss der Design Spec wird zu genau einem Task. "Datenfluss-Schritt = Task" ist eine gute Faustregel.

Quiz

1. Du hast eine Requirement Spec aber keine Design Spec. Du fangst trotzdem mit Tasks an. Was passiert?

B — Ohne Design Spec gibt es keinen vereinbarten Datenfluss. TASK-01 und TASK-02 koennen inkompatible Schnittstellen bauen ohne es zu merken.

2. Wie viele Tasks sollte "Implementiere den kompletten Erinnerungen-Bot" sein?

B — Pro Datenfluss-Schritt ein Task. Hier: load/save, scheduler-setup, send_reminder, command-handler, integration = 5 Tasks. Das ist die richtige Granularitaet.

3. Was ist die richtige Reihenfolge der Spec-Kette?

C — Requirement zuerst (Was?), dann Design (Wie?), dann Tasks (Mach!). Die Kette kann nur in diese Richtung fliessen.
