AI SPECS · LEKTION 04

Task Spec schreiben

Atomare, KI-ausfuehrbare Aufgaben aus der Design Spec ableiten

Was eine Task Spec ist

Eine Task Spec beschreibt eine einzige, abgeschlossene Aenderung am Code. Sie ist so konkret, dass ein KI-Agent sie ohne Rueckfragen umsetzen kann. Das Ziel: Null Interpretationsspielraum.

Faustregel: Wenn ein Task in mehr als einer Datei oder mehr als einer Funktion etwas tut, ist er zu gross. Aufteilen.

Struktur einer Task Spec

# TASK-[NR]: [Verb] [Was] in [Wo] Kontext: (welche Design Spec, welcher Schritt im Datenfluss) Abhaengigkeiten: (TASK-NR die vorher erledigt sein muessen) Beschreibung: Exakt was zu tun ist. Eine Aktion pro Satz. Dateien: - `pfad/datei.py` — was dort geaendert/erstellt wird Akzeptanzkriterien: - [ ] Kriterium 1 (testbar) - [ ] Kriterium 2 (testbar) - [ ] Kriterium 3 (testbar)

Die 4 Eigenschaften eines guten Tasks

EigenschaftBedeutungTest
AtomarEine Aufgabe, eine AenderungNur eine Datei oder eine Funktion?
TestbarAkzeptanzkriterien sind pruefbarKann man Ja/Nein sagen?
KontextuellKI weiss Warum und WoKein Nachfragen noetig?
GeordnetAbhaengigkeiten explizitKann in beliebiger Reihenfolge gestartet werden wenn Deps erledigt?

Beispiel: 4 Tasks fuer das suche:-Feature

TASK-01: Erstelle handlers/search.py mit handle_search()
Abh: keineDatei: handlers/search.py (neu)
Erstelle neue Datei handlers/search.py.
Implementiere handle_search(update: Update, context: CallbackContext) -> None.
Extrahiere query aus update.message.text nach Praefix "suche: ".
Rufe (noch Stub) search_all_dbs(query) auf und sende Ergebnis mit update.message.reply_text().

Akzeptanzkriterien:
  • Datei existiert und ist syntaktisch korrekt
  • Funktion akzeptiert Update und CallbackContext
  • Query wird korrekt extrahiert (Test: "suche: Dart" -> query="Dart")
TASK-02: Erstelle services/notion_search.py mit search_all_dbs()
Abh: keineDatei: services/notion_search.py (neu)
Erstelle services/notion_search.py.
Implementiere search_all_dbs(query: str) -> dict[str, list[dict]].
Iteriere ueber alle 5 DB-IDs aus config.py.
Rufe fuer jede DB notion_client.search(query=query, filter={"property":"object","value":"page"}) auf.
Gib dict zurueck mit DB-Name als Key und max 5 Ergebnissen als Value.

Akzeptanzkriterien:
  • Funktion gibt dict mit genau 5 Keys zurueck (eine pro DB)
  • Jede Liste enthaelt max 5 Eintraege
  • Bei API-Fehler: leere Liste fuer diese DB, kein Crash
TASK-03: Implementiere format_results() in notion_search.py
Abh: TASK-02Datei: services/notion_search.py
Ergaenze services/notion_search.py um format_results(grouped: dict) -> str.
Formatiere pro DB: Ueberschrift mit DB-Name, darunter nummerierte Titel-Liste.
Bei leerem grouped: return "Nichts gefunden."

Akzeptanzkriterien:
  • Ausgabe enthaelt DB-Namen als Ueberschriften
  • Leere Datenbanken werden uebersprungen
  • Bei globalem Leer-Ergebnis: "Nichts gefunden."
TASK-04: Registriere Handler in bot.py
Abh: TASK-01, TASK-02, TASK-03Datei: bot.py
Importiere handle_search aus handlers.search in bot.py.
Registriere MessageHandler(filters.Regex(r"^suche: "), handle_search) bei application.
Position: nach den bestehenden CommandHandlern, vor dem Fallback-Handler.

Akzeptanzkriterien:
  • Import vorhanden in bot.py
  • Handler registriert mit korrektem Regex-Pattern
  • Bestehende Handler unveraendert

Zu gross vs. zu klein: die Balance finden

x Zu gross

"Implementiere das gesamte suche:-Feature"

Kein klarer Scope. KI macht eigene Architektur-Entscheidungen.

OK Atomar

"Erstelle handlers/search.py mit handle_search()"

Eine Datei, eine Funktion, klare Kriterien.

x Zu klein

"Schreibe Zeile 1 von handlers/search.py"

Overhead uebersteigt Nutzen. Micro-Tasks erzeugen Tausende von Tasks.

OK Richtig

Eine Funktion oder eine zusammenhaengende Gruppe von Zeilen in einer Datei.

Quiz

1. Warum ist "Implementiere den Suchbefehl" eine schlechte Task Spec?

B — Ohne konkrete Dateinamen, Funktionssignaturen und Akzeptanzkriterien muss die KI (oder du) alles selbst interpretieren. Das fuehrt zu inkonsistenten Loesungen.

2. Wozu dienen Akzeptanzkriterien in einer Task Spec?

B — Ohne Kriterien gibt es kein "Done". Die KI implementiert bis sie aufhoert, nicht bis sie fertig ist. Kriterien sind die Definition of Done.

3. TASK-04 haengt von TASK-01, 02, 03 ab. Was bedeutet das praktisch?

A — TASK-04 importiert Code aus TASK-01 und ruft Services aus TASK-02/03 auf. Ohne die Vorgaenger schlaegt TASK-04 mit ImportError fehl.

Naechste Lektion: Die Spec-Kette in der Praxis — ein Feature komplett von Req bis Tasks durchspecen.
