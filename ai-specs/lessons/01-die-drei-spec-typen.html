AI SPECS · LEKTION 01

Die drei Spec-Typen

Requirement · Design · Task — wer sie schreibt, wer sie liest, was sie beantworten

Das Kernprinzip

Jede Spec beantwortet eine andere Frage. Wenn du die drei Fragen kennst, weisst du immer welche Spec du gerade brauchst.

Requirement: Was?
→
Design: Wie?
→
Task: Mach!

Die drei Typen sind eine Kette. Ohne Requirement weiss das Design nicht, was es loesen soll. Ohne Design weiss der Task nicht, wie er es umsetzen soll.

Typ 1: Requirement Spec

REQUIREMENT
Was soll das System tun?
Zielgruppe: du selbst, Stakeholder, KI-Agent der plant

Beschreibt das Problem und die gewuenschten Faehigkeiten — nicht die Loesung. Kein Code, keine Architektur. Nur: Was braucht der Nutzer?

Beispiel — Telegram Bot Feature

"Der Bot soll Nutzern ermoeglichen, neue Aufgaben per Chat-Nachricht in Notion anzulegen. Der Nutzer soll Titel, Prioritaet und Datum angeben koennen. Der Bot bestaetigt die Erstellung."

Typ 2: Design Spec

DESIGN
Wie wird es gebaut?
Zielgruppe: du (als Entwickler), KI-Agent der implementiert

Uebersetzt Requirements in technische Entscheidungen. Welche Komponenten? Welche APIs? Welcher Datenfluss? Noch kein Code — aber konkret genug, dass ein Entwickler weiss wo er anfangen soll.

Beispiel — Telegram Bot Feature

"Handler handle_task in handlers/task.py. Parst Nachricht mit Regex task: <titel> [prio:hoch] [datum:YYYY-MM-DD]. Ruft notion_client.create_page() auf. Antwortet mit Inline-Keyboard fuer Bestaetigung."

Typ 3: Task Spec

TASK
Was genau ist zu tun?
Zielgruppe: KI-Agent der ausfuehrt, oder du beim Sprint-Planning

Atomare, ausfuehrbare Einheit. Ein Task = eine Aenderung, eine Datei, eine Funktion. Kann von einer KI ohne Rueckfragen umgesetzt werden. Abhaengigkeiten sind explizit.

Beispiel — Telegram Bot Feature

"Erstelle handlers/task.py mit Funktion handle_task(update, context). Importiere notion_client aus services/notion.py. Gib None zurueck. Registriere Handler in bot.py Zeile 47."

Im AI-Engineering-Kontext bist du oft alle drei Rollen gleichzeitig. Die Spec-Hierarchie hilft dir, nicht alles auf einmal zu denken — erst Was, dann Wie, dann Mach.

Haeufigster Fehler: Requirements und Design vermischen

x Vermischt

"Der Bot soll einen CommandHandler fuer /task registrieren und Notion API aufrufen."

OK Getrennt

Req: "Nutzer kann Task per Bot anlegen."
Design: "CommandHandler, Notion API."

Wenn Requirements Implementierungsdetails enthalten, verlierst du die Freiheit verschiedene Loesungsansaetze zu vergleichen.

Quiz

1. "Die App soll Spieler-Highscores speichern und nach Punktzahl sortiert anzeigen." Welcher Spec-Typ?

Requirement Spec — beschreibt WAS (speichern + anzeigen), nicht WIE (kein Schema, keine API, keine Klassen).

2. "Erstelle Tabelle scores(id SERIAL, player TEXT, points INT). Fuege Index auf points DESC hinzu." Welcher Typ?

Task Spec — atomar, ausfuehrbar, konkret bis auf Spaltennamen. Ein KI-Agent koennte das direkt umsetzen.

3. Was fehlt zwischen Requirement und Task?

Design Spec. Ohne sie wuerden verschiedene Entwickler (oder KI-Agenten) inkonsistente Architekturen bauen.

Naechste Lektion: Requirement Spec schreiben — Struktur, haeufige Fehler, Beispiel am Telegram-Bot.
