AI SPECS · LEKTION 02

Requirement Spec schreiben

Struktur, Regeln und ein vollstaendiges Beispiel am Telegram-Bot

Was eine Requirement Spec enthalten muss

Eine Requirement Spec hat genau fuenf Abschnitte. Jeder Abschnitt beantwortet eine spezifische Frage.

# [Feature-Name] — Requirement Spec ## Problem Wer hat das Problem? Was ist das Problem? Warum existiert es? ## Ziel Was soll nach der Umsetzung moeglich sein? (Ein Satz, messbar) ## Funktionale Anforderungen - FR-01: Das System MUSS ... [Pflicht] - FR-02: Das System SOLL ... [Wunsch] - FR-03: Das System KANN ... [Optional] ## Nicht-funktionale Anforderungen - NFR-01: Antwortzeit unter 3 Sekunden - NFR-02: Fehler werden dem Nutzer klar kommuniziert ## Nicht im Scope - Was explizit NICHT umgesetzt wird (verhindert Scope Creep)

Die drei Schluesselwoerter: MUSS / SOLL / KANN

Diese Unterscheidung ist wichtig. Nicht alle Anforderungen haben das gleiche Gewicht.

WortBedeutungKonsequenz
MUSSPflicht — ohne geht nichtBlocker fuer Release
SOLLStark gewuenschtWichtig, aber verhandelbar
KANNNice-to-haveNur wenn Zeit vorhanden

Die 4 Regeln

1

Loesungsunabhaengig: Beschreibe WAS, nicht WIE. Kein Code, keine Architektur, keine Bibliotheken.

2

Messbar: "schnell" ist keine Anforderung. "unter 3 Sekunden" ist eine.

3

Eine Anforderung pro FR: "Das System soll X und Y" → aufteilen in FR-01 und FR-02.

4

Out of Scope explizit: Was NICHT gemacht wird, ist genauso wichtig wie was gemacht wird.

Vollstaendiges Beispiel: suche:-Feature des Telegram-Bots

# suche: Dispatcher — Requirement Spec ## Problem Nutzer des Telegram-Bots moechten Inhalte aus ihren Notion-Datenbanken finden, ohne Notion oeffnen zu muessen. Aktuell gibt es keinen einheitlichen Sucheinstieg im Bot. ## Ziel Nutzer koennen per "suche: <Begriff>" alle fuenf Notion-Datenbanken durchsuchen und Ergebnisse innerhalb von 5 Sekunden erhalten. ## Funktionale Anforderungen - FR-01: Der Bot MUSS auf Nachrichten mit Praefix "suche: " reagieren - FR-02: Der Bot MUSS Tagesorganizer, Backlog, Lernthemen, Spieleideen und Archiv durchsuchen - FR-03: Ergebnisse MUESSEN nach Datenbank gruppiert angezeigt werden - FR-04: Leere Ergebnisse MUESSEN dem Nutzer klar kommuniziert werden - FR-05: Der Bot SOLL maximal 5 Ergebnisse pro Datenbank zeigen - FR-06: Der Bot KANN direkte Links zu gefundenen Seiten anzeigen ## Nicht-funktionale Anforderungen - NFR-01: Antwort in unter 5 Sekunden - NFR-02: Fehler (Notion API nicht erreichbar) werden als klare Fehlermeldung zurueckgegeben, nicht als leere Antwort ## Nicht im Scope - Volltextsuche innerhalb von Seiteninhalten (nur Titelsuche) - Suche in anderen Workspaces - Filterfunktionen (nach Datum, Status etc.)
Bemerkst du: Die Spec erwaehnt keine Python-Funktion, keine Notion-API-Methode, keine Dateistruktur. Das kommt erst in der Design Spec.

Gut vs. Schlecht

x Schlecht

"FR-01: Die Funktion handle_search() in handlers/search.py soll die Notion API mit client.search() aufrufen."

OK Gut

"FR-01: Der Bot MUSS auf Nachrichten mit Praefix 'suche: ' reagieren und die Notion-Datenbanken durchsuchen."

Quiz

1. Welche dieser Anforderungen ist loesungsunabhaengig formuliert?

B — "persistent speichern" beschreibt WAS ohne WIE. A und C nennen bereits konkrete Implementierungsdetails.

2. Was fehlt in dieser Anforderung: "Der Bot soll schnell antworten"?

B — "schnell" ist nicht testbar. Niemand kann pruefen ob "schnell" erfuellt ist. "Unter 3 Sekunden" schon.

3. Warum ist der "Nicht im Scope"-Abschnitt wichtig?

B — Ohne expliziten Out-of-Scope wachsen Features unkontrolliert. "Koennen wir noch schnell X hinzufuegen?" wird durch Out-of-Scope klar mit Nein beantwortet.
