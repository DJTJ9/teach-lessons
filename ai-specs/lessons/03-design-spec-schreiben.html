Design Spec schreiben

Technische Entscheidungen, Datenfluss und Komponenten — ohne Code zu schreiben

Was eine Design Spec leistet

Die Design Spec ist die Bruecke zwischen "Was der Nutzer will" (Requirement) und "Was der Entwickler baut" (Tasks). Sie beantwortet: Welche Architektur loest das Problem? Welche Komponenten braucht es? Wie fliessen Daten durch das System?

Sie enthaelt noch keinen Code — aber alle Entscheidungen die Code-Struktur bestimmen.

Struktur der Design Spec

# [Feature-Name] — Design Spec Referenz: [Link zur Requirement Spec] ## Uebersicht Ein Satz: Wie loest diese Implementierung das Problem aus der Requirement Spec? ## Komponenten Liste aller neuen oder veraenderten Module/Klassen/Funktionen: - `modul/datei.py` — Verantwortlichkeit - `modul/anderes.py` — Verantwortlichkeit ## Datenfluss Schritt-fuer-Schritt: Was passiert wenn der Nutzer X tut? 1. Nutzer sendet Nachricht 2. Handler X empfaengt und parst 3. Service Y ruft API Z auf 4. Ergebnis wird formatiert und zurueckgesendet ## Schnittstellen Signaturen der wichtigsten Funktionen (kein Implementation-Code): - `search_all_dbs(query: str) -> list[SearchResult]` - `format_results(results: list[SearchResult]) -> str` ## Entscheidungen Warum diese Architektur? Welche Alternativen wurden verworfen? ## Offene Fragen Was muss noch geklaert werden?

Datenfluss visualisieren

ASCII-Flussdiagramme sind in Design Specs valide und oft besser lesbar als formale UML-Diagramme:

Nutzer: "suche: Dart" | v [bot.py] MessageHandler | erkennt Praefix "suche: " v [handlers/search.py] handle_search(update, context) | extrahiert query = "Dart" v [services/notion_search.py] search_all_dbs(query) | ruft 5x notion_client.search() auf (parallel) v [services/notion_search.py] format_results(results) | gruppiert nach DB, max 5 pro DB v [bot.py] update.message.reply_text(formatted) | v Nutzer sieht Ergebnis-Nachricht

Vollstaendiges Beispiel: suche:-Feature

# suche: Dispatcher — Design Spec Referenz: Requirement Spec suche-command ## Uebersicht Neuer MessageHandler in bot.py leitet "suche: "-Nachrichten an einen dedizierten Search-Service weiter, der alle 5 Notion-DBs parallel abfragt und gruppierte Ergebnisse zurueckgibt. ## Komponenten - `handlers/search.py` (neu) — Handler-Logik, Eingabe-Validierung - `services/notion_search.py` (neu) — Notion-API-Calls, Ergebnis-Aggregation - `bot.py` (veraendert) — Handler-Registrierung, Zeile ~50 ## Datenfluss 1. Nutzer sendet Nachricht mit Praefix "suche: " 2. MessageHandler in bot.py matched auf filters.Regex("^suche: ") 3. handle_search() extrahiert query (alles nach "suche: ") 4. search_all_dbs(query) ruft notion_client.search() fuer jede DB auf 5. Ergebnisse werden nach DB gruppiert, je max. 5 Items 6. format_results() baut Telegram-Antworttext 7. Bei leerem Ergebnis: "Nichts gefunden fuer: {query}" ## Schnittstellen - `handle_search(update: Update, context: CallbackContext) -> None` - `search_all_dbs(query: str, dbs: list[str]) -> dict[str, list[dict]]` - `format_results(grouped: dict[str, list[dict]]) -> str` ## Entscheidungen - Sequentielle API-Calls (nicht parallel): einfachere Fehlerbehandlung, 5 Calls x ~500ms = ~2.5s bleibt unter dem 5s-Limit aus NFR-01 - Eigene Datei fuer Search-Handler: folgt bestehender Konvention (handlers/) - Max 5 Ergebnisse: Telegram-Nachrichten-Limit, ausreichend fuer Use Case ## Offene Fragen - Sollen Links (page_url) als Inline-Buttons oder als Text angezeigt werden?
Design Spec enthaelt noch keinen Python-Code — nur Signaturen. Das ist Absicht: Signaturen fixieren das Interface, lassen aber Implementation offen. Eine KI kann damit sofort Tasks ableiten.

ADRs: Entscheidungen dokumentieren

Architecture Decision Records (ADRs) sind kleine Eintrage die erklaeren WARUM eine Entscheidung getroffen wurde. In der Design Spec werden sie inline im "Entscheidungen"-Abschnitt gehalten:

Entscheidung: Sequentielle statt paralleler Notion-Calls
Warum: Einfachere Fehlerbehandlung. Bei parallelen Calls muss jeder Fehler einzeln behandelt werden. Sequentiell kann ein try/except alle failen lassen. Performance-Ziel (5s) wird auch sequentiell erreicht.
Entscheidung: Neue Datei handlers/search.py statt Erweiterung von bot.py
Warum: Konvention im Repo — jede Handler-Gruppe hat eigene Datei. bot.py wird nicht weiter aufgeblaehen.

Quiz

1. Was unterscheidet eine Design Spec von einer Requirement Spec?

B — Requirement: was der Nutzer braucht. Design: wie das technisch umgesetzt wird. Code kommt erst in der Implementierung.

2. Wozu dient der "Entscheidungen"-Abschnitt in einer Design Spec?

B — Ohne ADRs weiss niemand (auch du in 3 Monaten nicht) warum eine Entscheidung getroffen wurde. Das fuehrt zu "warum haben wir das so gemacht?" und unnoetigem Refactoring.

3. Welche Information gehoert NICHT in eine Design Spec?

B — Fertiger Code gehoert in die Implementierung, nicht in die Spec. Signaturen (Interfaces) sind OK — sie definieren Vertraege ohne Implementation zu erzwingen.

Naechste Lektion: Task Spec schreiben — atomare, KI-ausfuehrbare Aufgaben aus der Design Spec ableiten.
