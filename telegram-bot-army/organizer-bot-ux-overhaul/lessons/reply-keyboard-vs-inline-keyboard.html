Reply-Keyboard vs Inline-Keyboard

Telegram Bot UI · Organizer Bot UX Overhaul

Zwei Keyboard-Typen

📌 ReplyKeyboardMarkup

  • Ersetzt die Telegram-Schreibzeile
  • Buttons senden Text-Messages
  • Persistent — ein Keyboard pro Chat
  • Kein callback_query

⬜ InlineKeyboardMarkup

  • Hängt an einer einzelnen Nachricht
  • Buttons senden callback_data
  • Mehrere gleichzeitig möglich
  • Benötigt answer_callback_query

JSON-Struktur

# ReplyKeyboardMarkup
REPLY_KEYBOARD = {
    "keyboard": [
        ["📋 Task", "📅 Termin", "💡 Ideen"],
        ["📚 Lern", "🌅 Morgen", "🌙 Abend"],
    ],
    "resize_keyboard": True,
    "persistent": True,   # Telegram 6.9+ — bleibt immer sichtbar
}

# InlineKeyboardMarkup — pro Nachricht
inline_kb = {"inline_keyboard": [[
    {"text": "✅ Erledigt",    "callback_data": f"done:{pid}"},
    {"text": "📅 Verschieben", "callback_data": f"reschedule:{pid}"},
]]}

Update-Flow-Unterschied

# Reply → normale text message
if "message" in upd:
    text = upd["message"]["text"]   # "📋 Task"
    if text in BUTTON_MAP:
        start_workflow(BUTTON_MAP[text], chat_id)

# Inline → callback_query
if "callback_query" in upd:
    cq = upd["callback_query"]
    answer_callback_query(TOKEN, cq["id"])  # Pflicht!
    data = cq["data"]                       # "done:abc123"

Entscheidungsframework

Reply-Keyboard wenn:
  • Aktionen immer verfügbar sind (Haupt-Navigation)
  • Single-User-Bot
  • Kein Daten-Transport im Button nötig
Inline-Keyboard wenn:
  • Buttons kontextspezifisch (z.B. pro Task)
  • Daten im Button: callback_data = f"done:{page_id}"
  • Buttons nach Klick editierbar (edit_message)
  • Multi-User-Chat

⚠️ callback_data max 64 Bytes

Notion page_id = 32 Hex-Zeichen. Mit Prefix "done:" (5) = 37 Bytes ✓. Lange Prefixes können das Limit knacken.

🎯 Quiz

1. User klickt "📋 Task" im Reply-Keyboard — wo landet das Update?

Antwort
Als message.text = "📋 Task". Kein callback_query. Handler prüft if t in BUTTON_MAP.

2. Warum sind die ✅-Buttons im Morgen-Workflow Inline statt Reply?

Antwort
Jeder Button trägt eine andere Notion page_id als callback_data. Reply-Buttons senden nur festen Text — kein Daten-Transport möglich.

3. 5-User Gruppen-Chat — warum ist Reply-Keyboard hier problematisch?

Antwort
ReplyKeyboardMarkup gilt für den ganzen Chat. Alle 5 User sehen und klicken dieselben Buttons — Bot kann Aktionen nicht einem User zuordnen.

📖 Summary

  • Reply: Persistent, Text-Messages, global per Chat → Haupt-Navigation in Single-User-Bots
  • Inline: Nachrichtengebunden, callback_data, editierbar → kontextspezifische Aktionen mit Daten
  • Kombination: Reply für Navigation + Inline für aufgabenspezifische Aktionen ist bewährt
  • answer_callback_query nach jedem Inline-Click zwingend
  • callback_data Limit: 64 Bytes — kompakte Prefixes wählen
