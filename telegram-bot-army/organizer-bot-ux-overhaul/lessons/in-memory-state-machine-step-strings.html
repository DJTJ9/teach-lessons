In-Memory State Machine mit Step-Strings

Organizer Bot UX Overhaul · Workflow-Architektur

Das Problem

Multi-Step-Dialoge in Telegram brauchen State zwischen Nachrichten. callback_data-Carrier scheitert bei Freitext-Steps. FSM-Klassen sind Overkill für 4 Workflows × 3 Schritte.

Datenstruktur

# Global — ein Entry pro aktivem Chat
_workflow: dict = {}

# Struktur:
_workflow[chat_id] = {
    "step": "task:name",          # "kind:expected_input"
    "data": {"name": "PR Review"},  # gesammelter Input
}

# Step-String Konvention:
# "task:name"      → wartet auf Freitext
# "task:priority"  → wartet auf Inline-Button-Callback
# "lern:name"      → wartet auf Freitext
# "lern:kategorie" → wartet auf Callback

Workflow starten

def start_workflow(kind: str, chat_id: int) -> None:
    _workflow[chat_id] = {"step": f"{kind}:init", "data": {}}
    _abort = [[{"text": "✗ Abbrechen", "callback_data": "wf:abort"}]]

    if kind == "task":
        _workflow[chat_id]["step"] = "task:name"
        send_message(TOKEN, chat_id, "📋 Name?",
            reply_markup={"inline_keyboard": _abort})

    elif kind == "morgen":
        # Kein multi-step → direkt ausführen + clearen
        _send_moin_messages(json.loads(run_claude_parse(...)))
        _workflow.pop(chat_id, None)

Text-Input verarbeiten

def handle_workflow_step(text: str, chat_id: int, today: str) -> bool:
    wf = _workflow.get(chat_id)
    if not wf:
        return False   # kein aktiver Workflow

    step = wf["step"]
    if step == "task:name":
        wf["data"]["name"] = text.strip()
        wf["step"] = "task:priority"
        buttons = [[
            {"text": "🔴 Hoch",   "callback_data": "task:priority:hoch"},
            {"text": "🟡 Mittel", "callback_data": "task:priority:mittel"},
        ], [{"text": "✗ Abbrechen", "callback_data": "wf:abort"}]]
        send_message(TOKEN, chat_id, f"Priorität für '{text.strip()}'?",
            reply_markup={"inline_keyboard": buttons})
        return True   # konsumiert

    elif step == "task:priority":
        return True   # Callback-Step — Text trotzdem konsumieren

    return False

Callback-Steps in _handle_callback()

elif data.startswith("task:priority:"):
    prio_key = data.split(":")[2]  # "hoch" / "mittel" / "niedrig"
    wf = _workflow.get(chat_id, {})
    name = wf.get("data", {}).get("name", "")
    prio = {"hoch": "Hoch", "mittel": "Mittel", "niedrig": "Niedrig"}[prio_key]
    _workflow.pop(chat_id, None)   # Workflow abschließen
    response = run_claude(f"Name: {name}. Prio: {prio}.", ...)
    send_message(TOKEN, chat_id, response, reply_markup=REPLY_KEYBOARD)

elif data == "wf:abort":
    _workflow.pop(chat_id, None)
    send_message(TOKEN, chat_id, "❌ Abgebrochen.", reply_markup=REPLY_KEYBOARD)

Routing-Reihenfolge im Haupt-Loop

Workflow-Check vor Button-Map — sonst würde "📋 Task" als Task-Name den Workflow überschreiben.

if handle_workflow_step(text, chat_id, today):
    continue       # 1. Workflow konsumiert

t = text.strip()
if t.lower().startswith("erinnere"): ...  # 2. Infrastruktur
elif t in BUTTON_MAP:                        # 3. Button → Workflow starten
    start_workflow(BUTTON_MAP[t], chat_id)

# 4. Alles andere: ignorieren

Grenzen

  • Restart-Loss: Bot-Neustart leert Dict. Für Single-User-Bots negligible.
  • Kein Timeout: Halboffener Workflow bleibt. Fix: Timestamp in data + Cleanup-Thread.
  • 1 Workflow pro Chat: Neuer start_workflow-Aufruf überschreibt bestehenden.

🎯 Quiz

1. handle_workflow_step gibt False zurück — was bedeutet das für den Haupt-Loop?

Antwort
Kein aktiver Workflow → Nachricht normal routen (Infrastruktur-Kommandos, Button-Map, etc.).

2. Warum konsumiert handle_workflow_step auch bei Step "task:priority" (Callback-Step) Text-Input mit True?

Antwort
Um zu verhindern dass unerwartetem Freitext die Routing-Pipeline passiert und z.B. einen neuen Workflow startet. Text wird ignoriert aber als konsumiert markiert.

3. Was passiert wenn der User "📋 Task" tippt während ein "task:name"-Step aktiv ist?

Antwort
"📋 Task" wird als Task-Name gespeichert (wf["data"]["name"] = "📋 Task"), Step wechselt zu "task:priority". Kein neuer Workflow wird gestartet — der Workflow-Check kommt vor dem Button-Map-Check.

📖 Summary

  • _workflow: dict = {} — global, chat_id → {step, data}
  • Step-Strings: "kind:field" — Text-Steps vs Callback-Steps explizit benannt
  • handle_workflow_step: True = konsumiert, False = weiter routen
  • Workflow-Check vor Button-Map — verhindert Überschreiben laufender Workflows
  • Abort: wf:abort Callback + _workflow.pop(chat_id)
  • Grenzen: Restart-Loss, kein Timeout, 1 Workflow/Chat
