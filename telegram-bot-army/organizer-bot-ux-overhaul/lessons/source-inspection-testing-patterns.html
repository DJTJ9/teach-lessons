Source-Inspection Testing Patterns

Organizer Bot UX Overhaul · Test-Strategie ohne Runtime

Was ist Source-Inspection Testing?

Quelldatei als String lesen, Muster prüfen — kein Runtime nötig. Ideal für Cleanup-Checks, Architektur-Invarianten und Feature-Presence-Checks.

def _src():
    with open("bots/organizer.py", encoding="utf-8") as f:
        return f.read()

class TestCleanup(unittest.TestCase):
    def test_no_pending_task_input(self):
        self.assertNotIn("pending_task_input", _src())

    def test_routing_order(self):
        src = _src()
        wf_idx  = src.index("handle_workflow_step(text, chat_id, today)")
        btn_idx = src.index("if t in BUTTON_MAP")
        self.assertLess(wf_idx, btn_idx)

Vier Test-Typen

TypAssertionWofür
PresenceassertIn("BUTTON_MAP", src)Feature implementiert
AbsenceassertNotIn("dead_code", src)Toter Code entfernt
OrderassertLess(src.index(A), src.index(B))Routing-Reihenfolge korrekt
Scopebody = src[start:end]; assertIn(X, body)Pattern nur in bestimmter Funktion

⚠️ Pitfall 1: Substring-Problem

assertNotIn("TASK_SYSTEM_PROMPT", src) schlägt fehl wenn PROJEKT_TASK_SYSTEM_PROMPT noch im Code ist — "TASK_SYSTEM_PROMPT" ist ein Substring davon.

# ❌ Trifft auch PROJEKT_TASK_SYSTEM_PROMPT
self.assertNotIn("TASK_SYSTEM_PROMPT", src)

# ✓ Mit Zuweisungs-Operator — nur Deklaration trifft
self.assertNotIn("TASK_SYSTEM_PROMPT =", src)

# ✓ Oder: alle Konstanten mit dem Suffix entfernen
# (PROJEKT_TASK_SYSTEM_PROMPT war toter Code → mitlöschen)
Regel: Bei assertNotIn(name, src) immer prüfen ob name Substring anderer Bezeichner ist. Mit " =" oder Leerzeichen anhängen macht den Match präziser.

⚠️ Pitfall 2: Index trifft Deklaration, nicht Handler

src.index("_workflow") findet die globale Deklaration _workflow: dict = {} (Zeile ~489), nicht den Check im Message-Handler (Zeile ~1604).

# ❌ Findet Deklaration statt Handler
wf_idx = src.index("_workflow")

# ✓ Einmalig-vorkommenden vollständigen Ausdruck nutzen
wf_idx  = src.index("handle_workflow_step(text, chat_id, today)")
btn_idx = src.index("if t in BUTTON_MAP")
self.assertLess(wf_idx, btn_idx)

⚠️ Pitfall 3: Scope ohne Funktionsgrenzen

# ✓ Scope auf eine Funktion beschränken
src = _src()
start = src.index("def _send_moin_messages(")
end   = src.index("\ndef ", start + 1)
body  = src[start:end]

self.assertIn('f"done:{pid}"', body)
self.assertNotIn("_task_buttons(pid)", body)  # alter Aufruf entfernt?

Wann benutzen?

✓ Gut für
  • Cleanup-Checks (kein toter Code)
  • Architektur-Invarianten
  • Feature-Presence nach Refactoring
  • Schnell, zero Runtime
✗ Nicht für
  • Logik-Korrektheit
  • Dynamisches Verhalten
  • String in Strings/Kommentaren
  • API-Integrations-Checks

🎯 Quiz

1. assertNotIn("CHAT", src) schlägt fehl obwohl du CHAT_SYSTEM_PROMPT gelöscht hast. Was ist der Grund?

Antwort
"CHAT" ist Substring von CHAT_ID, chat_id, etc. die noch im Code sind. assertNotIn prüft wörtliche Substring-Präsenz — auch in Variablennamen, String-Literals, Kommentaren.

2. Scope-Test: Du willst sicherstellen dass _send_moin_messages keine _task_buttons() mehr aufruft. Wie?

Antwort
src = _src()
s = src.index("def _send_moin_messages(")
e = src.index("\ndef ", s + 1)
self.assertNotIn("_task_buttons(", src[s:e])

3. Order-Test: src.index("my_func") wirft ValueError. Was bedeutet das?

Antwort
str.index() wirft wenn nicht gefunden. Für Order-Tests bedeutet das: das Feature fehlt vollständig — der Test zeigt damit korrekt einen Fehler. Für Absence-Tests stattdessen assertNotIn verwenden (kein Error bei Nichtvorhandensein).

📖 Summary

  • Source-Inspection: Quelldatei als String, 4 Typen: Presence / Absence / Order / Scope
  • Substring-Pitfall: assertNotIn("FOO", src) trifft BAR_FOO — " =" anhängen oder verwandte Bezeichner mitlöschen
  • Index-Pitfall: Kurze Namen finden Deklaration — vollständige Ausdrücke nutzen
  • Scope-Pitfall: Funktionsgrenzen manuell extrahieren: src[start:end]
  • Nicht für Logik-Korrektheit oder dynamisches Verhalten
