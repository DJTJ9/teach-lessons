Lektion 2: Routing-Map Fallbacks designen

Wann dict.get(key, default) gefährlich ist — und wie man es richtig macht

Der Bug: produktionskritischer Fallback

routing.py — der stille Fehler: 
SESSION_BOT_MAP = {
    "brainstorming": "brain",
    "teach":         "teach",
    "dev":           "brain",
}

def _get_target_bot_name(active_session):
    return SESSION_BOT_MAP.get(active_session, "permissions")  # BUG
Wenn active_session = None → Fallback "permissions" → Nachricht geht an Permissions Bot.
Kein Fehler, kein Log. Einfach falscher Empfänger.

Warum das passiert ist

Permissions Bot war der erste Bot — "permissions" als Default war intuitiv. Funktionierte solange active_session immer gesetzt war. Sobald finish.md set_session.py bedingt nicht aufgerufen hat → active_session = null → Fallback griff still.

Die drei Fallback-Strategien

StrategieCodeWann sinnvoll
None (no-op).get(key) → None-CheckWenn kein Match = kein Send. Sicher.
Exceptionmap[key] → KeyErrorWenn unbekannte Session ein Programmierfehler ist.
Produktiver Default.get(key, "permissions")Nur wenn Default wirklich immer richtig ist.

Sicherer Ansatz: None als Fallback

def _get_target_bot_name(active_session):
    return SESSION_BOT_MAP.get(active_session)  # None wenn kein Match

def get_notify_token(settings, env=None):
    bot_name = _get_target_bot_name(settings.get("active_session"))
    if bot_name is None:
        return None   # Caller: if not token: sys.exit(0)
    return env.get(f"TOKEN_{bot_name.upper()}")
Warum routing.py hier nicht geändert wurde: Der gewählte Fix war --bot teach caller-seitig. Routing-Fallback bleibt "permissions" — für on_stop.py ist das tatsächlich richtig. Beide Ansätze valide — caller-seitig ist minimaler.

Erkennen gefährlicher Fallbacks im Code Review

Frage: "Was passiert wenn der Key nicht in der Map ist?"

# Gefährlich:
result = routing_map.get(session, "send_somewhere")

# Sicher — no-op:
result = routing_map.get(session)
if result is None:
    return

# Sicher — expliziter Fehler:
result = routing_map[session]  # KeyError wenn unbekannt

Quiz

Warum ist .get(None, "permissions") ein Problem aber .get(None) nicht?

Antwort

Mit "permissions" als Default wird eine Nachricht an den Permissions Bot gesendet — stilles Fehlverhalten. Mit None gibt get_notify_token None zurück → sys.exit(0) → kein Send, kein falscher Empfänger.

Wann wäre SESSION_BOT_MAP[session] besser als .get()?

Antwort

Wenn unbekannte Session ein Programmierfehler ist, der sofort sichtbar sein soll. KeyError ist lauter als stilles Fehlverhalten — gut für interne APIs, zu riskant für robuste Notify-Scripts.

Zusammenfassung

  • Routing-Map Fallback mit produktiver Aktion ist stille Fehlerquelle
  • Bevorzuge .get(key) → None-Check → no-op für Notify-Scripts
  • Oder map[key] → KeyError für interne APIs
  • Code Review Frage: "Was passiert wenn Key fehlt?"
  • Fix kann caller-seitig oder routing-seitig sein — minimal gewinnt
