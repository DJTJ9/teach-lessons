Lektion 3: Setup-Schritte mit mehreren Konsumenten

Warum bedingte Setup-Schritte gefährlich werden wenn mehrere Features darauf angewiesen sind

Das Problem

finish.md — bedingte Ausführung: 
If `notifications_enabled: true`, run:
    python3 "$WORK_DIR/scripts/set_session.py" dev <SLUG>
Gedacht für: Telegram-Routing.
Unbeabsichtigt auch gebraucht von: Teach Skill für per-Projekt Workspace.
Bei notifications_enabled: false → active_dev_slug = null → Teach Skill nutzt flachen Workspace.
set_session.py dev <slug> │ ├── Konsument A: telegram_ask.py │ liest active_session → routet zu Brain Bot │ → braucht: notifications_enabled: true │ └── Konsument B: teach SKILL.md liest active_dev_slug → per-Projekt Workspace → braucht: IMMER (unabhängig von Notifications)

Warum das schwer zu entdecken ist

Konsument B wurde nach der bedingten Logik hinzugefügt. Die Bedingung war ursprünglich korrekt. Bug tritt nur bei notifications_enabled: false auf — ein Zustand der selten aktiv getestet wird.

Der Fix: immer ausführen

# Vorher:
If `notifications_enabled: true`, run:
    python3 "$WORK_DIR/scripts/set_session.py" dev <SLUG>

# Nachher:
Always run (required for teach workspace routing,
regardless of notifications_enabled):
    python3 "$WORK_DIR/scripts/set_session.py" dev <SLUG>

Konsumenten-Analyse vor Feature-Flags

# set_session.py schreibt:
# - active_session   → gelesen von: telegram_ask.py, telegram_notify.py
# - active_dev_slug  → gelesen von: teach SKILL.md, telegram_ask.py

# Alle Konsumenten prüfen:
# Hat jeder Konsument dieselbe Bedingung wie die Flag?
# → Ja: Flag ist sicher
# → Nein: Flag entfernen, immer ausführen
Faustregel: Schreibt ein Setup-Schritt in eine geteilte State-Datei (settings.json)? Dann ist er wahrscheinlich Multi-Konsument. Geteilter State = Analyse aller Konsumenten vor Feature-Flag.

Quiz

Ein neuer Setup-Schritt schreibt current_task_id in settings.json. Er soll nur bei task_mode: true aktiv sein. Welche Frage musst du stellen?

Antwort

Wer liest current_task_id? Wenn alle Konsumenten ebenfalls nur bei task_mode: true aktiv sind → Flag sicher. Wenn ein Konsument current_task_id unabhängig liest → Flag entfernen.

Warum hat implementation_mode kein Konditionsproblem?

Antwort

Weil es immer via Edit-Tool gesetzt wird (true/false), nicht nur bei bestimmter Flag. Alle Konsumenten lesen den Wert direkt.

Zusammenfassung

  • Setup-Schritte in geteilten State-Files haben oft mehrere Konsumenten
  • Vor Feature-Flag: alle Konsumenten analysieren — haben alle dieselbe Bedingung?
  • Wenn ein Konsument die Flag nicht hat → immer ausführen
  • Bug tritt erst auf wenn Flag false ist — selten getestet
  • Geteilter State = Warnsignal für Multi-Konsument-Setup
