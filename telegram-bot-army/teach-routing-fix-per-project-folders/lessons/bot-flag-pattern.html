Lektion 1: Das --bot Flag Pattern

Manuelles sys.argv-Parsing für CLI-Overrides ohne argparse

Das Problem

Ein Script wird von vielen Callers aufgerufen. Meistens soll es Routing-Logik nutzen — aber manchmal braucht ein spezifischer Caller einen direkten Override ohne State-Dependency.

Beispiel: telegram_notify.py routet via active_session → SESSION_BOT_MAP. Aber der Teach Skill will immer den Teach Bot erreichen, egal welche Session aktiv ist.

Die Lösung: Prefix-Check auf sys.argv

# Vorher: nur positionales Argument
if len(sys.argv) < 2:
    sys.exit(0)
token = get_notify_token(settings)   # abhängig von active_session
message = sys.argv[1]
# Nachher: optionaler --bot Override
_args = sys.argv[1:]
bot_override = None
if len(_args) >= 2 and _args[0] == "--bot":
    bot_override = _args[1]
    _args = _args[2:]

if not _args:
    sys.exit(0)

token = (
    os.environ.get(f"TOKEN_{bot_override.upper()}")
    if bot_override
    else get_notify_token(settings)
)
message = _args[0]

Warum manuelles Parsing statt argparse?

argparse ruft sys.exit(2) bei unbekannten Flags auf — kaputte Callers brechen das Script. Bei einem Notify-Script das überall aufgerufen wird ist das zu riskant.

Manuelles Parsing: wir greifen auf sys.argv[1:] zu, konvertieren zu Liste, checken nur ob erstes Element "--bot" ist. Kein Framework nötig.

Aufruf-Protokoll

# Bestehende Callers — unverändert:
python telegram_notify.py "Nachricht"

# Neuer Override-Caller:
python telegram_notify.py --bot teach "Nachricht"
python telegram_notify.py --bot brain "Nachricht"
Reihenfolge ist fest: --bot <name> muss vor der Nachricht stehen. Bei komplexeren CLIs mit optionalen Flags in beliebiger Reihenfolge → argparse verwenden.

Token-Auflösung via f-String Convention

token = os.environ.get(f"TOKEN_{bot_override.upper()}")
# --bot teach       →  TOKEN_TEACH
# --bot brain       →  TOKEN_BRAIN
# --bot permissions →  TOKEN_PERMISSIONS

Kein hard-coded Dict nötig. Convention TOKEN_{BOTNAME.upper()} macht den Override-Code wartungsfrei — neue Bots funktionieren automatisch.

Quiz

sys.argv = ["notify.py", "--bot", "teach", "Hallo"]. Was ist bot_override und _args danach?

Antwort

bot_override = "teach", _args = ["Hallo"]

sys.argv = ["notify.py", "Hallo"] (kein --bot). Was passiert?

Antwort

bot_override = None, _args = ["Hallo"] → normales Routing via get_notify_token(settings).

Zusammenfassung

  • sys.argv[1:] in Liste kopieren → Elemente konsumieren statt indexieren
  • Prefix-Check: if len(_args) >= 2 and _args[0] == "--bot"
  • Token-Auflösung: os.environ.get(f"TOKEN_{bot_override.upper()}")
  • Bestehende Callers ohne --bot bleiben unverändert
  • Manuelles Parsing für simple 1-Flag-Overrides, argparse für komplexere CLIs
