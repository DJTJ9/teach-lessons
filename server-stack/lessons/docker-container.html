Docker — Container-Grundlagen

~16 Min · Baustein 4/11

Das Problem

NocoDB braucht eine bestimmte Node.js-Version, eine Postgres/CouchDB-Datenbank, bestimmte System-Bibliotheken. Würdest du das alles direkt auf dem Server installieren, kollidiert es potenziell mit anderen Python-/Node-Versionen, die deine Bots brauchen ("Dependency Hell"). Und: willst du NocoDB später auf einem anderen Server laufen lassen, müsstest du die gesamte Installation manuell reproduzieren.

Die Lösung: Isolierte, portable Umgebungen

Ein Container verpackt eine Anwendung mit allem, was sie zum Laufen braucht (Runtime, Bibliotheken, Konfiguration) in ein Image. Docker führt dieses Image isoliert aus — der Container sieht ein eigenes Dateisystem, eigene Prozesse, kann aber den Host-Kernel mitnutzen (kein voller Hypervisor wie bei einer VM nötig, daher viel leichtgewichtiger).

Auf deinem Server laufen aktuell (docker ps):

nocodb-data-couchdb-1   couchdb:3
nocodb-data-nocodb-1    nocodb/nocodb:latest

Der Namens-Präfix nocodb-data- zeigt: das ist ein Docker-Compose-Projekt — mehrere Container (NocoDB-App + seine CouchDB-Datenbank), die zusammen als eine Einheit gestartet/gestoppt werden, definiert in einer docker-compose.yml.

Container vs. virtuelle Maschine: Eine VM emuliert komplette Hardware inkl. eigenem Betriebssystem-Kernel — schwer (GBs), langsamer Start (Minuten). Ein Container teilt sich den Host-Kernel und isoliert nur Prozess/Dateisystem-Sicht — leicht (MBs), Start in Sekunden. Für "NocoDB + seine DB isoliert von den Python-Bots betreiben" ist ein Container die richtige Größenordnung.

Warum genau hier Docker eingesetzt wird

Deine Telegram-Bots (bots/brain.py etc.) laufen nativ über systemd — kein Docker. NocoDB dagegen läuft in Docker. Der Unterschied: die Bots sind dein eigener, einfacher Python-Code mit wenigen Abhängigkeiten (per requirements.txt im venv lösbar). NocoDB ist eine komplexe Drittanbieter-Anwendung mit eigener Datenbank-Anforderung — dafür ist ein fertiges, vom Hersteller gepflegtes Docker-Image die schnellste, sauberste Betriebsart: docker compose up statt manueller Installation von Node, korrekten npm-Paketen, DB-Setup.

Wie das in deine Kette passt

NocoDB (Lesson 5) läuft in einem Container, ist aber über NOCODB_API_URL für deine Bot-Skripte (nocodb_sync.py) erreichbar wie jeder andere lokale Service — Docker mappt den Container-internen Port auf einen Host-Port, den Caddy oder deine Python-Skripte ganz normal ansprechen. Die Isolation ist für den Rest deiner Architektur unsichtbar.

Quiz: Warum reicht bei den Python-Bots ein einfaches venv statt Docker?

Weil die Bots wenig Systemabhängigkeiten haben (reines Python + pip-Pakete) und alle auf demselben Server im selben Kontext laufen sollen — kein Bedarf an Isolation zwischen ihnen. Docker lohnt sich, wenn eine Anwendung komplexe/konfligierende Systemabhängigkeiten hat (wie NocoDB mit Node + eigener DB) oder portabel auf andere Hosts soll.

Quiz: Was bedeutet der Name nocodb-data-couchdb-1 konkret?

nocodb-data ist der Compose-Projektname (meist der Ordnername, in dem die docker-compose.yml liegt), couchdb ist der Service-Name innerhalb der Compose-Datei, 1 ist die Instanznummer (falls der Service mehrfach skaliert würde). Docker Compose generiert diesen Namen automatisch aus <projekt>-<service>-<nummer>.

Zusammenfassung

Docker isoliert Anwendungen mit komplexen/eigenen Abhängigkeiten (wie NocoDB) in portable, reproduzierbare Container — ohne dass sie mit dem Rest des Servers (deinen nativ laufenden Python-Bots) kollidieren. Docker Compose bündelt mehrere zusammengehörige Container (App + DB) zu einer startbaren Einheit. Das ist die Basis, auf der NocoDB (nächste Lesson) betrieben wird.
