Cloudflare + Caddy — die TLS-Kette

~20 Min · Baustein 3/11

Das Problem

code.thinkshark.de muss öffentlich per HTTPS erreichbar sein (Telegram schickt Webhooks nur an HTTPS-URLs). Dafür brauchst du: (1) ein gültiges TLS-Zertifikat, (2) Schutz vor DDoS/Bot-Traffic, (3) einen Weg, eine Domain auf mehrere lokale Services (Port 8001-8004, 8080...) zu verteilen. Das alles selbst zu bauen (Let's-Encrypt-Renewal, DDoS-Mitigation, Reverse-Proxy-Routing) ist viel Betriebsaufwand für ein Ein-Personen-Setup.

Die Lösung: Zwei Schichten, zwei Jobs

Deine Kette hat zwei TLS-Terminierungspunkte hintereinander:

Client (Browser/Telegram)
   │  HTTPS
   ▼
Cloudflare Edge (DNS + Proxy + DDoS-Schutz)
   │  HTTPS, Origin-Zertifikat
   ▼
Caddy auf deinem Server (Reverse-Proxy)
   │  HTTP (localhost)
   ▼
Bot/App auf Port 8001-8004 / 8080 / ...

Schicht 1 — Cloudflare

Cloudflare sitzt als DNS-Provider und als Proxy vor deinem Server. Anfragen an code.thinkshark.de gehen zuerst zu Cloudflares Edge-Netzwerk (Rechenzentren weltweit), nicht direkt zu deinem Server. Das bringt: DDoS-Absorption, Caching, verstecke deine echte Server-IP, und automatisches TLS zum Client hin.

Schicht 2 — Caddy mit Origin-Zertifikat

Zwischen Cloudflare und deinem Server läuft die Verbindung ebenfalls verschlüsselt ("Full (strict)"-Modus) — dafür braucht dein Server ein eigenes Zertifikat, das Cloudflare als vertrauenswürdig akzeptiert: ein Cloudflare-Origin-Zertifikat (cf-origin.pem / cf-origin-key.pem), von Cloudflare selbst ausgestellt, nur für die Verbindung Cloudflare↔Origin gültig.

Dein echtes /etc/caddy/Caddyfile:

{
    auto_https off
}

code.thinkshark.de {
    tls /etc/caddy/cf-origin.pem /etc/caddy/cf-origin-key.pem

    @webhook_brain path /webhook/brain/*
    handle @webhook_brain {
        reverse_proxy localhost:8001
    }
    @webhook_organizer path /webhook/organizer/*
    handle @webhook_organizer {
        reverse_proxy localhost:8002
    }
    @webhook_teach path /webhook/teach/*
    handle @webhook_teach {
        reverse_proxy localhost:8003
    }
    @webhook_permissions path /webhook/permissions/*
    handle @webhook_permissions {
        reverse_proxy localhost:8004
    }
    handle {
        reverse_proxy localhost:8080
    }
}

app.thinkshark.de {
    tls /etc/caddy/cf-origin.pem /etc/caddy/cf-origin-key.pem
    root * /var/www/shopping-navigator
    @wasm path *.wasm
    header @wasm Content-Type application/wasm
    file_server
}

auto_https off heißt: Caddy soll nicht selbst versuchen, Let's-Encrypt-Zertifikate zu holen (das würde mit dem Cloudflare-Origin-Zertifikat kollidieren) — stattdessen nutzt jeder tls-Block explizit die Cloudflare-Origin-Datei.

Warum Caddy und nicht nginx: nginx braucht für TLS-Config und Reverse-Proxy-Routing deutlich mehr Boilerplate (separate location-Blöcke, manuelles SSL-Setup, kein eingebautes automatisches HTTPS). Caddys Config-Sprache ist knapp lesbar und pfad-basiertes Routing (@webhook_brain path ...) ist eine Zeile — genau das, was du hier für 4 Bots + 2 Domains brauchst, ohne einen 200-Zeilen-nginx-Config zu pflegen.

Wie eine Telegram-Nachricht konkret durchläuft

  1. Nutzer schreibt dem Teach-Bot in Telegram
  2. Telegram schickt POST an https://code.thinkshark.de/webhook/teach/<token>
  3. Cloudflare Edge nimmt die Anfrage an, terminiert TLS zum Client, prüft auf Bot-Traffic/DDoS
  4. Cloudflare leitet über eine eigene TLS-Verbindung an deinen Server weiter (Origin-Zertifikat)
  5. Caddy matcht path /webhook/teach/* und reverse-proxied unverschlüsselt an localhost:8003
  6. Der Teach-Bot-Prozess (systemd-Service, siehe Lesson 7) verarbeitet die Nachricht
Quiz: Warum reicht localhost:8003 im letzten Schritt ohne eigenes TLS?

Weil die Verbindung Caddy → Bot-Prozess nie das physische Gerät verlässt — beides läuft auf demselben Server über die Loopback-Schnittstelle. Es gibt keinen Netzwerkpfad, den ein Angreifer abhören könnte, also ist die Verschlüsselung dort unnötiger Overhead. Die beiden TLS-Terminierungen (Client→Cloudflare, Cloudflare→Caddy) decken den gesamten Weg ab, der tatsächlich über ein Netzwerk läuft.

Quiz: Was würde passieren, wenn auto_https off fehlen würde?

Caddy würde versuchen, für code.thinkshark.de automatisch ein Let's-Encrypt-Zertifikat per ACME-Challenge zu holen — das würde mit dem expliziten tls-Directive kollidieren bzw. wäre unnötig, da Cloudflare den Domain-Traffic ohnehin zuerst empfängt und die Validierung dadurch erschwert/blockiert würde.

Zusammenfassung

Cloudflare übernimmt die öffentliche Angriffsfläche (DNS, DDoS, Edge-TLS), Caddy übernimmt lokal das Routing verschiedener Pfade/Subdomains auf die richtigen Ports — verbunden durch ein Origin-Zertifikat, das nur diese eine Strecke absichert. Diese zwei Schichten sind der Grund, warum 4 Bots + 1 App über eine einzige Domain-Struktur sauber getrennt erreichbar sind, ohne dass du selbst Zertifikate verwalten musst.
