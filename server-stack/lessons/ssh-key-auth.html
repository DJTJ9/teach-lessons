SSH Key-Auth — Server-Zugriff absichern

~18 Min · Baustein 1/11 · Grundlage für alle weiteren Lessons dieser Reihe

Das Problem

Dein Server (erreichbar über Tailscale und öffentlich über code.thinkshark.de) hat SSH auf Port 22 offen — jeder im Internet kann sich verbinden versuchen. Mit Passwort-Login reicht ein einziges erratenes/geleaktes Passwort für vollen Root-Zugriff. Bots durchsuchen das Internet ständig nach Servern mit Passwort-SSH und probieren automatisiert Passwörter (Brute-Force).

Die Lösung: Asymmetrische Kryptographie

Statt eines Geheimnisses, das du eintippst (Passwort), nutzt SSH ein Schlüsselpaar:

  • Private Key (id_ed25519) — bleibt nur auf deinem Client-Gerät, verlässt es nie
  • Public Key (id_ed25519.pub) — wird auf dem Server in ~/.ssh/authorized_keys eingetragen

Beim Verbindungsaufbau schickt der Server eine zufällige Challenge. Dein Client signiert sie mit dem Private Key. Der Server prüft die Signatur mit dem hinterlegten Public Key. Kann er sie verifizieren, bist du authentifiziert — ohne dass ein Geheimnis je über das Netzwerk übertragen wird.

Warum das sicherer ist als Passwort: Ein Passwort kann erraten, abgephisht oder über einen unsicheren Kanal mitgelesen werden. Ein Ed25519-Private-Key hat 2^256 mögliche Werte — Brute-Force ist praktisch unmöglich, und der Key wird nie übertragen, kann also nicht "mitgeschnitten" werden.

Konkret auf deinem Server

~/.ssh/
├── authorized_keys   # Public Keys, die sich einloggen dürfen
├── config            # Client-seitige Verbindungs-Shortcuts
├── id_ed25519         # dein Private Key (Berechtigung 600 — nur du liest ihn)
├── id_ed25519.pub      # dein Public Key
└── known_hosts        # Fingerprints der Server, denen du schon vertraut hast

Die Dateirechte sind entscheidend: id_ed25519 hat -rw------- (600) — nur der Owner darf lesen/schreiben. SSH verweigert den Login komplett, wenn die Rechte zu offen sind (z.B. 644), weil ein zu offener Private Key als kompromittiert gilt.

/etc/ssh/sshd_config — die Server-Seite

# relevante Zeilen (auskommentiert = Distribution-Default aktiv)
#Port 22
#PermitRootLogin prohibit-password
#PasswordAuthentication yes

PermitRootLogin prohibit-password heißt: Root-Login ist erlaubt, aber nur mit Key — ein Passwort-Login als root wird abgelehnt, selbst wenn eins gesetzt wäre. Das ist der Kompromiss zwischen "Root braucht man für Admin-Arbeit" und "Passwort-Root-Login ist das erste, was jeder Angreifer probiert".

Wie das in deine Kette passt

SSH ist die Verwaltungs-Ebene — du selbst (und andere Geräte über Tailscale, siehe nächste Lesson) kommst darüber auf den Server, um Code zu deployen, Logs zu lesen, Services neu zu starten. Es ist getrennt von der Nutzer-Ebene (Cloudflare → Caddy → Bots), über die Telegram-Nutzer oder App-Nutzer den Server erreichen. Zwei komplett unterschiedliche Zugriffspfade mit unterschiedlichen Sicherheitsanforderungen.

Quiz: Warum verlässt der Private Key nie den Client?

Weil die Authentifizierung über ein Challenge-Response-Verfahren läuft: der Server schickt eine zufällige Zahl, der Client signiert sie lokal mit dem Private Key, nur die Signatur (nicht der Key selbst) geht über das Netz. Der Server verifiziert die Signatur mit dem öffentlichen Gegenstück. Würde der Private Key übertragen, könnte ihn jeder auf dem Weg mitlesen und danach selbst benutzen.

Quiz: Was passiert, wenn id_ed25519 Datei-Rechte 644 statt 600 hat?

SSH verweigert die Nutzung dieses Keys mit einem "UNPROTECTED PRIVATE KEY FILE" Fehler. Grund: 644 bedeutet "jeder auf dem System kann lesen" — SSH geht dann davon aus, der Key könnte kompromittiert sein, und bricht lieber ab statt ein unsicheres Setup stillschweigend zu erlauben.

Zusammenfassung

SSH-Key-Auth ersetzt "ein Geheimnis eintippen" durch "einen Beweis erbringen, ohne das Geheimnis preiszugeben" (Public-Key-Kryptographie). authorized_keys auf dem Server + id_ed25519 auf deinem Client + PermitRootLogin prohibit-password in sshd_config ergeben zusammen: Admin-Zugriff, der gegen Brute-Force und Passwort-Leaks praktisch immun ist. Das ist die Basis, auf der Tailscale (nächste Lesson) noch eine Netzwerk-Ebene Sicherheit obendrauf legt.
