Dev-Skill + Memory-System — Projekt-Gedächtnis über Sessions hinweg

~18 Min · Baustein 10/11

Das Problem

Ein KI-Coding-Assistent wie Claude Code hat standardmäßig keinen Zustand zwischen Sitzungen — jede neue Konversation startet ohne Wissen darüber, in welcher Phase ein Projekt gerade ist, welche Entscheidungen bereits gefallen sind, oder was letztes Mal gelernt wurde. Bei einem Projekt wie shopping-navigator, das sich über Wochen/Monate entwickelt, würdest du sonst jede Session erneut den ganzen Kontext manuell erklären müssen.

Die Lösung: Strukturierte Markdown-Dateien als externes Gedächtnis

Statt sich auf das Sprachmodell-Gedächtnis zu verlassen, legt der Dev-Skill für jedes Projekt einen festen Satz Dateien im Hub ($HUB_DIR/topics/<slug>/) an, die bei jeder Session automatisch gelesen werden:

DateiZweck
STATUS.mdAktuelle Phase + Roadmap-Zustand — wo stehen wir gerade
VISION.mdLangfristiges Ziel, wofür das Projekt existiert
DECISIONS.mdGetroffene Architektur-Entscheidungen + Begründung
TIMELINE.mdChronologie abgeschlossener Meilensteine
LEARNINGS.mdWas aus Fehlern/Erfahrungen gelernt wurde

Echtes STATUS.md für den shopping-navigator:

Active: Testmarkt (Seed-Daten)
Phase: discuss

## Roadmap
- [done]  Phase 1 Shopping List + Barcode + GPS Position
- [done]  Store Capture Walk-Through MVP
- [done]  Supabase Foundation + Auth
- [done]  Server-Infrastruktur (Caddy + Cloudflare, app.thinkshark.de)
- [done]  Phase 2 A*-Routenplanung + Store Map
- [idea]   Testmarkt (Seed-Daten)
- [idea]   Email-Auth
- [idea]   Mehrere Filialen pro Nutzer

Phase: discuss ist der Schlüssel für die Routing-Logik des Dev-Skills: Ruft man /dev (ohne Argumente) auf, liest der Skill genau diese Zeile und entscheidet automatisch, in welche Phase (discuss/plan/implement/finish) er springt — kein manuelles "wo waren wir stehengeblieben" mehr nötig.

Warum Markdown-Dateien statt einer Datenbank für "Projekt-Gedächtnis": Diese Dateien werden von einem Sprachmodell gelesen und geschrieben, nicht von einer klassischen Anwendung. Klartext-Markdown ist für ein LLM direkt konsumierbar (keine Parsing-Logik nötig), git-versionierbar (Verlauf der Projekt-Entscheidungen ist selbst Teil der Historie) und menschen-lesbar, falls du selbst reinschaust. Für strukturierte Abfragen (z.B. "zeig mir alle offenen Tasks über alle Projekte") wäre eine DB besser — aber genau das ist hier nicht das Problem, das gelöst werden soll.

Wie das in deine Kette passt

Der Hub (projects-hub) ist die zentrale Registry (projects-registry.json) aller Projekte (telegram-bot-army, shopping-navigator, dev-skill selbst, ...) — jedes referenziert per slug seinen eigenen topics/<slug>/-Ordner mit genau diesem Datei-Set. Das /teach-Skill, das gerade diese Lesson erzeugt hat, folgt demselben Muster (MISSION.md, NOTES.md, learning-records/) — beide Systeme lösen dasselbe Grundproblem ("Zustand über Sessions hinweg persistieren") mit derselben Technik (strukturierte Markdown-Dateien an einem festen Ort).

Quiz: Warum liegen STATUS.md/VISION.md im Hub-Repo statt im Projekt-Repo selbst (z.B. im shopping-navigator-Repo)?

Weil der Hub projektübergreifend funktionieren soll — ein Sprachmodell, das gerade an irgendeinem Projekt arbeitet, muss dieselbe zentrale Stelle kennen, um herauszufinden "was ist der Zustand von Projekt X". Läge der Zustand verteilt in jedem einzelnen Projekt-Repo, bräuchte man für einen Überblick über alle Projekte n verschiedene Repos statt eines zentralen Registry-Punkts.

Quiz: Was passiert, wenn Phase: discuss in STATUS.md nicht aktuell gehalten wird?

Der Dev-Skill würde bei /dev in die falsche Phase routen — z.B. erneut Klärungsfragen stellen, obwohl längst ein Plan geschrieben wurde. Das Feld ist die einzige Quelle der Wahrheit für "wo stehen wir" — hält man es nicht aktuell, verliert das ganze System seinen Zweck (Kontext ohne manuelles Nacherzählen wiederherzustellen).

Zusammenfassung

Das Dev-Skill+Memory-System löst "kein Zustand zwischen KI-Sessions" durch einen festen Satz Markdown-Dateien pro Projekt (STATUS/VISION/DECISIONS/TIMELINE/LEARNINGS), zentral im Hub abgelegt und bei jeder Session automatisch gelesen. Phase: in STATUS.md steuert dabei, in welche Arbeitsphase der Skill beim nächsten Aufruf automatisch springt.
