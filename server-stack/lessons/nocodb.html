NocoDB — No-Code-Datenbank für die Bot-Army

~17 Min · Baustein 5/11 · baut auf Docker (Lesson 4) auf

Das Problem

Deine Telegram-Bots verwalten Tasks, Habits, Sport-Tracking, Backlog-Ideen — das sind klassische, tabellarische Daten mit Feldern, Filtern, Relationen. Eine eigene Datenbank + eigenes Admin-Interface dafür zu bauen (Tabellen-CRUD-UI, Auth, API) wäre viel Aufwand für etwas, das im Kern "Excel mit API" ist.

Die Lösung: Postgres/CouchDB + generierte REST-API + UI, fertig

NocoDB nimmt eine Datenbank (bei dir: die CouchDB-Instanz aus dem gleichen Docker-Compose-Projekt) und legt automatisch eine Tabellen-UI im Airtable-Stil sowie eine REST-API pro Tabelle darüber. Du definierst Tabellen/Felder per Klick-UI, bekommst sofort:

  • Ein bearbeitbares Grid/Kanban/Kalender-Interface für dich selbst
  • Eine REST-API, die deine Python-Skripte ansprechen können
  • Views, Filter, Relationen zwischen Tabellen — ohne SQL zu schreiben

Deine .env-Konfiguration zeigt die Struktur:

NOCODB_API_URL=...
NOCODB_API_TOKEN=...
NOCODB_BASE_ID=...
NOCODB_TASKS_TABLE_ID=...
NOCODB_HABITS_TABLE_ID=...
NOCODB_SPORT_TABLE_ID=...
NOCODB_BACKLOG_TABLE_ID=...
NOCODB_IDEENSAMMLUNG_TABLE_ID=...
NOCODB_ARCHIV_TABLE_ID=...
NOCODB_FOCUS_TABLE_ID=...

Ein BASE_ID (ein NocoDB-Projekt) enthält mehrere Tabellen, jede mit eigener ID — deine Bot-Skripte (nocodb_sync.py, nocodb_create_table.py, nocodb_promote_backlog.py) sprechen über den API-Token gezielt einzelne Tabellen an, z.B. um einen Backlog-Eintrag in die Tasks-Tabelle zu "promoten".

Warum NocoDB statt Airtable direkt: Airtable ist SaaS mit Kosten pro Nutzer/Zeile und deine Daten liegen bei Airtable. NocoDB ist Open-Source, self-hosted (dein Docker-Container) — gleiche Bedienlogik, aber Daten bleiben auf deinem Server, kein Vendor-Lock-in, keine laufenden Kosten pro Zeile.

Wie das in deine Kette passt

NocoDB ist die strukturierte Datenschicht für alles, was der Organizer-/Brain-Bot verwaltet (Aufgaben, Gewohnheiten, Sport-Challenges). Es läuft in Docker (Lesson 4), auf demselben Server wie die Bots, aber komplett isoliert — die Bots reden nur über HTTP+Token mit der NocoDB-API, nie direkt mit der darunterliegenden Datenbank.

TasksAktuelle Aufgaben
BacklogNoch nicht priorisierte Ideen/Aufgaben
IdeensammlungRohe Ideen, noch unstrukturiert

Quiz: Warum braucht jede Tabelle eine eigene TABLE_ID statt nur einer BASE_ID?

Weil die BASE_ID nur das Projekt/die Datenbank als Ganzes identifiziert — für einen konkreten API-Call (z.B. "füge eine Zeile in Tasks ein") muss NocoDB wissen, welche Tabelle innerhalb dieser Base gemeint ist. Die Tabellen-ID ist der eindeutige Zeiger auf genau eine Tabelle mit ihrem eigenen Feld-Schema.

Quiz: Was passiert konkret bei nocodb_promote_backlog.py?

Der Name legt nahe: das Skript liest einen Eintrag aus der Backlog-Tabelle (NOCODB_BACKLOG_TABLE_ID) und erstellt daraus einen neuen Eintrag in der Tasks-Tabelle (NOCODB_TASKS_TABLE_ID) — eine Idee wird zur aktiven Aufgabe "befördert". Das ist ein typisches Muster: Bot-Logik orchestriert Datenbewegung zwischen NocoDB-Tabellen über deren REST-API, statt dass das in der NocoDB-UI manuell gemacht wird.

Zusammenfassung

NocoDB verwandelt eine Datenbank in eine bedienbare Tabellen-UI + REST-API, ohne dass du selbst ein Admin-Interface bauen musst. Für dein Bot-Army-Setup ist es die zentrale Datenablage für Tasks/Habits/Sport/Backlog — self-hosted in Docker, angesprochen über tabellenspezifische IDs und einen API-Token aus deinen Bot-Skripten.
