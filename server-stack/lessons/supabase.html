Supabase — Backend-as-a-Service für die Shopping-App

~19 Min · Baustein 6/11

Das Problem

Der Shopping Navigator (Mobile App, Flutter) braucht: Nutzer-Accounts, eine Datenbank für Läden/Einkaufslisten/Artikel, Echtzeit-Sync zwischen Geräten (wenn du eine Liste am Handy änderst, soll sie überall aktuell sein) — und das alles, ohne dass du selbst einen Backend-Server (Auth-Logik, DB-Hosting, API-Layer, Skalierung) betreiben musst.

Die Lösung: Postgres + Auth + Realtime, als ein Paket

Supabase ist eine Open-Source-Alternative zu Firebase: eine gehostete Postgres-Datenbank, vor die automatisch generiert werden:

  • Auth — Nutzer-Accounts (bei dir: anonyme Auth, jeder Nutzer bekommt ohne Login-Zwang eine user_id)
  • Realtime — Änderungen an Tabellen werden per WebSocket live an verbundene Clients gepusht
  • Auto-generierte REST/GraphQL-API direkt aus dem Datenbank-Schema
  • Row Level Security (RLS) — Zugriffsregeln direkt in Postgres, nicht im App-Code

Reales Schema aus deiner Roadmap (specs/2026-06-17-roadmap-supabase-design.md):

stores (
  id            uuid primary key,
  user_id       uuid references auth.users,
  geohash       text,
  name          text,
  map_json      jsonb
)

shopping_lists (
  id            uuid primary key,
  user_id       uuid references auth.users,
  store_id      uuid references stores nullable,
  name          text
)

shopping_items (
  id          uuid primary key,
  list_id     uuid references shopping_lists,
  name        text,
  menge       numeric,
  einheit     text,
  kategorie   text,
  wichtigkeit text
)

Wichtig: user_id uuid references auth.users — das ist Supabases eingebaute Auth-Tabelle. Jede Zeile in stores/shopping_lists ist direkt mit einem Nutzer verknüpft, und RLS-Regeln (nicht im Schema-Ausschnitt gezeigt, aber Standardmuster) sorgen dafür, dass ein Nutzer nur seine eigenen Zeilen sieht — ohne dass die App das selbst prüfen muss.

Warum Supabase statt eigenem Backend (z.B. FastAPI + eigene Postgres): Für eine Solo-App-Entwicklung ist "Auth + DB + Realtime + Row-Security selbst bauen" Wochen Arbeit, die nichts mit dem eigentlichen Produkt (A*-Routenplanung, Barcode-Scan) zu tun hat. Supabase liefert diese Grundbausteine fertig und Postgres-basiert (kein proprietäres Firebase-Datenmodell) — das Repository-Pattern in der App (StoreRepository, ShoppingListRepository) kapselt den Zugriff, sodass man später theoretisch tauschen könnte.

Warum genau hier (anders als NocoDB)

NocoDB (Lesson 5) ist für dich persönlich als Admin-Tool gedacht (Tasks/Habits via Telegram-Bot) — eine Handvoll Nutzer, kein Mobile-Client, kein Realtime-Bedarf. Supabase ist für eine App mit potenziell vielen Endnutzern gedacht, die Auth, Mobile-SDKs (supabase_flutter) und Realtime-Sync tatsächlich braucht. Zwei verschiedene Probleme, zwei verschiedene Tools — beide "Datenbank mit UI/API drumherum", aber für komplett unterschiedliche Nutzergruppen.

Wie das in deine Kette passt

Die App selbst wird als statische Flutter-Web-Build über Caddy ausgeliefert (app.thinkshark.de, siehe Lesson 3) — aber die Daten (Listen, Läden, Artikel) laufen nicht über deinen Server, sondern direkt vom Flutter-Client zu Supabases eigener Infrastruktur. Dein Server liefert nur die App-Dateien aus; Supabase ist das komplette Backend dahinter.

Quiz: Warum "anonyme Auth" statt Email/Passwort von Anfang an?

Weil das Feature "geteilte Listen" (mehrere Nutzer sehen dieselbe Liste) in der aktuellen Roadmap noch nicht gebraucht wird — jeder Nutzer bekommt eine stabile user_id, ohne dass er sich registrieren muss. Erst wenn Teilen zwischen echten Accounts nötig wird, lohnt sich der Aufwand für Email-Auth. Klassisches YAGNI-Prinzip: nur so viel Auth-Komplexität wie aktuell gebraucht.

Quiz: Was macht Row Level Security konkret unnötig, was man sonst im App-Code prüfen müsste?

Ohne RLS müsste die App (oder ein eigenes Backend) bei jeder Anfrage selbst prüfen "gehört diese Zeile dem anfragenden Nutzer?". Mit RLS steht diese Regel direkt in Postgres als Policy (z.B. user_id = auth.uid()) — die Datenbank verweigert den Zugriff auf fremde Zeilen automatisch, unabhängig davon, über welchen Client/welche API-Route zugegriffen wird. Ein Sicherheitsfehler im App-Code kann dann keine fremden Daten mehr leaken.

Zusammenfassung

Supabase bündelt Postgres, Auth, Realtime-Sync und Zugriffsregeln (RLS) zu einem sofort nutzbaren Backend für die Shopping-App — Aufgaben, die sonst Wochen eigener Backend-Arbeit wären. Repository-Pattern in der App kapselt den Zugriff. Anders als NocoDB (persönliches Admin-Tool) ist Supabase für einen echten Multi-User-Mobile-Client gebaut.
