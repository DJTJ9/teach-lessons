ecc-Plugin — Subagent-Suite und Delegation

~19 Min · Baustein 11/11

Das Problem

Claude Code selbst ist ein Generalist. Bittest du es, einen Python-Reviewer-Durchlauf zu machen, einen React-Build-Fehler zu beheben und einen Security-Scan durchzuführen — alles im selben Gesprächskontext — wächst der Kontext schnell zu groß, und jede Aufgabe verwässert die spezifische Tiefe der anderen. Ein einzelner System-Prompt kann nicht gleichzeitig "extrem gründlicher Python-Reviewer" und "schneller Build-Fixer" und "Security-Spezialist" optimal sein.

Die Lösung: Spezialisierte Subagenten + Delegation

Das ecc-Plugin (installiert als ecc@ecc, Version 2.0.0 in installed_plugins.json) liefert eine große Bibliothek von Subagenten — jeweils eigener System-Prompt, eigenes Tool-Set, auf genau eine Domäne zugeschnitten:

SubagentJob
python-reviewerPEP-8, Pythonic Idiome, Type Hints, Security
react-build-resolverNur Build-Fehler fixen, keine Architektur-Änderungen
security-reviewerOWASP Top 10, Secrets, Injection-Schwachstellen
database-reviewerSQL-Optimierung, Schema-Design
silent-failure-hunterVerschluckte Fehler, fehlende Error-Propagation

Der Haupt-Agent (Claude Code selbst) ruft diese über das Agent-Tool mit einem subagent_type-Parameter auf — jeder Subagent startet mit frischem, leerem Kontext und nur der spezifischen Aufgabenbeschreibung, die er braucht, nicht dem gesamten bisherigen Gesprächsverlauf.

Warum Delegation statt "alles im Hauptkontext machen": Zwei Gründe. Erstens Spezialisierung — ein Subagent mit System-Prompt "du bist ausschließlich Security-Reviewer, ignoriere Stil-Fragen" liefert fokussiertere, tiefere Ergebnisse als ein Generalist, der nebenbei noch an zehn anderen Dingen denkt. Zweitens Kontext-Schutz — ein Subagent, der 50 Dateien durchsucht, um einen Bug zu finden, würde den Hauptkontext mit Zwischenschritten fluten; er liefert stattdessen nur die komprimierte Erkenntnis zurück, der Rest bleibt in seinem eigenen, isolierten Kontext.

Isolation als Kernprinzip

Jeder Subagent-Aufruf ist wie ein neuer, unabhängiger Mitarbeiter, dem du eine Aufgabe mit vollständigem Briefing gibst (nicht "mach den Rest", sondern konkrete Dateipfade, Zeilen, Kontext) — er hat kein Gedächtnis an frühere Gespräche in dieser Session. Das erzwingt gute Aufgabenbeschreibung (die im Prompt alles Nötige mitgeben muss) und hält den Hauptkontext klein, selbst wenn eine Unteraufgabe viel Rechercheaufwand braucht.

Wie das in deine Kette passt

Für den /teach-Skill, der diese Lesson gerade erzeugt hat, wurde kein Subagent benötigt (reine Content-Erstellung im Hauptkontext) — aber für Aufgaben wie "review diesen Pull Request" oder "finde den Bug in dieser React-Komponente" würde Claude Code automatisch den passenden ecc-Subagenten (code-reviewer, react-build-resolver) delegieren, statt die Spezialaufgabe selbst mit generischem Prompt anzugehen. Das Muster ist dasselbe Prinzip wie bei den Telegram-Bots (Lesson 8): eine Verantwortlichkeit pro Einheit, statt ein Monolith für alles.

Quiz: Warum bekommt ein Subagent nicht automatisch den ganzen bisherigen Gesprächsverlauf?

Weil das den Zweck der Isolation zunichtemachen würde — der Hauptkontext bliebe genauso groß, nur an den Subagenten weitergereicht. Stattdessen bekommt der Subagent gezielt nur die Informationen, die er für seine konkrete Aufgabe braucht (im Prompt explizit mitgegeben) — das zwingt dazu, präzise zu briefen, spart Tokens, und der Subagent kann sich voll auf seine enge Aufgabe konzentrieren.

Quiz: Wann lohnt sich kein Subagent, sondern direkte Bearbeitung im Hauptkontext?

Bei kleinen, schnellen, klar abgegrenzten Aufgaben (ein einzelner Grep, eine Ein-Zeilen-Änderung) — der Overhead, einen Subagenten zu briefen und auf sein Ergebnis zu warten, übersteigt dort den Nutzen der Isolation. Subagenten lohnen sich bei Aufgaben mit viel Rechercheaufwand, hoher Domänenspezifität, oder wenn die Zwischenschritte selbst den Hauptkontext unnötig aufblähen würden.

Zusammenfassung

Das ecc-Plugin liefert eine Bibliothek fachspezifischer Subagenten (Reviewer, Build-Fixer, Security-Scanner je nach Sprache/Domäne), die Claude Code gezielt mit isoliertem, frischem Kontext für eng abgegrenzte Aufgaben aufruft. Das schützt den Hauptkontext vor Aufblähung und liefert fokussiertere Ergebnisse als ein Generalist-Prompt für alles — dasselbe "eine Verantwortlichkeit pro Einheit"-Prinzip, das auch die Bot-Army-Architektur trägt.
