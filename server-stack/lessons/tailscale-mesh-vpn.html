Tailscale — Mesh-VPN zwischen Geräten

~18 Min · Baustein 2/11

Das Problem

Dein Server steht bei einem Hoster, dein PC, iPad und Handy sind woanders — jeweils hinter NAT/Router, ohne feste erreichbare IP. Klassische Lösung wäre: Ports am Server-Firewall öffnen (SSH, App-Ports) und hoffen, dass niemand Unbefugtes sie findet. Jeder offene Port ist eine Angriffsfläche.

Die Lösung: Ein privates Overlay-Netzwerk

Tailscale baut ein virtuelles privates Netzwerk (VPN) auf Basis von WireGuard auf, in dem jedes deiner Geräte eine feste private IP im Bereich 100.64.0.0/10 (CGNAT-Range) bekommt — unabhängig davon, wo es gerade physisch steht oder welches lokale Netz es nutzt.

Dein aktueller Tailnet (Auszug aus tailscale status):

100.64.150.74  dev                 (dieser Server, Linux)
100.88.190.7   desktop-vdik3ep     (Windows-PC)
100.112.87.51  ipad-air-gen-4      (iOS)
100.67.54.96   z-flip5-von-tjark   (Android, Termux-Gerät)

Jedes Gerät erreicht jedes andere über diese 100.x.x.x-Adresse, als wären sie im selben LAN — egal ob eins im Hoster-Rechenzentrum steht und das andere im Handynetz hängt.

Warum "Mesh" und nicht "Server-Client-VPN"

Klassisches VPN: alle Geräte verbinden sich zu einem zentralen VPN-Server, der Traffic weiterleitet (Flaschenhals, single point of failure). Mesh: jedes Gerät verbindet sich (wenn möglich) direkt mit jedem anderen — Tailscale nutzt dafür NAT-Traversal-Techniken; nur wenn eine direkte Verbindung unmöglich ist, wird über einen Relay (DERP-Server) vermittelt.

Warum genau Tailscale statt eigenem WireGuard-Setup: WireGuard selbst konfigurierst du pro Gerätepaar manuell (Keys austauschen, IPs zuweisen, Firewall-Regeln). Tailscale übernimmt Key-Distribution, NAT-Traversal und Gerätediscovery automatisch über einen Koordinationsserver — bei 4 Geräten in deinem Tailnet ist das der Unterschied zwischen 5 Minuten Setup und Stunden Fummelei.

Wie das in deine Kette passt

Tailscale ist der Grund, warum du SSH nie öffentlich exponieren musst: Du verbindest dich zum Server nicht über seine öffentliche IP, sondern über 100.64.150.74 — erreichbar nur für Geräte in deinem Tailnet. Öffentlich zugänglich ist ausschließlich das, was bewusst über Cloudflare+Caddy freigegeben wird (Bots, App). Zwei getrennte Zugriffsebenen: Tailscale = "nur meine Geräte", Cloudflare+Caddy = "die Öffentlichkeit, aber kontrolliert".

Quiz: Warum bekommt jedes Gerät eine feste 100.x.x.x-IP statt seiner echten lokalen IP zu nutzen?

Weil lokale IPs (die dein Router per DHCP vergibt) sich ändern können und hinter NAT liegen — von außen nicht adressierbar. Die Tailscale-IP ist stabil und im gesamten Tailnet gültig, egal welches physische Netz das Gerät gerade nutzt (Heim-WLAN, Mobilfunk, Firmennetz). Das macht sie zur zuverlässigen Adresse für SSH-Configs, Skripte etc.

Quiz: Was passiert, wenn zwei Geräte hinter symmetrischem NAT (z.B. restriktives Mobilfunknetz) keine direkte Verbindung aufbauen können?

Tailscale fällt automatisch auf einen DERP-Relay-Server zurück (von Tailscale betrieben) — der Traffic bleibt Ende-zu-Ende verschlüsselt (WireGuard), wird aber über den Relay weitergeleitet statt direkt. Langsamer als eine Direktverbindung, aber funktioniert immer.

Zusammenfassung

Tailscale gibt allen deinen Geräten eine stabile, private Adresse im selben virtuellen Netz — ohne Ports am Server öffnen zu müssen. Es ist die Verwaltungs-Ebene für SSH-Zugriff (Lesson 1) und macht Termux auf dem Handy (Lesson 9) zu einem vollwertigen Client, der genauso auf den Server zugreift wie dein PC.
