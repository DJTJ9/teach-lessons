# Mission

## Why
Als angehender Spiele-/App-Entwickler die komplette Server- und Tool-Kette hinter den eigenen Projekten (telegram-bot-army, shopping-navigator, hub) selbst verstehen und begründen können — nicht nur benutzen, sondern im Bewerbungsgespräch oder bei einem neuen Projekt erklären können, warum genau diese Tools gewählt wurden.

## Success looks like
Für jede der 11 Komponenten kann der Lerner ohne Nachschlagen erklären:
- welches konkrete Problem sie löst
- warum genau dieses Tool statt einer Alternative gewählt wurde
- wie sie mit den anderen Komponenten in der Kette zusammenspielt (z.B. Client → Cloudflare → Caddy → Bot)

Komponenten: SSH Key-Auth, Tailscale, Cloudflare+Caddy (TLS-Kette), Docker, NocoDB, Supabase, systemd, Telegram-Bot-Army-Architektur, Termux, Dev-Skill+Memory-System, ecc-Plugin (Subagent-Suite).

## Constraints
- Vorwissen Netzwerk: Grundkenntnisse (DNS/SSH/Port bekannt, aber nicht die konkrete Architektur hier)
- Vorwissen Claude Code: aktiver Nutzer (/dev, /teach etc.), will Innenleben verstehen
- Lesson-Tiefe: ausführlich (15-20 Min), mit Beispielen aus den echten Server-Configs
- Alle Beispiele sollen auf realen Configs/Code aus diesem Server basieren (Caddyfile, systemd-Units, installed_plugins.json, etc.)

## Out of scope
CouchDB+Obsidian LiveSync, Notion API, GitHub Pages, superpowers-Workflow, sonstige CC-Plugins (caveman, codex, frontend-design, security-guidance, dataviz) — im Survey gefunden, aber vom Lerner nicht ausgewählt.
