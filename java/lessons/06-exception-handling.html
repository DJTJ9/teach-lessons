☕ Lektion 06 — Exception Handling

Ziel: try/catch/finally in Java schreiben, und den größten Java-Unterschied zu C# verstehen: Checked Exceptions.

try/catch/finally — fast identisch zu C#

☕ Java

try {
    int ergebnis = 10 / 0;
} catch (ArithmeticException e) {
    System.out.println("Fehler: " + e.getMessage());
} catch (Exception e) {
    System.out.println("Unbekannter Fehler");
} finally {
    System.out.println("Wird immer ausgeführt");
}

💜 C#

try {
    int ergebnis = 10 / 0;
} catch (DivideByZeroException e) {
    Console.WriteLine("Fehler: " + e.Message);
} catch (Exception e) {
    Console.WriteLine("Unbekannter Fehler");
} finally {
    Console.WriteLine("Wird immer ausgeführt");
}

if (alter < 0) {

Der größte Unterschied: Checked Exceptions

Das ist eine Java-Eigenheit die C# nicht kennt. In Java gibt es zwei Arten von Exceptions:

TypWasMuss behandelt werden?
Checked ExceptionErbt von Exception (nicht RuntimeException)Ja — Compiler erzwingt es
Unchecked ExceptionErbt von RuntimeExceptionNein — optional
// Checked Exception — muss behandelt werden:
try {
    FileReader file = new FileReader("daten.txt");  // FileNotFoundException = Checked!
} catch (FileNotFoundException e) {
    System.out.println("Datei nicht gefunden: " + e.getMessage());
}

// Erzwungen — ohne try/catch: KOMPILIERUNGSFEHLER!
// Unchecked Exception — optional:
int[] array = {1, 2, 3};
System.out.println(array[10]);  // ArrayIndexOutOfBoundsException — Runtime-Fehler
// Kein try/catch nötig (aber empfohlen)

throws-Keyword — Exception weitergeben

// Statt selbst zu behandeln, nach oben weitergeben:
public void dateiLesen(String pfad) throws IOException {
    FileReader file = new FileReader(pfad);
    // IOException wird an den Aufrufer weitergegeben
}

// Aufrufer muss dann behandeln:
try {
    dateiLesen("daten.txt");
} catch (IOException e) {
    System.out.println("Fehler: " + e.getMessage());
}
Wichtiger Unterschied zu C#: In C# gibt es keine Checked Exceptions — alle Exceptions sind optional zu behandeln. Java zwingt dich bei bestimmten Exceptions (IOException, SQLException, etc.) dazu, sie entweder zu behandeln (try/catch) oder weiterzugeben (throws). Das macht Code expliziter aber auch geschwätziger.

Eigene Exception-Klasse erstellen

// Eigene Checked Exception:
public class SpielFehlerException extends Exception {
    public SpielFehlerException(String nachricht) {
        super(nachricht);
    }
}

// Eigene Unchecked Exception:
public class UngueltigerZugException extends RuntimeException {
    public UngueltigerZugException(String nachricht) {
        super(nachricht);
    }
}

// Nutzen:
throw new UngueltigerZugException("Zug außerhalb des Spielfelds");

try-with-resources (Java 7+)

// Automatisches Schließen von Ressourcen (wie C# using):
try (FileReader file = new FileReader("daten.txt");
     BufferedReader reader = new BufferedReader(file)) {
    String zeile = reader.readLine();
    System.out.println(zeile);
}
// file und reader werden automatisch geschlossen — auch bei Exception

🧠 Quiz

Frage 1: Was ist eine Checked Exception in Java?

A) Eine Exception die der Compiler zu behandeln zwingt
✅ Richtig! Entweder try/catch oder throws — der Compiler akzeptiert nichts anderes.
B) Eine Exception die selten auftritt
❌ Falsch. Checked/Unchecked beschreibt Compile-Zeit-Zwang, nicht Häufigkeit.

Frage 2: Was ist das Java-Äquivalent zu C# using (var f = new File())?

A) try-with-resources: try (File f = new File())
✅ Richtig! Gleiche Semantik — automatisches Schließen nach dem Block.
B) finally { f.close(); }
❌ Geht auch, aber umständlicher. try-with-resources ist der moderne Weg.

Aufgabe: Schreibe eine Methode dividieren(int a, int b) die eine eigene DivisionDurchNullException (unchecked) wirft wenn b = 0.

Zeig mir die Lösung
public class DivisionDurchNullException extends RuntimeException {
    public DivisionDurchNullException() {
        super("Division durch Null ist nicht erlaubt");
    }
}

public int dividieren(int a, int b) {
    if (b == 0) throw new DivisionDurchNullException();
    return a / b;
}

📋 Zusammenfassung

  • try/catch/finally: fast identisch zu C#
  • Checked Exceptions: Compiler erzwingt Behandlung oder throws — nur in Java!
  • Unchecked Exceptions: RuntimeException und Unterklassen — optional
  • throws im Methodenkopf: Exception nach oben weitergeben
  • try-with-resources: automatisches Schließen (= C# using)
  • Eigene Exceptions: extends Exception (checked) oder RuntimeException (unchecked)
