/dev  (oder /dev-status)

Lektion 3 von 9 · Status-Anzeige und Routing

Was macht dieser Befehl?

/dev ohne Argument (oder /dev-status) ist der Orientierungs-Befehl: Er liest STATUS.md und zeigt, was gerade aktiv ist — und bietet den direkten Weitersprung zur richtigen Phase an.

Warum braucht man das?
Nach einer Pause (Schlaf, anderen Tasks, einem anderen Projekt) weiß man oft nicht mehr: "Wo war ich? Welcher Befehl kommt als nächstes?" /dev beantwortet diese Frage in Sekunden — ohne Dateien manuell zu lesen.

STATUS.md — Die Datei dahinter

STATUS.md ist die einzige Datei, die /dev liest. Ihr Format ist strikt:

# Project Status — mein-projekt
Active: Task-Filter ← aktuell bearbeitetes Feature
Phase: implement ← none | plan | implement | finish
Spec: topics/mein-projekt/specs/2026-06-10-task-filter-design.md
Plan: topics/mein-projekt/plans/2026-06-10-task-filter.md
Updated: 2026-06-23

## Roadmap
- [idea] Dark Mode
- [discussed] Push-Notifs
- [planned] Task-Filter ← aktiv
- [done] ✅ Login-Flow implementiert 2026-06-01
Warum so strikt formatiert?
Jeder Skill liest STATUS.md maschinell — kein Parsing per Regex, sondern direkte Zeilenabgleiche. Das Format ist so einfach wie möglich, damit es robust ist: Ein Feldname pro Zeile, Doppelpunkt als Trennzeichen. Mensch und Maschine können es gleich gut lesen.

Was /dev anzeigt

▶ Aktiv:    Task-Filter (implement)
   Spec/Plan: topics/.../specs/...design.md
▷ Nächstes: Push-Notifs
💡 Ideen:   Dark Mode

Fortfahren?
A) Ja — aktuelle Phase weitermachen
B) Anderen Schritt wählen

Routing-Logik

Wenn du A wählst, leitet /dev dich automatisch zur richtigen Phase-Datei weiter:

Phase-WertWeiterleitung zuNächste Aktion
(none)—Schlägt /dev-discuss oder /dev-idea vor
planphases/discuss.mdWeiter mit /dev-plan
implementphases/plan.md → implement.mdWeiter mit /dev-implement
finishphases/finish.mdWeiter mit /dev-finish
Warum Routing über STATUS.md statt manuell?
Du musst nicht wissen, in welcher Phase du bist — der Skill weiß es. Das verhindert den häufigen Fehler, /dev-plan zu starten obwohl noch kein /dev-discuss gelaufen ist. STATUS.md ist der einzige Source of Truth.

Slug-Erkennung

/dev findet den Slug genauso wie alle anderen Dev-Befehle:

  1. Liest $HUB_DIR/projects-registry.json
  2. Matched CWD gegen jeden path-Eintrag
  3. Wenn Treffer → Slug automatisch gesetzt
  4. Wenn kein Treffer → zeigt die 4 zuletzt aktiven Projekte als Auswahl

Wenn STATUS.md fehlt

Falls kein STATUS.md für das erkannte Projekt existiert, gibt /dev aus:

Kein STATUS.md für dieses Projekt. Starte mit `/dev init`.

Quiz

1. Welche Datei liest /dev (ohne Argument)?
  • VISION.md
  • STATUS.md
  • projects-registry.json
2. Was passiert wenn du Option "A) Ja" bei Phase=finish wählst?
  • Der Skill startet /dev-plan
  • Der Skill leitet zur Finish-Phase weiter (phases/finish.md)
  • Der Skill fragt nach dem nächsten Feature
3. Was zeigt /dev unter "💡 Ideen"?
  • Alle Roadmap-Items
  • Nur [idea]-Items aus der Roadmap
  • Das aktuelle Feature

Zusammenfassung

  • /dev = Orientierung nach jeder Pause
  • Liest STATUS.md und zeigt: aktives Feature, Phase, nächsten Punkt, Ideen
  • Option A: automatisches Routing zur richtigen Phase
  • STATUS.md-Format ist strikt — Feldname + Doppelpunkt, eine Zeile pro Feld
  • Wenn STATUS.md fehlt → Hinweis auf /dev-init
