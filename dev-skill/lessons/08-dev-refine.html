Was macht dieser Befehl?

/dev-refine öffnet eine bestehende Spec und überarbeitet sie gezielt — ohne eine neue Spec-Datei anzulegen. Die bestehende Datei wird in-place aktualisiert, STATUS.md wird zurück auf Phase: plan gesetzt, und das Plan-Feld wird geleert.

Warum ein eigener Refine-Befehl statt einfach /dev-discuss nochmal?
/dev-discuss geht durch den vollen Dialog: Feature wählen, alle Clarifying Questions, alle Ansätze, alle Sektionen. Das kostet Zeit wenn die Spec nur in einem Punkt falsch ist. /dev-refine springt direkt zur relevanten Änderung — "Was soll geändert werden?" statt alles neu aufzurollen.

Wann /dev-refine nutzen?

✓Scope-Grenze war falsch definiert — ein Teilbereich muss raus oder rein
✓Architektur-Entscheidung nach nochmaligem Überlegen geändert
✓Neue technische Constraints entdeckt (Bibliothek nicht kompatibel etc.)
✓UI-Wireframes müssen angepasst werden
✗Komplett neues Feature — dafür /dev-discuss
✗Spec war so falsch dass ein Neuanfang besser wäre — dafür /dev-discuss

Warum wird das Plan-Feld geleert?
Der bestehende Plan (falls vorhanden) wurde auf Basis der alten Spec geschrieben. Wenn die Spec sich ändert, ist der Plan potenziell invalid. Durch Leeren des Plan-Felds wird erzwungen, dass /dev-plan nochmal läuft — mit der aktualisierten Spec als Basis.

Knowledge Extraction nach Refine

Auch nach einem Refine läuft Knowledge Extraction mit context="discuss". Wenn der Grund für die Spec-Änderung eine neue Erkenntnis war, landet diese in DECISIONS.md:

## [2026-06-23] Scope-Reduktion: Server-Side Filter verworfen
- Kategorie: architecture
- Grund: Latenz-Anforderungen nicht erfüllbar mit aktuellem DB-Schema
- Outcome: offen — Client-Side Filter gewählt

Quiz

1. Was macht /dev-refine mit der bestehenden Spec-Datei?
  • Erstellt eine neue Datei mit neuem Datum
  • Überschreibt die bestehende Datei in-place (selber Dateiname)
  • Löscht die alte und erstellt eine neue Datei
2. Warum wird das Plan-Feld nach /dev-refine geleert?
  • Weil ein bestehender Plan auf der alten Spec basiert — die Spec-Änderung macht ihn potenziell invalid
  • Weil Plan-Dateien automatisch gelöscht werden
  • Um Speicherplatz zu sparen
3. Wann ist /dev-discuss besser als /dev-refine?
  • Wenn die Architektur geändert werden muss
  • Wenn die Spec so grundlegend falsch ist dass ein kompletter Neustart sinnvoller ist
  • Wenn die UI-Wireframes geändert werden

Zusammenfassung

  • Gezieltes Überarbeiten der bestehenden Spec — kein Neuanfang
  • Springt direkt zur Frage "Was soll geändert werden?" — kein voller Discuss-Dialog
  • Spec-Datei wird in-place überschrieben (selber Pfad)
  • Plan-Feld in STATUS.md wird geleert → /dev-plan muss danach nochmal laufen
  • Knowledge Extraction: Neuer Entscheidungsgrund → DECISIONS.md
  • Wann nutzen: Kleine bis mittlere Spec-Korrekturen, neue Constraints, Scope-Anpassungen
