/dev-plan

Lektion 5 von 9 · Implementierungsplan schreiben

Was macht dieser Befehl?

/dev-plan liest die Spec aus Phase 1 und schreibt daraus einen vollständigen, ausführbaren Implementierungsplan — mit echtem Code, TDD-Struktur und keinen Platzhaltern. Das Ergebnis ist eine plans/*.md-Datei, die direkt von einem Agenten ausgeführt werden kann.

Warum ein separater Plan statt direkt zu coden?
Ein Plan ohne Spec ist Raten. Eine Spec ohne Plan ist eine gute Absicht. Erst wenn beides da ist, kann man sicher implementieren — weil klar ist: welche Dateien, welche Interfaces, welche Reihenfolge, wie jeder Schritt getestet wird.

Schritt 1: File Map

Bevor Tasks geschrieben werden, muss eine vollständige Datei-Übersicht her:

Create: src/filters/task-filter.ts  — Filter-Logik
Modify: src/ui/task-list.tsx  — Filter-UI einbauen
Modify: src/store/tasks.ts  — Filter-State hinzufügen
Test:    tests/filters/task-filter.test.ts  — Unit-Tests
Warum erst File Map, dann Tasks?
Wenn man direkt Tasks schreibt, entstehen Lücken: Datei A vergessen, Datei B implizit vorausgesetzt. Die File Map zwingt zur vollständigen Inventur zuerst. Jede Datei mit einem klaren Zweck — keine Datei die "irgendwie passt".

Schritt 2: TDD-Tasks

Jeder Task = kleinste unabhängig testbare Einheit. Jeder Task hat denselben 5-Schritt-Zyklus:

Step 1
Test schreiben
Echter Test-Code
Step 2
FAIL
Erwarteter Fehler
Step 3
Implementieren
Echter Impl-Code
Step 4
PASS
Erwartetes Ergebnis
Step 5
Commit
Exakter Befehl
Warum TDD im Plan erzwingen?
Ohne Test-First werden Tests nachträglich geschrieben — und testen dann nur was der Code macht, nicht was er machen soll. TDD kehrt das um: Erst der Wunsch (Test), dann die Erfüllung (Impl). Der erwartete FAIL in Step 2 ist der Beweis, dass der Test wirklich etwas prüft.

Wie ein Task aussieht

### Task 1: Filter-Logik

Files
Create: src/filters/task-filter.ts Test: tests/filters/task-filter.test.ts
Interfaces
Produces: filterTasks(tasks: Task[], filter: FilterConfig): Task[] 
- [ ] Step 1: Test schreiben
  test('filtert nach Status', () => {
    const tasks = [{ id:1, status:'done' }, { id:2, status:'open' }]
    expect(filterTasks(tasks, { status:'open' })).toEqual([{ id:2, status:'open' }])
  })

- [ ] Step 2: Run — expect FAIL
  Run: npm test task-filter
  Expected: FAIL — "filterTasks is not a function"

- [ ] Step 3: Implementieren
  export function filterTasks(tasks, filter) { ... }

- [ ] Step 4: Run — expect PASS
  Expected: PASS

- [ ] Step 5: Commit
  git add src/filters/ tests/filters/
  git commit -m "feat(task-filter): add status filter"
Kein Placeholder erlaubt. Weder // TODO, noch // similar to Task 1, noch // appropriate error handling. Jeder Schritt enthält echten Code — sonst kann kein Agent den Plan ausführen ohne Rückfragen.

Schritt 3: Self-Review

Bevor der Plan commitet wird, prüft der Skill drei Dinge:

  1. Spec coverage: Jede Spec-Anforderung hat einen Task? Lücken werden gelistet.
  2. Kein Placeholder: Kein TBD, kein TODO, kein "similar to..."
  3. Type/Name-Konsistenz: Was Task N als Produces listet, muss exakt in Task N+1 als Consumes auftauchen.

Der Plan-Header (für Agenten)

# Task-Filter Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development
> Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Task-Filter nach Status, Priorität und Datum
**Architecture:** Filter-Logik rein funktional in eigenem Modul...
**Tech Stack:** TypeScript, Vitest
Warum der Agenten-Hinweis im Header?
Der Plan wird oft in einer neuen Session ausgeführt — ohne den Kontext dieser Conversation. Der Header gibt dem frischen Agenten alle Infos in 3 Zeilen: was, wie, womit.

Quiz

1. Was ist der Zweck von "Step 2: FAIL" in jedem TDD-Task?
  • Debugging-Hilfe
  • Beweis, dass der Test tatsächlich etwas prüft (nicht schon vor der Implementierung grün ist)
  • Dokumentation was schiefgehen kann
2. Was ist die Konsequenz wenn Task 2 einen Funktionsnamen anders benennt als Task 1 ihn produziert?
  • Der Plan ist inkonsistent und ein Agenten-Fehler wird auftreten — Self-Review erkennt das
  • Kein Problem, der Agent findet den richtigen Namen
  • Der Skill korrigiert es automatisch
3. Warum kommt die File Map vor den Tasks?
  • Damit die Dateiliste im Output schön aussieht
  • Um zuerst vollständige Inventur aller betroffenen Dateien zu machen, bevor einzelne Tasks geschrieben werden
  • Weil Agenten die File Map zuerst brauchen

Zusammenfassung

  • Phase 2: Spec → ausführbarer Plan
  • Erst File Map (vollständige Datei-Inventur), dann Tasks
  • Jeder Task: Test schreiben → FAIL → Impl → PASS → Commit
  • Kein Placeholder — echter Code in jedem Schritt
  • Self-Review: Spec-Coverage, kein Placeholder, Name-Konsistenz
  • Output: plans/YYYY-MM-DD-<feature>.md
  • Danach automatisch: STATUS.md auf Phase: implement
