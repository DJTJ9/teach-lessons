/dev-init

Lektion 1 von 9 · Projekt-Initialisierung

Was macht dieser Befehl?

/dev-init richtet ein neues Projekt im Hub ein. Er läuft genau einmal pro Projekt — danach nie wieder. Er legt alle Dateien an, die der Rest des Workflows braucht.

Warum ein eigener Init-Schritt?
Ohne Init gibt es keinen definierten Ausgangszustand. Spätere Befehle wie /dev-discuss lesen STATUS.md, DECISIONS.md und VISION.md — wenn diese fehlen, bricht der Workflow ab. Init stellt sicher, dass diese Dateien von Anfang an existieren und korrekt formatiert sind.

Ablauf Schritt für Schritt

1

Projekt identifizieren

Der Skill erkennt anhand der CWD, ob das Projekt schon in der Registry ist. Wenn nicht, fragt er: "Projektname?" und "Slug? (Vorschlag: kebab-case)".

Warum Slug? Alle Hub-Pfade basieren auf dem Slug: $HUB_DIR/topics/<slug>/. Kebab-case ist URL-sicher und Git-freundlich.
2

Registry-Eintrag anlegen

Der Skill fügt einen Eintrag in $HUB_DIR/projects-registry.json ein:

{ "slug": "mein-projekt", "name": "Mein Projekt", "path": "/root/projekte/mein-projekt", "path_windows": "", "repo": "", "description": "" }
Warum Registry? Jeder /dev-Befehl findet den Slug automatisch anhand der CWD — du musst ihn nie manuell angeben. Die Registry ist die einzige Quelle der Wahrheit für Projekt-Metadaten.
3

Vision-Interview (3 Fragen)

Der Skill stellt drei Fragen für VISION.md:

  • "Was ist das Projekt?"
  • "Für wen?"
  • "Was ist das Hauptziel?"
Warum VISION.md? Eine schriftliche Vision verhindert Scope-Creep. Wenn Feature-Ideen kommen, kann man sie gegen die Vision prüfen. VISION.md ist auch der Ort, wo die Roadmap akkumuliert — jedes fertige Feature landet dort als ✅.
4

Alle Basis-Dateien schreiben

Der Skill erstellt das vollständige Hub-Verzeichnis:

$HUB_DIR/topics/<slug>/
  VISION.md  ← Vision + leere Roadmap
  STATUS.md  ← Phase: (none), Active: (none)
  DECISIONS.md  ← leer, wartet auf Discuss-Phase
  LEARNINGS.md  ← leer, wartet auf Finish-Phase
  TIMELINE.md  ← leer, wartet auf Finish-Phase
  specs/  ← leer, wartet auf Discuss-Phase
  plans/  ← leer, wartet auf Plan-Phase
Warum diese 5 Dateien? Jede hat eine eigene Lebensdauer:
STATUS.md ist flüchtig (ändert sich mit jeder Phase), DECISIONS/LEARNINGS/TIMELINE akkumulieren über Monate. Durch die Trennung bleibt der Projektkontext dauerhaft lesbar — auch wenn STATUS.md täglich wechselt.
5

CLAUDE.md im Projekt-Repo (optional)

Wenn ein lokales Repo vorhanden ist, erstellt der Skill CLAUDE.md mit @-Imports:

@/root/projects-hub/topics/<slug>/DECISIONS.md
@/root/projects-hub/topics/<slug>/LEARNINGS.md
Warum @-Imports? Claude lädt DECISIONS.md und LEARNINGS.md automatisch in den Kontext, wenn du in diesem Repo arbeitest. Dadurch kennt Claude alle vergangenen Entscheidungen — ohne dass du sie jedes Mal neu erklären musst. Das ist der Kern des "project memory" Konzepts.
6

Commit

git -C "$HUB_DIR" add topics/<slug>/ projects-registry.json
git -C "$HUB_DIR" commit -m "chore(<slug>): initialize developer memory"

Der Skill commitet direkt auf dem Hub-Repo — das Projekt-Repo wird hier nicht berührt.

Das Ergebnis

Nach /dev-init zeigt /dev (Status) folgendes:

▶ Aktiv:    (none)
▷ Nächstes: (keine Roadmap-Items)

Fortfahren?
A) Ja — aktuelle Phase weitermachen
B) Anderen Schritt wählen

Starte danach mit /dev-idea "text" um Ideen zu sammeln, oder direkt mit /dev-discuss.

Häufige Fehler

  • Init mehrfach aufrufen: Kein Problem — der Registry-Eintrag wird doppelt, die Dateien werden überschrieben. Besser: Init wirklich nur einmal nutzen.
  • Slug mit Großbuchstaben/Leerzeichen: Kebab-case erzwingen. mein-projekt ✓, MeinProjekt ✗
  • CWD vergessen: Der Skill braucht CWD im Projekt-Verzeichnis, um den Pfad korrekt in die Registry einzutragen.

Quiz

1. Wie oft läuft /dev-init pro Projekt?
  • Einmal pro Feature
  • Einmal pro Projekt
  • Bei jedem /dev-discuss
2. Warum werden DECISIONS.md und LEARNINGS.md per @-Import in CLAUDE.md eingebunden?
  • Damit Claude die Dateien bearbeiten kann
  • Damit vergangene Entscheidungen automatisch im Kontext sind — ohne manuelle Wiederholung
  • Als Backup der Dateien im Projekt-Repo
3. Was ist der Unterschied zwischen VISION.md und STATUS.md?
  • VISION.md ist für Features, STATUS.md für Bugs
  • VISION.md akkumuliert langfristig (Roadmap, fertige Features), STATUS.md ist flüchtig (aktuelle Phase)
  • Kein Unterschied — beide enthalten das Gleiche

Zusammenfassung

  • /dev-init läuft einmal pro Projekt
  • Erstellt Hub-Verzeichnis mit 5 Dateien + 2 Ordnern + Registry-Eintrag
  • Der Slug ist das Fundament aller weiteren Hub-Pfade
  • CLAUDE.md mit @-Imports gibt Claude dauerhaften Projektspeicher
  • Danach: /dev-idea oder /dev-discuss
