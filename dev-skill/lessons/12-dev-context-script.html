Lektion 12: dev_context.py

Token-optimierte Projektauswahl via Python-Skript statt LLM-Leserei

Das Problem

Vorher: LLM liest alles

Claude liest alle STATUS.md-Dateien, parsed sie in seinem Kopf, filtert Features, zeigt Auswahl an.

~800 Tokens pro Projekt
Nachher: Script liefert JSON

Ein Bash-Call gibt gefilterte JSON-Ausgabe. Claude zeigt Auswahl direkt an.

~50 Tokens gesamt
Kernprinzip: Für strukturierte Daten (STATUS.md parsen, filtern, sortieren) ist ein Python-Skript immer besser als LLM-Lesearbeit. LLM arbeitet gut mit Bedeutung, schlecht mit Mustererkennung auf Dateisystem-Ebene.

Wo liegt das Skript?

$HUB_DIR/scripts/dev_context.py
→ stdlib only: argparse, json, os, sys, pathlib
→ kein requirements.txt, keine venv nötig
→ liest direkt aus $HUB_DIR/topics/*/STATUS.md

Interface

ArgumentBedeutung
--command statusAlle Projekte, alle Features
--command visionAlle Projekte, alle Features
--command discussFeatures mit Status [vision] oder [idea]
--command planFeatures mit Status [discussed]
--command implementFeatures mit Status [planned]
--slug <slug>Nur ein Projekt — gibt Feature-Array zurück

Beispielaufruf (Skill)

# Im Skill-Step, wenn kein CWD-Projekt erkannt:
projects_json=$(python "$HUB_DIR/scripts/dev_context.py" \
  --command discuss --hub-dir "$HUB_DIR")

# Wenn Projekt bekannt, nur Features filtern:
features_json=$(python "$HUB_DIR/scripts/dev_context.py" \
  --command implement --slug my-project --hub-dir "$HUB_DIR")

Was der Skill damit macht

Bash: dev_context.py
→
JSON ausgabe
→
AskUserQuestion
→
Session startet

Multi-Select (multiSelect: true) ist auch möglich: kleine Features zusammenfassen, dann als gebündelte Session bearbeiten.

STATUS.md Roadmap-Format

Das Skript parsed genau dieses Format:

## Roadmap
- [idea]      Schnelle Idee ohne Spec
- [vision]    Besprochen, Triage OK
- [discussed] Spec vorhanden
- [planned]   Plan fertig, bereit für Implement
- [active]    Wird gerade implementiert
- [done]      Fertig

Hinter dem Status-Tag kann optional ein #-Kommentar stehen — wird beim Parsen abgeschnitten.

Schlüsselfunktion: parse_status_md

def parse_status_md(path):
    lines = path.read_text().splitlines()
    result = {"active": "", "phase": "", "features": []}
    in_roadmap = False
    for line in lines:
        if line.startswith("Active:"): ...
        elif line.startswith("## Roadmap"): in_roadmap = True
        elif in_roadmap and line.startswith("- ["):
            # parst [status] und name (ohne #-Kommentare)
            ...
    return result
Design-Entscheidung: stdlib-only bedeutet kein pip install, keine venv, keine Abhängigkeit. Überall lauffähig wo Python 3 vorhanden. Skript fällt auch nicht aus wenn ein STATUS.md fehlt — es gibt None zurück.

Sortierung

Projekte werden nach mtime der STATUS.md sortiert — zuletzt bearbeitetes Projekt zuerst. So landet das relevanteste Projekt oben ohne dass der LLM sortieren muss.

Quiz

1. Warum gibt dev_context.py JSON aus statt Text?

  • Weil JSON schneller zu parsen ist als Text
  • Claude kann JSON direkt in AskUserQuestion-Optionen umwandeln, ohne selbst zu parsen
  • Weil das STATUS.md-Format JSON ist
  • Wegen Telegram-Integration

2. Welcher --command-Wert liefert Features mit Status [discussed]?

  • --command discuss
  • --command plan
  • --command implement
  • --command status

3. Was ist der Hauptvorteil von stdlib-only?

  • Schnellere Ausführung
  • Bessere Fehlerbehandlung
  • Keine Installation nötig — überall lauffähig
  • Kleinere Dateigröße

Zusammenfassung

  • dev_context.py ersetzt LLM-Leserei: ~800 → ~50 Token pro Aufruf
  • Skript liegt in $HUB_DIR/scripts/, stdlib-only, keine venv
  • --command filtert nach Phase: discuss→[vision|idea], plan→[discussed], implement→[planned]
  • --slug für bekanntes Projekt — gibt nur Feature-Array zurück
  • Projekte sortiert nach mtime — zuletzt bearbeitetes zuerst
  • Skill nutzt JSON-Output direkt für AskUserQuestion mit optional multiSelect: true
