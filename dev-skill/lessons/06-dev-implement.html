/dev-implement

Lektion 6 von 9 · Plan ausführen

Was macht dieser Befehl?

/dev-implement ist Phase 3: Er liest den Plan aus STATUS.md und startet die eigentliche Implementierung. Dabei werden drei Dinge geklärt: Ausführungsmodus, Zeitplanung, und das implementation_mode-Flag wird gesetzt.

Warum ist Implement eine eigene Phase und kein Teil von Plan?
Plan und Implement können zeitlich weit auseinanderliegen: Plan morgens schreiben, Implement abends starten — oder an einem anderen Tag. Durch die Trennung kann man den Plan reviewen, ändern, oder verwerfen bevor Ressourcen in die Umsetzung fließen.

Schritt 1: Ausführungsmodus wählen

A) Subagent-Driven empfohlen
Modus: Frischer Agent pro Task
Review: Nach jedem Task
Kontext: Jeder Agent startet clean
Vorteil: Kein Kontext-Drift, klare Task-Grenzen
B) Inline
Modus: Alle Tasks in dieser Session
Review: Am Ende
Kontext: Wächst mit jedem Task
Vorteil: Schneller bei kleinen Plänen
Warum Subagent-Driven bevorzugt?
Ein einzelner langer Kontext akkumuliert Fehler: Frühe Annahmen beeinflussen späte Tasks. Ein frischer Agent pro Task verhindert das. Außerdem erlaubt Review zwischen Tasks, Fehler früh zu erkennen — bevor sie sich in 5 weitere Tasks fortpflanzen.

Schritt 2: Zeitplanung (Scheduling)

14:30Geplant für 14:30 Uhrscheduled_time: "14:30"
späterKein Zeitpunkt, wartet auf manuellen Startscheduled_time: null
Warum Scheduling in der Implement-Phase?
Implementierungen laufen oft lang (30+ Minuten für große Features). Scheduling entkoppelt "Plan reviewen" von "Maschine arbeiten lassen" — man kann um 23:00 den Plan finalisieren und auf 02:00 schedulen.

Schritt 3: implementation_mode setzen

Bevor die Implementierung startet, setzt der Skill ein Flag in settings.json:

{
  "implementation_mode": true,
  "implementation_mode_until": "2026-06-23T18:30:00"
}
Wichtig: Dieses Flag wird mit dem Edit-Tool gesetzt, nicht via Bash. Nach Abschluss aller Implementierungsschritte wird es zurückgesetzt:
implementation_mode: false und implementation_mode_until: null
Warum dieses Flag?
Der Permission-Hook (GateGuard und andere) prüft dieses Flag. Wenn es gesetzt ist, werden alle Tool-Calls auto-genehmigt — kein Telegram-Relay, kein manuelles Bestätigen. Das erlaubt fully-autonomous Implementierung ohne ständige Unterbrechungen. Das Flag läuft nach 4 Stunden ab, als Sicherheitsnetz.

Schritt 4: Ausführen

Je nach gewähltem Modus ruft der Skill einen der zwei Superpowers-Skills auf:

  • Subagent-Driven: superpowers:subagent-driven-development — spawnt Agenten pro Task
  • Inline: superpowers:executing-plans — führt Plan in dieser Session aus

Nach der Implementierung

STATUS.md wird automatisch aktualisiert:

  • Phase: finish
  • Roadmap-Tag: [planned] → [active]
  • implementation_mode: false (letzter Schritt)

Quiz

1. Warum wird implementation_mode mit dem Edit-Tool gesetzt, nicht via Bash?
  • Weil Bash-Befehle langsamer sind
  • Weil Bash-Befehle ihrerseits durch Hooks blockiert werden könnten — das Edit-Tool ist immer verfügbar
  • Weil settings.json kein Shell-Script ist
2. Was ist der Vorteil von Subagent-Driven gegenüber Inline?
  • Subagent-Driven ist schneller
  • Jeder Task startet mit frischem Kontext — kein Drift durch frühe Annahmen, Review zwischen Tasks möglich
  • Subagent-Driven kann mehrere Sprachen gleichzeitig
3. Was steht in scheduled_plans.json wenn der User "später" antwortet?
  • Kein Eintrag wird erstellt
  • scheduled_time: null, status: "pending"
  • scheduled_time: "23:59"

Zusammenfassung

  • Phase 3: Plan → Ausführung
  • Modus: Subagent-Driven (empfohlen) oder Inline
  • Scheduling: jetzt / HH:MM / später → scheduled_plans.json
  • implementation_mode: true via Edit-Tool → Auto-Approve aller Tool-Calls
  • Läuft superpowers:subagent-driven-development oder superpowers:executing-plans
  • Nach Abschluss: Phase: finish, implementation_mode zurücksetzen
