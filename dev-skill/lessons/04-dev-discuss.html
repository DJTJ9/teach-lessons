Lektion 4 von 9 · Feature-Design und Spec

Was macht dieser Befehl?

/dev-discuss ist Phase 1 des Feature-Workflows: Er verwandelt eine vage Idee in eine präzise, schriftliche Spezifikation — durch strukturierten Dialog. Am Ende gibt es eine *-design.md Datei im Hub.

Warum erst diskutieren, dann coden?
Die meisten Bugs entstehen nicht im Code, sondern im Anforderungs-Verständnis. Wenn "Benachrichtigungen" implementiert werden ohne zu klären "für wen? welche? wann?"— entstehen Annahmen. Annahmen werden zu falschen Features. /dev-discuss erzwingt Klarheit, bevor irgendeine Zeile Code geschrieben wird.

Schritt 1: Feature wählen

Der Skill fragt zunächst, ob ein bestehendes [idea]-Item aus der Roadmap genommen werden soll oder ein neues Feature beschrieben wird. Das erlaubt schnelle Auswahl aus dem Backlog.

Schritt 2: Clarifying Questions

Die 4 Pflicht-Fragen werden einzeln gestellt — eine Frage pro Nachricht:

1
"Was soll dieses Feature leisten — welches Problem löst es?"
Verhindert "Solution Shopping": zuerst Problem verstehen, dann Lösung.
2
"Für wen genau? Welcher Nutzertyp?"
Nutzerkontext bestimmt UX-Entscheidungen und Edge Cases.
3
"Was ist definitiv NICHT teil dieses Features (Scope-Grenze)?"
Explizite Ausschlüsse verhindern Scope-Creep in der Implementierung.
4
"Wie weißt du, dass das Feature fertig und korrekt ist?"
Erzwingt messbare Erfolgskriterien — kein "sieht gut aus" als Definition of Done.
5
"Gibt es technische Constraints oder Abhängigkeiten?" (optional)
Nur wenn noch unklar nach den 4 Pflichtfragen.
6
"Hast du bereits eine Präferenz für den Ansatz?" (optional)
Respektiert vorhandenes Vorwissen statt Lösungen aufzuzwingen.
Warum eine Frage pro Nachricht (nicht alle auf einmal)?
Menschen beantworten Listen von Fragen oberflächlich. Eine Frage zwingt zur Tiefe. Außerdem kann der Skill basierend auf der Antwort entscheiden, ob weitere Fragen nötig sind — das ist bei einem Fragenkatalog nicht möglich.

Schritt 3: UI-Check

Hat das Feature UI-Komponenten? Wenn ja → der frontend-design-Skill wird aufgerufen und liefert Farbpalette, Typografie, ASCII-Wireframes. Diese fließen direkt in den "UI/UX Design"-Abschnitt der Spec.

Warum UI-Design in der Spec-Phase?
Wenn UI erst beim Implementieren entworfen wird, kostet das teure Refactors. Eine früh festgelegte visuelle Sprache verhindert "Wir bauen das Feature nochmal weil es hässlich ist."

Schritt 4: Ansätze präsentieren

Der Skill schlägt 2–3 konkrete Ansätze vor, jeder mit Name, Ein-Satz-Summary und Trade-off. Der empfohlene Ansatz kommt zuerst mit Begründung. User muss explizit zustimmen bevor es weitergeht.

Warum mehrere Ansätze?
Ein einziger Vorschlag ist keine Entscheidung — es ist Diktat. Durch Alternativen wird klar, was wir verwerfen und warum. Das wandert dann als verworfene Optionen in DECISIONS.md — wertvoll wenn man in 3 Monaten fragt "Warum haben wir damals nicht X gemacht?"

Schritt 5: Design sektionsweise zeigen

Der Skill präsentiert das Design in 4 Sektionen, jeweils mit Approval-Gate:

1. Architektur + Datenfluss
Welche Schichten, welche Datenstrukturen, wie fließen die Daten?
2. UI/UX Design
Nur wenn UI-Feature. ASCII-Wireframes, Farben, Layouts.
3. Betroffene Dateien
Liste: Create / Modify mit Beschreibung der Änderung.
4. Nicht in diesem Scope
Explizite Ausschlüsse — was wir bewusst weglassen.
Warum sektionsweise statt alles auf einmal?
Ein komplettes Design auf einmal zu reviewen überfordert. Wenn die Architektur falsch ist, brauchen wir den Rest nicht zu lesen. Sektions-Gates erlauben early exit und gezieltes Feedback.

Output: Die Spec-Datei

$HUB_DIR/topics/<slug>/specs/2026-06-23-task-filter-design.md

Pflicht-Sektionen der Spec:

  • Ziel — Warum wird das gebaut?
  • Ansatz — Gewählt + Verworfen (mit Begründung)
  • Architektur / Datenfluss
  • UI/UX Design (wenn UI)
  • Betroffene Dateien
  • Nicht in diesem Scope

Was danach passiert (automatisch)

Nach der Spec läuft Knowledge Extraction still im Hintergrund: Der gewählte Ansatz und die verworfenen Alternativen werden in DECISIONS.md geschrieben. STATUS.md wird auf Phase: plan gesetzt.

2. Was passiert wenn das Feature UI-Komponenten hat?
  • Der User beschreibt das UI in Textform
  • Der frontend-design Skill wird aufgerufen und liefert Wireframes + Farbpalette
  • Das UI wird in /dev-plan spezifiziert
3. Warum werden verworfene Ansätze in der Spec dokumentiert?
  • Als Backup falls der gewählte Ansatz scheitert
  • Damit man in Zukunft nachvollziehen kann, warum bestimmte Optionen verworfen wurden
  • Weil die Spec sonst zu kurz wäre

Zusammenfassung

  • Phase 1: Vage Idee → präzise Spec
  • 4 Pflicht-Clarifying-Questions, eine pro Nachricht
  • UI-Check → ggf. frontend-design Skill für Wireframes
  • 2–3 Ansätze mit Trade-offs → User wählt explizit
  • Design sektionsweise mit Approval-Gates
  • Output: specs/YYYY-MM-DD-<feature>-design.md
  • Danach automatisch: Knowledge Extraction + STATUS.md auf Phase: plan
