/teach Integration NEU

Lektion 10 · Lerneinheiten automatisch nach jeder Implementierung

Was ist neu?

Ab sofort ruft /dev-finish automatisch /teach auf — direkt nach der Knowledge Extraction. Jedes implementierte Feature bekommt eine Lerneinheit im Projekt-Workspace. Kein manueller Schritt nötig.

Warum /teach in den Workflow einbauen?
Implementierung und Verstehen sind zwei verschiedene Dinge. Man kann Code schreiben ohne zu verstehen warum er so funktioniert — besonders wenn Claude den Großteil übernimmt. /dev-finish erzwingt jetzt den Reflexionsmoment: Was haben die Änderungen eigentlich bewirkt? Wie spielen die Teile zusammen?

Wie /teach aufgerufen wird

1

Workspace-Slug ableiten

Der Feature-Name wird zu kebab-case: "teach Integration in Dev Workflow" → teach-integration-in-dev-workflow

Warum kebab-case?
Der Slug wird Ordnername im Dateisystem und URL-Segment für GitHub Pages. Leerzeichen und Sonderzeichen würden beides brechen.
2

Neu oder Fortsetzen?

Workspace existiert nicht
Neuer Workspace
/teach "<feature> — was beim Implementieren gelernt wurde, Architektur und Konzepte"

→ Klärungsfragen, MISSION.md, neue Lektionen
Workspace existiert bereits
Continue
/teach <feature-slug>

→ Bestehende Lektionen lesen, Lücken schließen, neue Konzepte ergänzen
Warum continue statt neu?
Ein Feature kann in mehreren Iterationen weiterentwickelt werden. Der Workspace akkumuliert Wissen über alle Iterationen. Jedes Mal neu starten würde diesen Kontext vernichten.
3

Lesson: Feld in STATUS.md

Nach dem /teach-Aufruf schreibt finish.md den Workspace-Slug ins STATUS.md:

Active: (keine aktive Entwicklung)
Phase: (none)
Spec:
Plan:
Lesson: teach-integration-in-dev-workflow
Updated: 2026-06-23

Damit ist jederzeit nachvollziehbar welcher /teach-Workspace zu welchem Feature gehört.

Auto-Finish: kein manueller Schritt mehr

Früher endete /dev-implement mit einem Handoff-Prompt: "Weiter mit /dev finish..." — ein Schritt den man leicht überspringt. Jetzt liest implement.md nach dem letzten Commit direkt finish.md ein und führt es aus.

Warum Auto-Finish statt Prompt?
Optionale Schritte werden übersprungen. Wenn /dev-finish und /teach manuell aufgerufen werden müssen, passiert das unter Zeitdruck nie. Die Integration ist nur nützlich wenn sie erzwungen wird — genau wie die Knowledge Extraction die schon immer automatisch lief. 
## Auto-Finish

After STATUS.md is updated (Phase: finish) and implementation_mode cleared:

Read and execute finish.md directly — do NOT show a handoff prompt
or wait for user input.
Pass through: slug=<SLUG>, feature=<feature-name>.

Quiz

1. Wann wird /teach in finish.md aufgerufen?
  • Vor der Knowledge Extraction
  • Nach der Knowledge Extraction, vor dem Hub-Commit
  • Nach dem Hub-Commit und Push
2. Was steht im Lesson: Feld in STATUS.md?
  • Der absolute Pfad zum /teach-Workspace
  • Der kebab-case Workspace-Slug (z.B. teach-integration-in-dev-workflow)
  • Die URL der GitHub Pages Lektion
3. Was passiert wenn der /teach-Workspace für ein Feature schon existiert?
  • Der Workspace wird überschrieben
  • Es wird ein neuer Workspace mit einem Suffix angelegt
  • /teach wird im Continue-Modus aufgerufen — bestehende Lektionen bleiben, neue kommen dazu
4. Warum wurde der manuelle Handoff-Prompt durch Auto-Finish ersetzt?
  • Weil der Prompt technisch nicht mehr funktionierte
  • Weil optionale Schritte unter Zeitdruck übersprungen werden — Automatisierung erzwingt die Reflexion
  • Weil /dev-finish zu lang zum Eintippen ist

Zusammenfassung

  • finish.md hat jetzt einen neuen Block: Teach Integration (nach Knowledge Extraction)
  • implement.md: ## Handoff → ## Auto-Finish — finish.md wird direkt ausgeführt
  • SKILL.md: STATUS.md Init-Template bekommt Lesson: Feld
  • Workspace-Slug = kebab-case Feature-Name; continue wenn Workspace existiert
  • Lesson: Feld in STATUS.md zeigt welcher /teach-Workspace zu diesem Feature gehört
  • Kein manueller Schritt: implement → finish → teach läuft durch ohne Prompt
